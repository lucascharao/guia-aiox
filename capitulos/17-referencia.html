17.1 Tabela de Agentes

Referência rápida de todos os agentes oficiais do AIOX, suas personas, escopos, e comandos exclusivos.

Notas sobre Autoridade

Delegação exclusiva: Operações listadas como exclusivas de um agente não podem ser realizadas por outros agentes, mesmo que tecnicamente possível. Isso é uma regra de processo, não apenas uma convenção.

Escalação: Quando qualquer agente não consegue completar uma tarefa, a escalação vai para @aiox-master, que tem autoridade irrestrita para mediar e resolver.

Ativação: No Claude Code, agentes são ativados com /nome (spawn completo com scripts e dependências). O @nome apenas carrega o arquivo .md sem executar workflows auxiliares. Commands usam o prefixo *. Exemplo: /dev para ativar, depois *task dev-develop-story para executar.

Modelo por agente: Configurável em core-config.yaml. Recomendação: Opus para decisões estratégicas (@architect, @pm), Sonnet para execução (@dev, @qa), Haiku para tasks definidas (@sm). Gemini para UX/design.

17.3 Tabela de Workflows

Workflow Quando Usar Agentes Envolvidos Output Principal
Story Development Cycle (SDC) Todo desenvolvimento de feature @sm, @po, @dev, @qa, @devops Code merged + Story Done
QA Loop Após QA Gate com FAIL ou múltiplos CONCERNS @qa, @dev Story aprovada pelo QA
Spec Pipeline (SIMPLE) Feature com incerteza moderada de requisitos @pm, @qa spec.md aprovada
Spec Pipeline (STANDARD) Feature complexa ou com integração externa @pm, @architect, @analyst, @qa spec.md + implementation.yaml
Spec Pipeline (COMPLEX) Feature de alta complexidade ou risco Todos + ciclo de revisão spec.md + implementation.yaml revisado
Brownfield Discovery Projeto existente sem documentação @architect, @data-engineer, @ux-design-expert, @qa, @analyst, @pm TECHNICAL-DEBT-REPORT.md + Epic de modernização
Epic Execution Objetivo de produto multi-story @pm orquestrado SDC por story Todas as stories do épico Done

Fluxo de Seleção de Workflow

Projeto novo com código existente?
    Sim → Brownfield Discovery primeiro
    Não → continua

Feature com requisitos ambíguos ou complexidade >= STANDARD?
    Sim → Spec Pipeline → SDC
    Não → continua

Objetivo de produto com múltiplas stories interdependentes?
    Sim → Create Epic → Execute Epic → SDC por story
    Não → continua

Story individual com critérios claros?
    Sim → SDC diretamente
    Bugs simples → SDC modo YOLO

Status de Épicos

Status Descrição
Planning Épico criado, stories sendo decompostas
Ready Todas as stories iniciais em Draft ou Ready
InProgress Pelo menos uma story em InProgress
Done Todas as stories em Done e critérios de sucesso verificados
Blocked Épico bloqueado por dependência externa ou decisão estratégica

Verdicts do QA Gate

Verdict Significado Próximo Passo
PASS Todos os 7 checks passaram Story → InReview, avança para @devops
CONCERNS Checks passaram com ressalvas não-bloqueantes Story → InReview (com nota de concerns)
FAIL Um ou mais checks bloqueantes falharam Story → InProgress, volta para @dev com lista de issues
WAIVED Um ou mais checks dispensados com justificativa documentada Story → InReview (com waiver documentado)

Verdicts do QA Loop

Verdict Significado Próximo Passo
APPROVE Story aprovada após todas as correções Fim do loop, story avança
REJECT Issues identificados, correção necessária @dev *qa-loop-fix
BLOCKED Problema não resolvível entre @qa e @dev Escalação imediata para @aiox-master

17.5 Resumo da Constitution

A Constitution define os princípios inegociáveis do AIOX. Gates automáticos bloqueiam violações.

Artigo Princípio Severidade Significado Prático
I CLI First NON-NEGOTIABLE Toda inteligência e automação vive no CLI. UI é terciária.
II Agent Authority NON-NEGOTIABLE Cada agente tem autoridade delimitada. Violações bloqueiam.
III Story-Driven Development MUST Não há código sem story. Sem exceção.
IV No Invention MUST Agentes não adicionam features não solicitadas. Spec = Lei.
V Quality First MUST Gates de qualidade não são opcionais. WAIVED requer justificativa.
VI Absolute Imports SHOULD Imports absolutos. Sem ../../../ no código.

Gates Automáticos

Os seguintes gates são aplicados automaticamente pelo sistema:

  • Push sem @devops: BLOQUEADO
  • Story sem ACs verificáveis: PO retorna NO-GO
  • Feature em spec sem rastreabilidade para FR/NFR/CON: Critique retorna BLOCKED
  • Agente realizando operação fora de sua autoridade: Escalação para @aiox-master

17.7 Glossário

AC (Acceptance Criteria): Critério verificável que define quando um requisito de uma story foi implementado com sucesso. Formato: "Dado X, quando Y, então Z" ou condição booleana clara.

Agent Authority: O conjunto de operações que um agente pode realizar e as que são explicitamente bloqueadas. Definido na Constitution e implementado como regra de processo.

Brownfield Discovery: Workflow de 10 fases para assessment técnico de projetos existentes com dívida técnica acumulada.

Code First: Princípio fundamental que prioriza Python/código determinístico sobre IA para qualquer operação que pode ser resolvida sem julgamento semântico.

Constitution: Documento de princípios inegociáveis do AIOX que governa o comportamento de todos os agentes e processos do framework.

EPIC-EXECUTION.yaml: Arquivo YAML que registra o estado completo de um épico em execução, incluindo stories, dependências, riscos, blockers, e métricas de progresso.

Epic: Objetivo de produto de maior escala que se traduz em múltiplas stories coordenadas com critérios de sucesso próprios.

Gate: Verificação obrigatória que valida qualidade ou conformidade antes que o trabalho avance para a próxima fase. Gates que falham retornam o trabalho para correção.

Handoff Artifact: Arquivo YAML compacto (~500 tokens) gerado quando um agente passa o controle para outro. Contém o essencial do contexto sem carregar a persona completa do agente anterior.

MEMORY.md: Arquivo que armazena a memória de longo prazo de um agente. É a única forma de persistir contexto entre sessões. Deve ser curado ativamente.

No Invention (Artigo IV): Princípio constitucional que proíbe agentes de adicionar features, comportamentos, ou requisitos que não foram explicitamente especificados nos documentos de requisitos.

QA Gate: Verificação de 7 checks executada por Quinn após a implementação de uma story. Produz veredicto: PASS, CONCERNS, FAIL, ou WAIVED.

QA Loop: Ciclo iterativo de revisão-correção entre Quinn e Dex após um QA Gate com FAIL. Máximo de 5 iterações antes de escalação automática.

SDC (Story Development Cycle): O workflow primário do AIOX — 4 fases (Criar, Validar, Implementar, QA) executadas por 5 agentes em sequência.

Spec Pipeline: Workflow de 3-6 fases para transformar requisitos informais em especificação executável antes de criar stories de implementação.

Squad: Equipe de agentes especializados em um domínio específico, com personas próprias, dados de primeiro partido, e memória compartilhada.

Story: Unidade fundamental de trabalho no AIOX. Tem status, critérios de aceite verificáveis, escopo delimitado, e rastreabilidade para requisitos documentados.

WAIVED: Veredicto especial do QA Gate que dispensa um check específico com justificativa documentada. Uso consciente e justificado, não omissão.

17.8 Integração com Supabase

O Que É o Supabase

Supabase é um Backend-as-a-Service (BaaS) baseado em PostgreSQL. Fornece banco de dados, autenticação, storage de arquivos, e edge functions. No contexto do AIOX, é usado para armazenamento de dados e backend.

Configuração Inicial

  1. Criar conta em supabase.com
  2. Criar novo projeto: New Project → preencher nome, senha do banco, região (South America - São Paulo recomendado)
  3. Aguardar provisionamento (~2 minutos)

Obtendo Credenciais

Caminho: Project → Settings → API

  • Project URL: https://xxxxx.supabase.co — endereço do projeto
  • anon public key: chave pública para uso no frontend (começa com eyJh...)
  • service_role key: chave administrativa — NUNCA use no frontend. Tem permissão total sobre o banco.

Interfaces Importantes

Interface Uso
SQL Editor Executar comandos SQL diretamente (CREATE TABLE, INSERT, etc.)
Table Editor Visualizar e editar dados das tabelas
Auth Configurar autenticação de usuários
Storage Gerenciar arquivos (imagens, PDFs, etc.)
Edge Functions Funções serverless em Deno/TypeScript

Variáveis de Ambiente

NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Estas variáveis ficam em .env.local (NUNCA comitado no Git).

Agente Responsável

@data-engineer (Dara) é o agente delegado para operações de banco: schema design, DDL, RLS policies, migrations, query optimization. Em modo avançado, pode usar MCP do Supabase para operações diretas — mas SOMENTE com permissões bem configuradas.

Precauções

  • Aprenda o Supabase manualmente antes de automatizar com MCP
  • MCP sem permissões restritivas pode executar comandos destrutivos
  • Sempre tenha backup antes de operações críticas
  • Use a chave anon public no frontend, NUNCA a service_role

17.9 Integração com Vercel

O Que É a Vercel

Vercel é uma plataforma de deploy para aplicações web modernas. Criadores do Next.js, com suporte nativo e otimizado para o framework. Funciona como um container web que lê código diretamente do GitHub e publica automaticamente.

Como Funciona

  1. Vercel conecta-se ao repositório GitHub
  2. A cada git push, detecta automaticamente e re-deploya
  3. Cada deploy gera uma URL única de preview
  4. A URL de produção é atualizada automaticamente no branch principal

Deploy Inicial

  1. Acessar vercel.com (login com conta GitHub)
  2. Add New → Project → Importar repositório do GitHub
  3. CONFIGURAÇÃO CRÍTICA — Root Directory: Se o projeto Next.js está numa subpasta (ex: landing/), clicar em Edit ao lado de "Root Directory" e digitar o nome da subpasta
  4. Environment Variables: clicar em "Import .env" → selecionar .env.local da máquina local
  5. Deploy → aguardar build (~2-3 minutos)

Domínio Personalizado

Para usar domínio próprio: 1. Comprar domínio (Registro.br, GoDaddy, etc.) 2. No Vercel: Settings → Domains → adicionar domínio 3. Configurar registros DNS na registradora conforme instruções do Vercel 4. Propagação: 24-48 horas

Debugging no Vercel

Aba Uso
Deployments Histórico de deploys com status (Success/Error)
Logs Erros detalhados de build e runtime
Settings Configurações do projeto, variáveis, domínios

Fluxo de Correção de Erros

  1. Identificar erro na aba Deployments/Logs
  2. Copiar texto do erro (ou tirar print da tela)
  3. Colar no Claude Code (Ctrl+V aceita imagens)
  4. Agente analisa e corrige
  5. @devops faz push → Vercel re-deploya automaticamente
  6. F5 na URL → versão atualizada

Agente Responsável

@devops (Gage) gerencia todo o fluxo de deploy. Após cada push na main, o Vercel detecta automaticamente via webhook do GitHub.

17.10 Troubleshooting Comum

Problema Causa Solução
Pasta landing/ aparece vazia no GitHub Dev criou projeto Next.js com .git interno — Git pai trata como submódulo @devops remove .git interno: rm -rf landing/.git antes do push
Vercel falha no build ("Module not found") Root Directory não configurado — Vercel busca na raiz mas projeto está em subpasta Configurar Root Directory para landing/ (ou nome da subpasta) nas settings do Vercel
Página não atualiza após deploy Cache do browser Hard refresh: Ctrl+Shift+R (ou Cmd+Shift+R no Mac)
Claude Code travou com HTTP 500 Instabilidade temporária da API Fechar e reabrir Claude Code — retoma contexto automaticamente
MCP Supabase deletou dados Sem permissões restritivas configuradas Configurar permissões ANTES de usar MCP. Sempre fazer backup
Prompt de merge/overwrite na instalação AIOX instalado em pasta com projeto existente Sempre criar pasta vazia nova antes de instalar
.env.local aparece no GitHub .gitignore não configurado corretamente Verificar que .env.local está listado no .gitignore. Se já foi comitado: git rm --cached .env.local
AIOX Pro não ativa no Mac Cache do NPX corrompido ou nome de pasta com espaços Limpar cache: npx clear-npx-cache, criar pasta sem espaços/acentos, max 3 dispositivos por licença
Contexto lotado — agente alucina Janela acima de 100k tokens, qualidade degrada até 50% /clear entre fases, desabilitar auto-compact, desabilitar MCPs não usados
Claude "finge" que chamou agente Usou @ em vez de / para ativar agente Sempre usar /agente no Claude Code. @ só carrega o .md, / executa o spawn completo
Banco de dados ficou vazio MCP ou agente executou DROP/TRUNCATE sem restrict Criar usuario restrito: REVOKE DROP, TRUNCATE ON ALL TABLES + deny rules em settings.json
Squad commands não aparecem Registry desatualizado após instalar squad Executar *refresh-registry após adicionar novo squad
iCloud/OneDrive corrompe arquivos Sincronização de nuvem interfere com escrita do agente Manter projetos AIOX fora de pastas sincronizadas

Apêndice D — Integração com Ferramentas Externas

CodeRabbit

O CodeRabbit fornece review automático de código antes do QA Gate.

Instalação:

pip install coderabbit
# ou
curl -sSL https://install.coderabbit.ai | bash

Uso no AIOX:

# Review de uncommitted changes
coderabbit --prompt-only -t uncommitted

# Review de tudo desde main
coderabbit --prompt-only --base main

Integração automática: Configure em core-config.yaml:

autoClaude:
  coderabbit:
    enabled: true
    trigger: "pre-qa-gate"  # Executa antes do QA Gate
    autoHeal:
      enabled: true
      maxIterations: 2

GitHub CLI

gh auth login

gh pr merge {number} --squash

Apêndice E — Troubleshooting

Problema: Agente não reconhece contexto do projeto

Sintomas: Agente faz perguntas que já foram respondidas em sessões anteriores.

Causa: MEMORY.md vazio ou desatualizado.

Solução: 1. Verifique se agents/{agente}/MEMORY.md existe e tem conteúdo 2. Adicione contexto crítico do projeto ao MEMORY.md 3. Na próxima sessão, verifique se o agente carrega o arquivo

Problema: QA Loop não converge

Sintomas: Após 3+ iterações, os mesmos issues continuam aparecendo.

Causa: Possível ambiguidade nos critérios de aceite ou problema de especificação.

Solução: 1. Execute *escalate-qa-loop para escalação imediata 2. @aiox-master vai analisar o histórico do loop 3. Verifique se o problema é de implementação ou de especificação

Problema: Custo aumentando sem aumento de uso

Sintomas: aiox stats mostra crescimento de tokens sem crescimento de atividade.

Causa: Contexto acumulado — MEMORY.md grande, sessões longas sem compactação.

Solução: 1. Execute aiox doctor — verificar tamanho de MEMORY.md por agente 2. Compacte MEMORY.md de agentes acima de 5KB 3. Inicie sessões novas mais frequentemente para limpar contexto acumulado

Problema: Story rejeitada repetidamente pelo PO

Sintomas: Pax retorna NO-GO nas mesmas dimensões várias vezes.

Causa: Requisitos genuinamente ambíguos que precisam ser esclarecidos com stakeholder.

Solução: 1. Identifique qual dimensão falha consistentemente 2. Se AC são ambíguos: execute Spec Pipeline para esclarecer requisitos 3. Se contexto está faltando: alimente informação ao MEMORY.md de @sm

Problema: @devops não consegue criar PR

Sintomas: gh pr create falha com erro de autenticação.

Causa: Token do GitHub CLI expirado ou permissões insuficientes.

Solução:

gh auth status  # Verificar status
gh auth login   # Re-autenticar se necessário
gh auth refresh # Atualizar token

Problema: Code Intelligence indisponível

Sintomas: aiox doctor reporta Code Intelligence em fallback.

Causa: Provider não configurado ou indisponível.

Impacto: Sistema opera normalmente sem enrichment — degradação graceful.

Solução: 1. Verifique core-config.yaml — code_intelligence.enabled: true? 2. Verifique se o provider está rodando: aiox doctor --verbose 3. Se provider necessário: configure ou reinstale conforme documentação do provider

Problema: Squad produzindo outputs inconsistentes

Sintomas: Respostas do squad variam significativamente para inputs similares.

Causa: Personas não suficientemente específicas, ou MEMORY.md compartilhado com conflitos.

Solução: 1. Revise as personas — são suficientemente específicas? 2. Verifique o MEMORY.md do squad — há entradas conflitantes? 3. Execute o Teste 1 (Coerência de Persona) descrito no Capítulo 12 4. Compare dois outputs recentes para o mesmo tipo de análise — onde divergem?
