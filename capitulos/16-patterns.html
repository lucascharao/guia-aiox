Patterns e Anti-Patterns

Introdução: Aprendendo com Quem Errou Primeiro

Nenhum framework nasce maduro. Os patterns documentados aqui emergiram de uso real — projetos que funcionaram, projetos que falharam, e a análise honesta de por quê cada um foi como foi.

Um pattern é uma solução que funcionou em contextos múltiplos e pode ser intencionalmente replicada. Um anti-pattern é um problema recorrente — uma solução aparente que cria problemas maiores do que resolve.

A diferença entre um usuário iniciante e um usuário avançado do AIOX é frequentemente a capacidade de reconhecer esses patterns em tempo real e ajustar o curso antes que os problemas se acumulem.

Os 10 Patterns

Pattern 1: Story Antes de Tudo

Descrição: Todo trabalho começa com uma story. Sem exceção. Bug fix urgente? Story. Refatoração rápida? Story. Hotfix às 23h? Story — mesmo que seja um parágrafo.

Por que funciona: A story não é burocracia. É o mecanismo que garante que existe um registro de por que o trabalho foi feito, o que foi feito, e quem aprovou. Sem story, você tem um repositório com commits que ninguém sabe de onde vieram. Com story, você tem rastreabilidade completa.

Implementação:

# Antes de qualquer implementação:
@sm *draft  # versão rápida para bugs e patches urgentes

O draft pode ser um parágrafo. O que importa é que existe e tem pelo menos um AC verificável.

Pattern 2: PO Gate Como Investimento

Descrição: Trate o checklist de 10 pontos do Pax como investimento, não como overhead. Cada minuto de validação economiza potencialmente horas de retrabalho.

A matemática: Uma story mal especificada que passa direto para implementação e descobre um problema de requisito no meio da implementação custa: o tempo já investido na implementação errada + o tempo de corrigir o problema + o custo cognitivo de trocar de contexto. Em média, 4-8 horas desperdiçadas.

O mesmo problema identificado por Pax no checklist: 15 minutos para corrigir a story.

Quando o pattern falha: Times que usam Pax como carimbo automático de aprovação sem realmente engajar com o checklist. O checklist precisa ser executado de forma genuína — não como formalidade.

Pattern 3: Commits Frequentes e Descritivos

Descrição: Dex (e você) devem fazer commits frequentes durante a implementação, com mensagens que referenciam a story e o AC sendo implementado.

Por que funciona: Commits granulares têm três benefícios: permitem reverter mudanças específicas sem perder o trabalho inteiro, tornam o code review mais focado (cada commit é uma mudança atômica), e criam uma trilha narrativa da implementação.

# Bom
git commit -m "feat: add JWT generation in auth.service [Story 3.3 AC-1]"
git commit -m "feat: implement refresh token rotation [Story 3.3 AC-4]"
git commit -m "test: add unit tests for JWT service [Story 3.3]"

# Ruim
git commit -m "implementei autenticação"

Pattern 4: Memória como Ativo

Descrição: Trate o MEMORY.md de cada agente como um ativo de projeto, não como um arquivo de log. Cuide dele ativamente — adicione decisões importantes, remova contexto obsoleto, organize por relevância.

O sinal de que está funcionando: No início de uma nova sessão, o agente tem contexto suficiente para continuar exatamente onde parou, sem você precisar re-explicar o projeto.

Implementação prática: Ao final de cada sessão significativa, peça ao agente que atualize seu MEMORY.md:

@dev: "Antes de finalizar, registre no MEMORY.md as decisões
      técnicas importantes que tomamos hoje."

Pattern 5: Fail Fast no QA Loop

Descrição: Quando o QA Loop identifica um problema HIGH severity, não continue com os outros checks. Escalate imediatamente.

Por que funciona: Um problema HIGH severity em check inicial frequentemente indica problemas fundamentais de implementação que vão invalidar os outros checks. Continuar avaliando numa implementação com problema fundamental desperdiça tempo e tokens.

Quinn: "Check 1 — Cobertura de ACs: AC-4 não implementado.
       Severidade: HIGH (AC-4 é o critério de segurança principal).
       Ação: REJEITAR imediatamente. Não continuar avaliação."

Pattern 6: Spec Pipeline para Incerteza Alta

Descrição: Sempre que você perceber que está redefinindo os critérios de aceite de uma story várias vezes antes de começar a implementar — pare. Isso é sinal de incerteza alta. Execute o Spec Pipeline antes de criar a story.

O teste de incerteza: Se você tem dificuldade de escrever três critérios de aceite verificáveis para a feature, você não entende a feature bem o suficiente para implementá-la. Spec Pipeline primeiro.

Pattern 7: Brownfield Antes de Qualquer Coisa em Projetos Legados

Descrição: Ao assumir um projeto existente sem documentação, o primeiro épico é sempre Brownfield Discovery. Não crie stories de feature antes de ter o assessment técnico.

A tentação a resistir: "Eu sei o suficiente para começar a implementar. O Brownfield vai demorar demais."

Por que resistir: O Brownfield Discovery não é optional. Projetos legados têm surpresas. O custo de descobrir essas surpresas no meio de uma implementação é sempre maior que o custo de mapeá-las antes de começar.

Pattern 8: Escalação Imediata de BLOCKED

Descrição: Quando qualquer agente retorna BLOCKED, escale para @aiox-master imediatamente. Não tente resolver um BLOCKED circulando entre @dev e @qa em loop.

O anti-pattern correspondente: @dev e @qa entrando em ciclo de "corrige / identifica problema / corrige / identifica problema" para um problema que é fundamentalmente de especificação e não de implementação.

BLOCKED significa "esse problema não pode ser resolvido neste nível". A escalação não é sinal de falha — é o protocolo correto.

Pattern 9: Dashboard de Custo como Ritual Semanal

Descrição: Reserve 15 minutos semanalmente para revisar o dashboard de custo via aiox stats. Identifique anomalias e operações que poderiam ser determinizadas.

O que procurar: Spikes de custo inexplicados, operações que cresceram desproporcionalmente, oportunidades de migração para Python.

O resultado a longo prazo: Projetos que fazem esse ritual consistentemente têm custo estável ou decrescente mesmo com volume crescente de uso. Projetos que não fazem têm custo que cresce linearmente com o volume.

Pattern 10: Versionamento de Personas

Descrição: Trate as personas dos agentes como código — versione-as, documente as mudanças, e tenha um processo formal para mudanças significativas.

Por que importa: Uma mudança inadvertida numa persona pode alterar o comportamento do agente de formas sutis que só se manifestam em situações específicas. Com versionamento, você pode reverter para a versão anterior quando identifica degradação.

# Mudança de persona é um commit com mensagem descritiva
git commit -m "feat(agents/dev): add React Native expertise to Dex persona [Story 5.1]"

Os 10 Anti-Patterns

Anti-Pattern 1: O Prompt Monolítico

Descrição: Em vez de usar o sistema de agentes, você escreve um único prompt gigante que tenta fazer tudo de uma vez: "Crie uma story, valide-a, implemente-a, e faça QA em um único response."

Por que falha: O resultado é superficial em tudo e profundo em nada. O que o sistema de agentes compra — perspectivas especializadas, gates de qualidade, autoridade delimitada — se perde completamente.

O sinal: Você está copiando e colando o resultado de uma "fase" como input da "próxima fase" manualmente, sem usar os agentes especializados.

Anti-Pattern 2: MEMORY.md Negligenciado

Descrição: O MEMORY.md de cada agente cresce indefinidamente sem curadoria, acumulando contexto desatualizado e conflitante.

Sintomas: - Agentes produzindo outputs inconsistentes entre sessões - Tempo crescente de "orientação" no início de cada sessão - Agente "lembrando" decisões que foram revertidas meses atrás

A correção: Auditoria mensal do MEMORY.md. Remova entradas com mais de 90 dias que não têm impacto ativo. Consolide entradas relacionadas. Mantenha abaixo de 5KB por agente.

Anti-Pattern 3: Skip do PO Gate

Descrição: Pular a validação do Pax para economizar tempo, especialmente em stories "simples".

Por que é um anti-pattern: Não existe story "simples o suficiente para pular o PO Gate". A complexidade não é o ponto — a clareza de critérios é. Uma story de aparência simples com AC ambíguo vai gerar retrabalho independente de quão simples o código seja.

A consequência real: Você não economiza tempo. Você desloca o custo para a fase de implementação e QA, onde ele é mais caro.

Anti-Pattern 4: @dev Fazendo Push

Descrição: Dar permissão para @dev fazer git push diretamente para o repositório remoto, "porque é mais rápido".

Por que é anti-pattern: Essa "otimização" remove o gate de QA da entrega. Você vai descobrir isso quando code sem QA causar um problema em produção que teria sido encontrado em 5 minutos de revisão.

A regra é inegociável: @devops *push apenas. Sem exceções.

Anti-Pattern 5: Épico Sem Critérios de Sucesso

Descrição: Criar um épico com uma lista de stories mas sem critérios de sucesso claros para o épico como um todo.

O problema: Você pode completar todas as stories do épico e ainda não saber se o épico foi bem-sucedido. Sem critérios de sucesso, o épico é um conjunto de trabalho, não um objetivo.

A correção: Todo épico precisa de pelo menos 3 critérios de sucesso verificáveis antes de ser iniciado.

Anti-Pattern 6: Squad Para Tudo

Descrição: Criar squads para toda necessidade de análise, mesmo para perguntas pontuais e não-recorrentes.

O custo: Setup de squad tem overhead. Para uma pergunta que você vai fazer uma vez, um agente individual com contexto ad-hoc é mais eficiente.

A heurística: Squads para problemas recorrentes de domínio específico. Agentes individuais para consultas pontuais.

Anti-Pattern 7: Spec Paralysis

Descrição: Usar o Spec Pipeline como razão para nunca começar a implementação — "a spec não está perfeita ainda".

Por que é anti-pattern: Uma spec 80% completa que inicia implementação gera feedback real. Uma spec 100% perfeita que nunca chega à implementação não entrega valor.

A regra prática: Spec Pipeline deve resultar em stories executáveis. Se após duas iterações de critique a spec ainda não está executável, o problema é de requisitos — não de spec. Escale para @aiox-master.

Anti-Pattern 8: Personas Genéricas

Descrição: Criar agentes (de squad ou de desenvolvimento) com personas vagas e sem especificidade de domínio. "Marina é uma analista experiente que sabe de finanças."

O resultado: Análises genéricas que não justificam o overhead de usar um squad em vez de um único agente genérico.

A correção: Personas específicas com background concreto, perspectiva característica definida, skills operacionais listadas, e limitações explícitas.

Anti-Pattern 9: Ignorar os Sinais de Degradação

Descrição: Não monitorar a qualidade dos outputs ao longo do tempo, ignorando sinais de degradação como aumento de rejeições do PO, mais iterações no QA Loop, ou feedback negativo de usuários.

O problema: Degradação de qualidade é geralmente gradual — não acontece abruptamente, acontece por acumulação de pequenas mudanças. Sem monitoramento, você descobre o problema tarde.

A correção: Monitoramento proativo. Dashboard de custo semanal. Auditoria de qualidade mensal.

Anti-Pattern 10: O Operador Ausente

Descrição: Configurar o AIOX e assumir que vai funcionar autonomamente sem supervisão ativa — o operador humano se torna apenas um receptor de outputs.

Por que é anti-pattern: O AIOX amplifica inteligência humana — não a substitui. O operador precisa estar presente para: identificar quando os critérios de produto estão corretos, reconhecer quando o contexto de negócio mudou e as decisions de produto anteriores precisam ser revisitadas, e validar que os critérios de aceite das stories correspondem ao valor que o produto deve entregar.

Um AIOX sem operador ativo produz código correto que pode não resolver o problema certo.

A Mentalidade de Operador

Os 10 patterns e 10 anti-patterns acima podem ser condensados numa única mentalidade: o operador do AIOX não é um usuário passivo do sistema — é o responsável pelo output final.

Os agentes executam. Os gates validam. Os workflows estruturam. Mas a inteligência de produto — o "por que estamos fazendo isso" e o "isso está entregando o valor que precisamos" — é sempre responsabilidade do operador humano.

A melhor forma de usar o AIOX não é "confiar no sistema e verificar o resultado". É "usar o sistema como multiplicador da sua própria inteligência, mantendo sempre o controle sobre as decisões que importam".

O Checklist do Operador Consciente

Aplique este checklist ao final de cada semana de uso do AIOX:

Rastreabilidade: - [ ] Todo código que foi para o repositório tem uma story correspondente? - [ ] Todas as stories têm critérios de aceite verificáveis? - [ ] As stories completadas correspondem ao que os stakeholders esperavam?

Processo: - [ ] O PO Gate foi genuinamente executado ou apenas carimbado? - [ ] O QA Gate identificou problemas reais ou está virando formalidade? - [ ] O QA Loop está convergindo ou circulando?

Qualidade: - [ ] Os outputs dos agentes estão melhores, iguais, ou piores que mês passado? - [ ] Os agentes estão produzindo outputs consistentes entre sessões? - [ ] O MEMORY.md reflete o estado atual do projeto?

Custo: - [ ] O custo semanal está dentro da expectativa? - [ ] Há operações que poderiam ser determinizadas? - [ ] O routing de modelos está otimizado?

Um operador que responde estas perguntas honestamente toda semana está usando o AIOX da forma para a qual foi projetado: como um sistema de amplificação da inteligência humana, não um substituto para ela.

Anti-Patterns Adicionais: Lições do Uso Real

Anti-Pattern 11: Pular o PRD e ir direto para o Dev

Descrição: Muitos estudantes tentam chamar o @dev diretamente com uma descrição vaga do que querem construir — "crie uma tela de login com autenticação social".

Por que falha: Sem PRD, o Dev inventa features, preenche lacunas com texto placeholder, e toma decisões arquiteturais que deveriam ter sido tomadas pelo @architect ou pelo @pm. O resultado é uma implementação tecnicamente funcional que resolve o problema errado.

A causa raiz: O PRD é a "lei" do projeto — define escopo, requisitos e restrições. Sem ele, cada agente opera no escuro, fazendo suposições que podem ou não corresponder à sua visão real. Um @dev sem PRD é como um desenvolvedor sem especificação: ele vai construir algo, mas não necessariamente o que você queria.

A correção: PRD antes de qualquer chamada ao @dev. Sempre. O investimento de 30 minutos no PRD economiza horas de retrabalho.

Anti-Pattern 12: Usar IA para tudo sem pensar no processo real

Descrição: Tentar usar IA sem antes pensar em como o processo funcionaria na vida real. "Quero lançar um produto digital" — e imediatamente chamar agentes, sem estruturar o pensamento.

O problema fundamental: Os agentes do AIOX espelham um time de software real. @pm faz o que um Product Manager faz. @architect faz o que um arquiteto sênior faz. @dev faz o que um desenvolvedor faz. Se você não saberia pular uma etapa num time real (como ignorar o QA antes de ir para produção), não pule com os agentes.

A raiz do anti-pattern: O material de Framework Thinking existe por esta razão. Pensar em blocos pequenos, identificar patterns, estruturar o raciocínio ANTES de engajar a IA — é o que separa quem usa o AIOX como ferramenta de quem o usa como gerador de texto aleatório.

A correção: Para cada tarefa, decomponha o processo real primeiro. Qual seria o fluxo num time humano? Quem você chamaria primeiro? Qual seria a sequência lógica? Só então engaje os agentes nessa mesma sequência.

Anti-Pattern 13: MCP do Supabase sem permissões configuradas

Descrição: Conectar o Claude Code ao Supabase via MCP sem configurar restrições de permissão adequadas.

O caso real: Um estudante da cohort teve o banco de dados Supabase inteiro deletado porque o Claude Code com MCP tinha permissões irrestritas configuradas. O agente, ao tentar "limpar" dados de desenvolvimento, executou operações que não deveriam ter sido possíveis.

Por que acontece: A velocidade de setup encoraja pular configurações de segurança. "Vou configurar as permissões depois" — e o depois não chega.

A correção em duas etapas: 1. Aprenda o Supabase manualmente primeiro — SQL Editor, Table Editor, entenda o que cada operação faz antes de automatizá-la 2. Configure permissões restritivas antes de conectar o MCP — defina exatamente quais operações o agente pode executar, e bloqueie todo o resto por padrão

Automatização sem entendimento é delegação às cegas. Entenda a ferramenta primeiro; automatize depois.

Anti-Pattern 14: Copy como pensamento posterior

Descrição: Solicitar ao @dev que crie páginas e interfaces sem fornecer o copy (conteúdo textual) antecipadamente.

O resultado previsível: O agente preenche a página com placeholder text — "Lorem ipsum", "Título aqui", "Descrição do produto", "Clique para saber mais". O design funciona, mas o produto parece uma maquete. Você então precisa voltar e substituir tudo manualmente, ou pedir ao agente que refaça — duas vezes o trabalho.

O pattern correto: 1. Crie o copy primeiro — via squad de copy, manualmente, ou com um agente especializado 2. Referencie o arquivo de copy no PRD 3. Só então chame o @dev

A mentalidade: O copy não é detalhe de acabamento — é parte da especificação. Uma tela sem copy definitivo não está especificada; está incompleta. O @dev precisa do copy para construir a tela correta, não uma versão que você vai ter que consertar depois.

Anti-Pattern 15: Squad Decorativo

Descrição: Criar squads que são consultivos — respondem perguntas, geram insights — mas nunca resolvem problemas reais. O squad existe, tem agentes bonitos, mas não produz output acionável.

Sintoma: Você tem 5 squads criados e nenhum problema concreto resolvido. O squad de copy gera textos que você não usa. O squad de análise produz relatórios que ninguém lê. É "epifania mode" — a criação é satisfatória, mas a execução não acontece.

A correção: Squad operacional > Squad decorativo. Antes de criar um squad, pergunte: "Que problema específico e recorrente este squad vai resolver?" Se a resposta for vaga, pare. Resolva o problema manualmente primeiro. Depois automatize com squad.

Anti-Pattern 16: Modo Epifania (Criar Sem Resolver)

Descrição: Ficar criando squads, agentes, e workflows infinitamente sem nunca entregar valor real. A criação é viciante — mas criar não é resolver.

O salto necessário: Do squad decorativo para o squad operacional. Do operacional para o produto. Cada nível exige parar de criar e começar a resolver. Como Alan enfatiza: "Método sem execução é apenas epifania."

A correção: Estabeleça um gate pessoal: não crie o próximo squad sem que o atual tenha resolvido pelo menos 3 problemas reais para clientes ou para você.

Anti-Pattern 17: Deixar a IA Criar Tabelas de Banco de Dados

Descrição: Pedir para um agente "crie o banco de dados para meu sistema" sem definir entidades, status, e relações antes.

O resultado: Overengineering. A IA cria 20 tabelas quando 5 resolveriam. Cria campos que ninguém pediu. Inventa relações que não existem no domínio. E o pior: o operador aceita porque "parece profissional".

A correção da Aula 07: Defina entidades e suas taxonomias ANTES de tocar no banco. Use enums e categorizações antes de criar tabelas. Considere o Content Type Pattern — uma tabela genérica de conteúdo pode substituir 20 tabelas específicas. Deixe @data-engineer implementar o DDL, mas as decisões de entidade são suas.

Anti-Pattern 18: Documentação Estática (Doc Rot)

Descrição: Manter documentação extensa que desatualiza rapidamente. CLAUDE.md de 300 linhas que ninguém atualiza. READMEs que descrevem o projeto de 3 meses atrás.

Caso real: Na mentoria, Alan encontrou um CLAUDE.md errado que causou 40.000 tokens desperdiçados — o agente seguia instruções desatualizadas e produzia output inútil repetidamente.

A correção: CLAUDE.md mínimo com ponteiros. Básico + referências para onde está o detalhe. O código É a documentação. Quando possível, renderização em tempo real (HTMLs que mostram estado atual) > documentação estática que envelhece.

Anti-Pattern 19: Conversar Demais com a IA

Descrição: Sessões longas de conversa com o agente, explicando, re-explicando, ajustando, pedindo reformulações.

O diagnóstico: Se você está conversando demais, está faltando uma Task ou um Workflow. A conversa é sinal de que o processo não está formalizado. Pedro Valério: "Eu não escrevo. Eu crio documento pra ele ler."

A correção: Identifique o padrão da conversa recorrente. Crie uma task que encapsula o que você está pedindo repetidamente. Na próxima vez, execute a task em vez de conversar.

Lições do Cohort: Insights dos Estudantes

O conhecimento teórico de patterns e anti-patterns é necessário, mas insuficiente. As lições mais valiosas emergem do uso real — dos erros cometidos, das descobertas inesperadas, e dos momentos de clareza que só acontecem quando você está com as mãos na massa.

Estes são insights reais da cohort da Academia Lendária:

"É viciante"

Quando você vê os agentes trabalhando em paralelo — criando coisas simultaneamente, cada um com seu domínio, o resultado emergindo da colaboração — você quer criar mais. Esse estado é valioso: use essa energia, mas mantenha a disciplina do processo. O entusiasmo sem estrutura é onde nascem os anti-patterns.

"O AIOX é um organismo vivo"

A ferramenta muda quase diariamente. O que funcionava na semana passada pode ser diferente hoje. Uma funcionalidade que não existia ontem pode estar disponível amanhã. Isso tem uma consequência direta: não memorize passos — entenda o raciocínio por trás deles. Quem memorizou "como fazer X" ficou perdido quando X mudou. Quem entendeu "por que X funciona assim" adaptou-se imediatamente. É por isso que o Framework Thinking é tão importante: ele ensina raciocínio, não receitas.

"Não decore, pratique"

Instalação, bootstrap, criação de squads — repita até virar automático. Como memória muscular. Assistir a aula não é suficiente; você precisa colocar a mão na massa. O gap entre "entender o conceito" e "executar sem pensar" só se fecha com repetição deliberada. Reserve tempo para praticar os fluxos básicos até que não precise mais consultar o material.

"Erros são parte do processo"

Erros são normais e esperados. A habilidade crítica não é evitar erros — é saber sair deles rapidamente. Vercel Logs e Claude Code são suas ferramentas para isso. Quem aprende a ler logs e interpretar mensagens de erro evolui exponencialmente mais rápido do que quem tenta construir sem errar. Errar rápido, aprender rápido, corrigir rápido — esse é o ritmo.

"A melhor prática vale a pena"

Como o estudante Aloysio colocou: "É interessante — tem muita coisa que a gente consegue descobrir por tentativa e erro. Mas aprender a melhor prática vale a pena, especialmente se você está construindo para clientes."

A tentativa e erro te leva a algum lugar. A melhor prática te leva ao lugar certo, mais rápido, com menos retrabalho, e com confiança para replicar o resultado. Para projetos pessoais, a exploração tem seu valor. Para clientes, a precisão é o que importa.

"Técnica é líquida, fundamentos são permanentes"

RAG era o rei há 6 meses. Agora YAML local com frontmatter resolve melhor e mais barato. A ferramenta específica muda constantemente — mas os fundamentos (processos, determinismo, qualidade de dados) são permanentes. Quem investe em fundamentos se adapta. Quem investe em ferramentas recomeça.

"Sem repertório, você é refém"

Sem vocabulário preciso do seu domínio, sem dados de primeira mão, sem processos documentados — você é refém do que o modelo inventa. Com repertório, você domina. É a diferença entre pedir "me faz um CRM" e fornecer 880 propostas históricas como contexto.

"Não se apegue ao código — gera 10x, descarta 10x"

O código gerado pelo AIOX é descartável. Se a base está ruim, recomeça. Gera 10x mais rápido do que humano. O apego ao código que não funciona é um viés cognitivo herdado da era pré-IA. O squad é rascunho — destrua e reconstrua até a excelência.

"A IA tenta te enganar constantemente"

Claude Code é um dev preguiçoso. Ele tenta mudar o teste para o teste passar, em vez de corrigir o código. Coloca True em verificações para pular validação. Finge que leu um arquivo. A sua função como operador é auditar — nunca confiar cegamente.

"Burnout era por causa das pessoas, não do trabalho"

Alan compartilhou: o burnout que sentia antes não era pelo volume de trabalho — era pela fricção de gerenciar pessoas. Com AIOX, o trabalho aumentou mas a fricção diminuiu. Agentes não reclamam, não faltam, não pedem aumento. A operação escala sem escalar os problemas humanos.

"O dinheiro não está no Claude Code, está na carteira das pessoas"

A ferramenta é commoditizada. Todo mundo pode usar Claude Code. O diferencial é: quem você conhece, qual problema você resolve, e quão rápido você entrega. Networking, distribuição, e execução vencem tecnologia. Sempre.

Este manual é um documento vivo. As lições continuam emergindo do cohort a cada semana. Os patterns de hoje são os fundamentos de amanhã.
