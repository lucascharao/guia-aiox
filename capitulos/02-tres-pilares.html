2.1 CLI First: Onde a Inteligência Vive

Existe um equívoco comum entre desenvolvedores que estão começando a trabalhar com sistemas de agentes de IA: a suposição de que o lugar mais importante do sistema é a interface gráfica, o painel bonito, o dashboard com gráficos coloridos. Afinal, é lá que as coisas parecem acontecer.

Essa suposição está errada.

No AIOX, existe uma hierarquia clara e inegociável sobre onde a inteligência do sistema reside. Essa hierarquia é chamada de CLI First, e ela diz o seguinte:

CLI → Observabilidade → Interface Gráfica

O terminal não é o lugar onde você acessa o sistema. O terminal é o sistema. A interface gráfica, quando existe, é apenas uma janela de visualização sobre o que o CLI já faz. A observabilidade — logs, métricas, dashboards — é um reflexo em tempo real do que acontece no CLI. E o CLI é o lugar onde toda decisão, toda execução, toda automação acontece.

Por que isso importa na prática?

Imagine dois desenvolvedores trabalhando no mesmo projeto. O primeiro, vamos chamá-lo de Alberto, constrói tudo em torno de uma interface gráfica. Sempre que precisa executar algo, abre o painel, clica em botões, monitora visualmente. O segundo, Beatriz, constrói tudo no CLI. Comandos, scripts, automações — tudo passa pelo terminal.

Seis meses depois, o projeto do Alberto tem dezenas de ações manuais que ninguém sabe como reproduzir em outro ambiente. "Você clica aqui, depois aqui, depois espera essa tela carregar..." — ninguém consegue automatizar o que vive numa interface gráfica. Beatriz, por sua vez, tem um repositório de scripts e comandos. Qualquer coisa que ela fez, pode ser repetida, versionada, auditada, e executada por qualquer agente de IA sem ambiguidade.

Esse é o ponto central do CLI First: automação é o destino de tudo que você constrói. E agentes de IA trabalham naturalmente com comandos e scripts — não com cliques em interfaces visuais.

Quanto Menos UI, Mais Poder a IA Tem

Pedro Valério resume o CLI First em uma frase que clarifica tudo: "Quanto menos UI, mais poder a IA tem."

Interfaces gráficas criam camadas de interpretação entre o comando e a execução. Cada botão, cada menu, cada confirmação visual é uma barreira que um agente de IA precisa contornar para executar uma tarefa. No CLI, essa barreira simplesmente não existe. O agente lê um comando, executa, e produz um resultado. Direto.

Comandos > Conversa: A Regra de Ouro

Existe uma regra de ouro no AIOX que precede qualquer outra: evite escrever para a IA — use comandos.

Quando você conversa livremente com um modelo de linguagem, ele sai do papel estruturado e começa a inventar. Cada palavra adicional que você escreve multiplica a busca da LLM por contexto, aumentando a probabilidade de alucinação e gastando tokens desnecessariamente.

Pedro Valério tem um princípio pessoal: "Eu não escrevo. Eu crio documento pra ele ler." Toda comunicação com agentes é via documentos estruturados e comandos padronizados — nunca conversa livre.

Há também o que Alan Nicolas chama de "Analogia do Elefante Rosa": se você diz para a IA "quero uma sala SEM elefante rosa", ela gera uma sala COM elefante rosa — porque você mencionou a palavra. Negações são perigosas em linguagem natural. Comandos estruturados eliminam essa armadilha.

Há um benefício econômico adicional: prompts repetidos ativam o cache do Claude Code, tornando-os significativamente mais baratos. Quando você padroniza seus comandos, está automaticamente otimizando custos.

O Terminal Como Centro de Comando

O terminal tem propriedades únicas que o tornam o ambiente natural para agentes de IA:

Composabilidade. Comandos podem ser encadeados, combinados, pipelines criados. Um agente pode executar aiox doctor | grep ERROR sem nenhuma fricção. A mesma operação em uma interface gráfica exigiria navegar por telas, copiar texto, colar em outro lugar.

Reprodutibilidade. Um comando digitado no terminal pode ser copiado, colado, modificado, versionado. A sessão pode ser gravada. O histórico pode ser auditado. Nada disso é possível com cliques em botões.

Versionamento. Scripts de CLI vivem em arquivos de texto. Arquivos de texto vivem em repositórios Git. Repositórios Git têm histórico, branches, revisões. Sua automação se torna parte do código.

Sem ambiguidade. Quando você diz para um agente "clique no botão azul no canto superior direito", você está criando um sistema frágil que quebra com qualquer mudança visual. Quando você diz "execute aiox graph --deps", você tem um comando preciso, sem ambiguidade, que funciona independente de resolução de tela, tema visual, ou estado da interface.

CLI Como Fonte da Verdade

No AIOX, o CLI não é apenas conveniente — é a fonte da verdade. Isso tem implicações concretas:

Qualquer estado relevante do sistema pode ser consultado via CLI. aiox doctor te diz se o ambiente está saudável. aiox graph --deps te mostra a árvore de dependências do projeto. aiox info te diz a versão, as configurações ativas, os agentes disponíveis. Você nunca precisa "abrir o painel" para saber o estado do sistema.

Qualquer operação relevante pode ser executada via CLI. Criar uma story? CLI. Validar uma story? CLI. Iniciar um ciclo de desenvolvimento? CLI. Fazer push para o repositório? CLI (via @devops). Se uma operação não tem um comando CLI correspondente, ela não é uma operação de primeira classe no sistema.

Qualquer automação começa no CLI. Quando você quer que um agente execute algo, ele precisa de um comando. Quando você quer que um workflow seja repetível, ele precisa de uma sequência de comandos. O CLI é o vocabulário que agentes e humanos compartilham.

Comandos Que Você Vai Usar Todo Dia

Dois comandos merecem destaque especial porque ilustram perfeitamente a filosofia CLI First:

aiox doctor — O diagnóstico completo do seu ambiente AIOX. Esse comando verifica se todas as dependências estão instaladas, se as configurações estão corretas, se os agentes estão acessíveis, se o projeto está em estado consistente. É o primeiro comando que você roda quando algo parece estranho. Com a flag --fix, ele tenta corrigir automaticamente os problemas encontrados. Um sistema saudável começa aqui.

aiox graph --deps — A visualização da árvore de dependências do projeto. Você pode exportar em ASCII para o terminal, em JSON para processamento programático, em HTML para abrir no browser como grafo interativo, em Mermaid para documentação, ou em DOT para o Graphviz. Com a flag --watch, o grafo se atualiza em tempo real enquanto você trabalha. Isso é observabilidade de CLI: você vê o sistema vivo, no terminal, sem precisar de um dashboard separado.

A Hierarquia na Prática

A hierarquia CLI → Observabilidade → UI não é uma regra estética. É uma regra operacional que define o que é prioridade de construção e de manutenção.

Quando você está construindo uma feature, a primeira pergunta é: "Isso tem um comando CLI?" Se não tem, crie o comando primeiro. Só depois, se necessário, construa uma interface visual por cima.

Quando você está depurando um problema, a primeira parada é o CLI. aiox doctor. aiox graph --stats. Os logs. Só se o CLI não revelar o problema você considera outras ferramentas.

Quando você está ensinando um colega a usar o sistema, você ensina os comandos. Não a interface. Porque a interface muda, melhora, é substituída. Os comandos permanecem.

2.2 Code First: Python Antes de IA

Se o CLI First define onde a inteligência do sistema vive, o Code First define quando você deve usar IA. E a resposta, que surpreende muita gente, é: muito menos do que você pensa.

O princípio Code First tem uma pergunta central, uma pergunta que você deve fazer antes de qualquer operação:

"O que dá pra fazer com Python?"

Só depois de responder essa pergunta, o que sobrar vai para a IA.

Python First → O que sobrar → Agente/LLM

Isso não é um detalhe técnico. É uma mudança de mentalidade fundamental. A maioria dos desenvolvedores que começa a trabalhar com IA comete o erro oposto: vai para a IA primeiro, e deixa Python para situações em que a IA não consegue ajudar. O AIOX inverte essa lógica.

Por Que Python Antes de IA?

A razão é simples: Python é determinístico. IA não é.

Quando você escreve len(lista) em Python, você sempre recebe o número correto de elementos. Sempre. Sem variação. Sem custo de token. Em milissegundos. Com 100% de reprodutibilidade.

Quando você pede para uma IA "conte quantos elementos tem nessa lista", você gasta tokens, espera segundos, e recebe uma resposta que provavelmente está correta — mas que pode variar dependendo do modelo, da versão, da temperatura configurada, e de fatores que você não controla.

Para operações determinísticas — contar, validar, transformar, gerar estruturas, parsear formatos — Python é sempre superior. Mais rápido, mais barato, mais confiável, mais auditável.

IA é superior apenas para operações que exigem julgamento, criatividade, ou compreensão semântica. Coisas que Python não consegue fazer bem: entender intenção implícita, gerar texto com voz consistente, mapear conceitos ambíguos, tomar decisões baseadas em contexto qualitativo.

A Tabela de Decisão

O AIOX tem uma tabela de decisão clara que você pode aplicar a qualquer sub-operação de qualquer tarefa:

A operação é... Use
Determinística (X sempre gera Y) Python
Repetitiva (mesma operação N vezes) Python com loop
Estrutural (criar pastas, arquivos, configs) Python
Parsing/transformação (JSON, YAML, CSV) Python
Requer julgamento humano IA
Requer criatividade IA
Requer contexto semântico IA
Requer compreensão de intenção IA
Híbrida Python faz o pesado, IA enriquece

Vamos aplicar essa tabela a um exemplo prático: você precisa criar a estrutura de um novo agente no AIOX, incluindo persona, comandos, memória, e configuração YAML.

Decompondo a tarefa:

  • Criar pastas e arquivos de esqueleto → Python. Estrutura determinística, repetível.
  • Validar o schema do YAML gerado → Python. Validação programática, sem ambiguidade.
  • Gerar o conteúdo da persona do agente → IA. Requer criatividade, voz, personalidade.
  • Escrever os comandos disponíveis para o agente → IA + Python. IA decide o vocabulário, Python formata e valida o YAML.
  • Calcular o número de agentes já existentes → Python. Contagem determinística.

Resultado: Python faz 60% do trabalho. IA faz 40%. E esse 40% é exatamente o que IA faz bem: conteúdo criativo, decisões semânticas, geração de linguagem natural.

Os Números Reais: 91,5% Determinístico

Se o princípio Code First ainda parece abstrato, existe um número concreto que o materializa. Pedro Valério compartilhou uma métrica de seus projetos em produção com AIOX:

91,5% das operações são determinísticas (scripts). 6,8% são executadas por modelo de linguagem. O resto é overhead.

Se você está gastando muito com API, algo está errado no seu processo. A proporção saudável de um projeto AIOX maduro é esmagadoramente Python — com IA entrando cirurgicamente nas operações que realmente exigem julgamento.

Determinismo em Camadas: Hooks, Skills e Scripts

O AIOX oferece diferentes mecanismos para automatizar operações, e cada um tem um grau diferente de determinismo:

Mecanismo Determinismo Velocidade Quando Usar
Hooks (bash) 100% ~5ms Operações automáticas em eventos (pre-commit, session-start)
Scripts Python 100% ~50ms Validação, transformação, scaffolding
Skills 50-80% Segundos Operações que integram LLM com estrutura
Agentes diretos Variável Segundos Operações criativas e semânticas

A recomendação prática: comece com hooks e scripts. Só escale para skills quando Python não resolver. Só chame agente direto quando a skill não cobrir.

Os Anti-Padrões

Existem operações que você nunca deve delegar à IA no AIOX. Memorize esses anti-padrões:

Nunca use LLM para contar. Contar linhas de código, contar arquivos em um diretório, contar elementos em uma lista, contar palavras em um documento. Python conta. IA não precisa.

Nunca use LLM para validar schema. Se um YAML está válido, Python com pyyaml resolve em milissegundos com 100% de precisão. Se um JSON está bem formatado, json.loads() resolve. IA pode validar schema — mas vai custar tokens, demorar mais, e introduzir variação desnecessária.

Nunca use LLM para rename/replace em massa. Renomear variáveis, substituir strings, formatar código — tudo isso é operação determinística. Python com expressões regulares, ou ferramentas como sed, resolvem instantaneamente.

Nunca use LLM para gerar estrutura de pastas. A estrutura de um projeto AIOX é definida em templates. Python cria essas estruturas baseado nos templates. IA não precisa "decidir" onde cada pasta vai.

Nunca use LLM para qualquer coisa que if/else resolve. Se a lógica é condicional e determinística, é código. Ponto.

Nunca pule a pergunta "o que Python faz aqui?" Antes de chamar qualquer agente, antes de montar qualquer prompt, antes de gastar qualquer token — faça a pergunta. A resposta pode te economizar minutos, dinheiro, e inconsistências.

2.3 A Economia de Tokens

O princípio Code First não é apenas filosófico. Ele tem um impacto financeiro real e mensurável. Para entender por que, você precisa entender como tokens funcionam economicamente.

Cada palavra que você envia para um modelo de linguagem tem um custo. Cada palavra que você recebe de volta também tem um custo. Esse custo é medido em tokens — unidades que correspondem, aproximadamente, a quatro caracteres de texto em inglês (um pouco mais em português, devido a caracteres especiais e palavras mais longas).

Em projetos típicos de desenvolvimento com IA, o custo mensal de tokens gira entre $30 e $100 para desenvolvedores individuais. Parece pouco? Multiplique por doze meses. Por cinco desenvolvedores. Por dez projetos simultâneos.

O Custo Real de Uma Operação Determinística via IA

Vamos fazer uma conta simples. Suponha que você precisa validar se 50 arquivos YAML estão no formato correto.

Via Python: - Tempo de execução: ~200ms para todos os 50 arquivos - Custo de tokens: zero - Confiabilidade: 100% - Variabilidade: zero

Via IA (Claude, GPT-4): - Você precisa enviar cada arquivo como contexto - Um YAML típico de configuração tem ~200 tokens - 50 arquivos = ~10.000 tokens de entrada - A resposta de cada validação tem ~50 tokens = 2.500 tokens de saída - Total: ~12.500 tokens por ciclo de validação - Custo estimado (Claude Sonnet): ~$0.04 por ciclo - Se você valida 10 vezes por dia durante um mês: ~$12/mês apenas nessa operação - Tempo de execução: vários segundos por arquivo - Confiabilidade: ~95% (pode haver falsos positivos/negativos)

A diferença não está apenas no dinheiro. Está na latência. Está na variabilidade. Está na auditabilidade. Uma validação Python produz um erro preciso: linha 47: campo 'name' é obrigatório, mas não foi encontrado. Uma validação via IA pode produzir "o arquivo parece ter um problema no campo de nome" — menos preciso, menos acionável.

Os Números do Projeto Típico

Com a aplicação sistemática do princípio Code First, os números de economia que o AIOX observa em projetos reais são:

  • ~$45/mês de economia por desenvolvedor individual
  • ~$540/ano de economia por desenvolvedor individual
  • ~70% de redução no número de tokens consumidos em operações de scaffolding e validação
  • ~10x de melhora na latência de operações que antes iam para IA

Mas o benefício mais subestimado não é o dinheiro — é a reprodutibilidade. Cada operação que vai para Python em vez de IA é uma operação que você pode executar em CI/CD, em ambientes sem API key, em máquinas sem internet, a qualquer hora, sem limite de rate, sem variação de output.

O Valor Real do Claude Max

Para colocar a economia de tokens em perspectiva prática: uma assinatura do Claude Max a $200/mês equivale a aproximadamente $2.500 a $3.500 em créditos de API. Esse é o valor real que você está recebendo.

A tentação natural quando se atinge o limite é ativar o "uso extra". Não faça isso. O uso extra cobra por token a taxas de API — eliminando a vantagem econômica do Max. A recomendação do cohort é clara: compre uma segunda conta Max de $200 em vez de ativar uso extra. Duas contas Max custam $400/mês e dão ~$5.000-7.000 em tokens. O uso extra de uma conta pode facilmente ultrapassar isso.

Os Vazamentos Silenciosos de Tokens

Existem custos de token que muitos operadores não percebem porque não aparecem nas operações explícitas:

Servidores MCP consomem contexto passivamente. Cada MCP registrado no seu ambiente consome entre 8% e 30% da janela de contexto apenas por estar registrado — mesmo que você não use nenhuma ferramenta dele. A solução: desabilite MCPs que não está usando ativamente.

Auto-compact consome ~20.000 tokens por sessão. O recurso de compactação automática do Claude Code gasta tokens para resumir o contexto. Desabilitar o auto-compact e usar /clear estrategicamente entre fases é mais econômico.

/clear é melhor que /compact. O /compact tenta preservar contexto, mas o resumo nem sempre captura o que importa. O /clear zera tudo — e quando combinado com uma story bem escrita que pode ser relida, produz resultados mais limpos.

Claude Teams custa ~7x mais tokens que o uso individual, com 44% de perda de contexto em cenários de equipe. Para a maioria dos operadores AIOX, contas individuais são mais eficientes.

YAML é 15-20% mais barato que JSON em tokens. Isso pode parecer marginal, mas em projetos com centenas de arquivos de configuração processados diariamente, a economia se acumula. É mais um motivo para a hierarquia de formatos Markdown > YAML > JSON do Capítulo 5.

Justificando Cada Token

O AIOX opera com uma filosofia de justificativa de token: todo token consumido deve justificar seu custo. Isso significa que antes de chamar qualquer agente, você deve ser capaz de responder:

  1. Essa operação realmente precisa de julgamento semântico?
  2. Python não consegue fazer isso?
  3. O resultado de IA será melhor que o resultado de código?

Se a resposta às três perguntas for "sim", use IA. Se qualquer resposta for "não", use código.

Esse princípio não é uma restrição que limita o uso de IA. É uma prática que torna o uso de IA mais valioso — porque quando você finalmente chama um agente, é para fazer algo que realmente justifica a inteligência artificial.

2.4 Story-Driven Development

O terceiro pilar do AIOX é o que dá estrutura ao trabalho em si: o Story-Driven Development, ou Desenvolvimento Orientado a Stories.

Para entender por que isso importa, considere duas formas de pedir a mesma coisa para um agente de IA:

Forma 1 (sem estrutura): "Me faz um login."

Forma 2 (story-driven):

Story 2.1 — Autenticação de Usuário
Contexto: Sistema precisa de login seguro com email/senha
Critérios de aceite:
  - Formulário valida email e senha antes de enviar
  - Senha tem mínimo 8 caracteres
  - Token JWT gerado com expiração de 24h
  - Falha após 5 tentativas erradas bloqueia por 15min
Dependências: Story 1.3 (Schema de usuário), Story 1.4 (API base)
Status: Ready
Branch: feat/auth-login

A primeira forma vai produzir código genérico que não segue os padrões do seu projeto, não tem os critérios de segurança que você precisa, e pode ou não ser compatível com o que você já tem. Você vai gastar horas ajustando.

A segunda forma vai produzir exatamente o que você precisa, na primeira tentativa, porque o agente tem contexto preciso, critérios mensuráveis, e sabe o que depende de quê.

A Story Como Unidade Atômica de Trabalho

No AIOX, a story é a unidade atômica de trabalho. Isso significa:

  • Todo desenvolvimento começa com uma story
  • Nenhum agente começa a implementar sem uma story validada
  • O progresso é rastreado na story (checkboxes que vão de [ ] para [x])
  • As mudanças de arquivo são registradas na story
  • O status da story define em qual fase do ciclo o trabalho está

Isso não é burocracia. É rastreabilidade. É a diferença entre um projeto onde você sabe exatamente o que foi feito, por quem, em que ordem, com quais decisões — e um projeto onde você tem um monte de código e ninguém sabe por onde começar para entender o que aconteceu.

O Ciclo de 5 Etapas

O ciclo completo de uma story no AIOX tem cinco etapas:

1. Criar (@sm — River) O Scrum Master cria a story a partir do epic e do PRD. A story nasce com status Draft e contém: título, contexto, critérios de aceite, dependências, e tarefas. Nenhuma linha de código é escrita aqui.

2. Validar (@po — Pax) O Product Owner valida a story contra um checklist de 10 pontos. Só com pontuação mínima de 7/10 a story avança. Esse gate existe para garantir que ninguém gasta tempo implementando algo mal especificado.

3. Implementar (@dev — Dex) O desenvolvedor implementa a story exatamente conforme os critérios de aceite. Nada a mais, nada a menos. O status muda de Ready para InProgress. Os checkboxes da story são marcados conforme as tarefas são completadas.

4. Testar (@qa — Quinn) O QA verifica sete pontos de qualidade: testes passando, lint limpo, type checking, critérios de aceite satisfeitos, sem regressões, documentação atualizada, e nada fora do escopo implementado. A decisão é PASS, CONCERNS, FAIL, ou WAIVED.

5. Entregar (@devops — Gage) Somente o Gage tem autoridade para fazer push e criar Pull Requests. Isso garante que nada vai para o repositório sem passar pelo ciclo completo.

Por Que "Me Faz Um Login" Falha

Voltando ao exemplo inicial: por que a instrução "me faz um login" falha sistematicamente?

Porque ela viola três dos cinco princípios do ciclo:

Não há contexto. O agente não sabe se é um login web ou mobile, qual é o stack tecnológico, quais são os padrões de segurança da empresa, se já existe um sistema de autenticação que precisa ser integrado.

Não há critérios de aceite. O agente não sabe o que significa "login pronto". Dois campos e um botão? JWT com refresh token? OAuth com Google? 2FA obrigatório? Sem critérios, o agente inventa — e a invenção raramente corresponde ao que você precisava.

Não há rastreabilidade. Quando você precisar voltar a esse login daqui a três meses para adicionar 2FA, como você vai saber o que foi implementado, quais decisões foram tomadas, o que pode ser modificado sem quebrar outras coisas? A story responde essas perguntas. O prompt solto não.

A Story Como Checkpoint de Contexto

A story tem uma função operacional que vai além da rastreabilidade: ela serve como checkpoint de contexto.

Quando a janela de contexto do Claude é compactada ou zerada com /clear, o agente perde a memória da sessão. Mas a story permanece — ela é um arquivo no disco. Basta o agente reler a story para recuperar instantaneamente: o que está sendo feito, quais critérios devem ser atendidos, quais tarefas já foram completadas (checkboxes [x]), e quais decisões foram tomadas.

Na prática, isso significa que uma story bem escrita torna o /clear seguro. Você pode zerar o contexto quantas vezes precisar durante uma implementação longa sem perder o fio da meada — desde que a story esteja atualizada.

Traceabilidade Como Superpoder

Um projeto story-driven tem uma propriedade que projetos ad-hoc nunca têm: traceabilidade completa.

Você consegue responder: - "Quando foi implementado o sistema de autenticação?" → Story 2.1, commit abc123, data X. - "Por que escolhemos JWT em vez de sessions?" → Decisão registrada na story 2.1, critério de aceite C3. - "Quais stories dependem da Story 2.1?" → Qualquer story que lista 2.1 na seção de dependências. - "O que mudou nessa sprint?" → Todas as stories com status Done criadas nesse período.

Isso não é documentação retroativa. É rastreabilidade que acontece automaticamente como subproduto do processo. Cada story é uma decisão registrada, um trabalho auditado, uma unidade de progresso mensurável.

Exercício do Capítulo 2

Objetivo: Aplicar o princípio Code First decompondo uma tarefa real.

Tarefa de exemplo: "Processar 100 transcrições de podcast, extrair timestamps, identificar os principais temas de cada episódio, e gerar um índice em YAML."

Sua missão é decompor essa tarefa em sub-operações e classificar cada uma como Python ou IA:

  1. Leia a instrução completa da tarefa.
  2. Quebre em sub-operações (mínimo de 8).
  3. Para cada sub-operação, responda: "Python resolve isso?" com SIM ou NÃO e por quê.
  4. Estime o custo de tokens de delegar cada sub-operação à IA.
  5. Estime o tempo de desenvolvimento de cada sub-operação como script Python.
  6. Calcule a economia total (tokens e tempo) de fazer Code First.
