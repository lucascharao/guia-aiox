4.1 A Pirâmide: Cinco Camadas de Abstração

Todo sistema de software resolve o problema da complexidade da mesma forma: através de abstração. Você agrupa operações simples em unidades mais complexas, que por sua vez se agrupam em unidades ainda mais complexas, até que operações que seriam impossíveis de gerenciar manualmente se tornam expressáveis em uma linha.

O AIOX resolve o problema da complexidade de desenvolvimento com IA através de uma pirâmide de cinco camadas:

Workflows
    ↑
  Agents
    ↑
  Skills
    ↑
  Tasks
    ↑
Commands

A analogia mais precisa para entender essa hierarquia é a linguagem escrita:

  • Commands são as letras. Unidades básicas, sem significado próprio, mas necessárias para tudo.
  • Tasks são as palavras. Combinações de commands que fazem algo específico e completo.
  • Skills são as frases. Agrupamentos de tasks com contexto de quando e como usá-las.
  • Agents são os parágrafos. Conjunto de skills com identidade, autoridade, e memória.
  • Workflows são o texto completo. Sequências de agentes executando tasks em ordem para produzir um resultado maior.

Uma letra sozinha não diz nada. Uma palavra tem significado. Uma frase tem contexto. Um parágrafo tem argumento. Um texto completo tem narrativa.

Da mesma forma: um comando sozinho é uma instrução. Uma task é um processo completo. Uma skill é um conjunto de capacidades relacionadas. Um agente é uma entidade com identidade e responsabilidade. Um workflow é um ciclo de desenvolvimento completo do início ao fim.

Cada camada depende das camadas abaixo e serve às camadas acima. Você não pula camadas — mas quando trabalha nas camadas superiores, não precisa pensar nas inferiores. É exatamente como escrever: você pensa em argumentos e parágrafos, não em letras e palavras individuais.

4.2 Commands: O Vocabulário Básico

Commands são o nível mais baixo da hierarquia — mas não subestime sua importância. São eles que tornam o sistema interativo e determinístico na ponta.

O Prefixo *

Todos os commands do AIOX usam o prefixo *. Esse prefixo não é decorativo: ele é um marcador semântico que diz "isso é um comando executável para o agente ativo, não uma instrução conversacional."

A distinção importa. Quando você diz para um agente "me ajuda a criar uma story", está tendo uma conversa. Quando você diz *create-story, está executando um comando. A conversa pode levar a qualquer lugar. O comando tem um comportamento definido, inputs esperados, outputs documentados, e pré-condições verificadas.

É a diferença entre pedir para alguém "fazer o que acharem melhor" e executar uma instrução de máquina.

Commands Fundamentais

*help — O primeiro command que qualquer usuário deve conhecer. Exibe a lista completa de commands disponíveis para o agente atualmente ativo, com descrição breve de cada um. Cada agente tem um *help diferente, porque cada agente tem commands diferentes. @dev *help mostra commands de desenvolvimento. @qa *help mostra commands de qualidade. É a porta de entrada para qualquer agente.

*create-story — Inicia o processo de criação de uma nova story a partir do contexto do epic ativo. Ao contrário do que o nome pode sugerir, esse command não escreve a story inteiro autonomamente — ele elicita as informações necessárias (contexto, critérios de aceite, dependências) e produz um arquivo de story formatado corretamente. Disponível para @sm.

*task {nome} — Executa uma task específica pelo nome. Este é o command que conecta o nível de Commands ao nível de Tasks: você está explicitamente solicitando a execução de uma task definida no sistema. Por exemplo, *task dev-develop-story executa a task de desenvolvimento completa conforme definida em .aiox-core/development/tasks/dev-develop-story.md.

*exit — Sai do modo agente e retorna ao contexto padrão. Importante para sessões longas onde você alternava entre agentes e precisa encerrar o contexto de um agente específico. Também aciona o protocolo de handoff, gerando o artefato de compactação do contexto do agente que está sendo encerrado.

O Papel do Front Matter

Antes de avançar para as camadas superiores, vale entender um conceito que permeia todas elas: o Front Matter.

Front Matter é um bloco YAML entre --- no início de um arquivo Markdown. Parece simples, mas é um dos conceitos mais importantes do AIOX — e frequentemente subestimado por iniciantes.

---
status: Ready
epic: EPIC-03
executor: "@dev"
model: sonnet
---

# Conteúdo do arquivo aqui...

Por que o Front Matter importa? Porque é a primeira coisa que a LLM lê. Quando um agente abre um arquivo, o Front Matter define imediatamente o contexto: que tipo de entidade é, em que estado está, quem é responsável, qual modelo usar. Sem Front Matter, o agente precisa ler o arquivo inteiro para inferir o contexto — gastando tokens e introduzindo ambiguidade.

O Front Matter é o metadado que torna documentos legíveis tanto por humanos quanto por máquinas. Uma story, uma task, um agente — todos usam Front Matter como sua "identidade" estruturada.

Uma dica prática do cohort: use Obsidian para se familiarizar com Front Matter. O Obsidian renderiza Front Matter de forma visual e ajuda a internalizar o conceito antes de aplicá-lo no AIOX.

Commands São Determinísticos

Uma propriedade fundamental dos commands é que eles são determinísticos: o mesmo command, no mesmo estado do sistema, sempre produz o mesmo comportamento. Não há variação, não há interpretação, não há criatividade.

Isso é intencional. Commands são a camada onde o AIOX garante previsibilidade. A criatividade e o julgamento existem nas camadas superiores — nos agentes e workflows — mas a execução básica é sempre determinística.

4.3 Tasks: A Unidade de Trabalho Executável

Se commands são o vocabulário, tasks são as palavras. Uma task é uma unidade de trabalho completa: ela tem um propósito claro, inputs definidos, outputs esperados, pré-condições verificadas, pós-condições validadas, e modos de execução configuráveis.

Anatomia de Uma Task

Cada task no AIOX é definida em um arquivo Markdown em .aiox-core/development/tasks/. Esse arquivo não é apenas documentação — é a especificação executável da task. O formato inclui:

Inputs: O que a task precisa para começar. Podem ser arquivos específicos (a story ativa), estados do sistema (branch em Git), ou informações fornecidas pelo usuário.

Outputs: O que a task produz quando concluída com sucesso. Arquivos criados, estados modificados, relatórios gerados.

Pré-condições: O que deve ser verdadeiro antes da task começar. A story deve estar em status Ready. O branch correto deve estar ativo. As dependências listadas na story devem estar completas.

Pós-condições: O que deve ser verdadeiro depois que a task termina. Os testes devem passar. O lint deve estar limpo. Os checkboxes da story devem estar marcados.

Modos de execução: Como a task deve se comportar em diferentes contextos. Ver abaixo.

Os Três Modos de Execução

Tasks com complexidade significativa suportam três modos de execução:

Modo Interactive: O agente pausa em pontos de decisão para consultar o usuário. Ideal quando você está aprendendo o sistema, quando o contexto é ambíguo, ou quando as decisões têm impacto significativo. O agente apresenta opções, explica trade-offs, e aguarda confirmação antes de prosseguir. Mais lento, mais seguro para quem está começando.

Modo YOLO: O agente executa o máximo possível de forma autônoma, tomando decisões razoáveis sem pausar. Ideal para developers experientes que conhecem os padrões do projeto, para automações em CI/CD, ou para trabalho em condições de tempo crítico (como um hotfix de produção). Mais rápido, requer confiança no agente e no contexto.

Modo Pre-Flight: O agente analisa a task completamente e produz um plano de execução — sem executar nada. Ideal para revisão antes de operações de alto impacto, para entender o que vai acontecer antes que aconteça, e para identificar potenciais problemas antes de comprometer mudanças.

A escolha de modo não é permanente. Você pode iniciar em Pre-Flight para revisar o plano, depois executar em Interactive para as partes críticas, depois YOLO para as partes rotineiras.

"Task Validada É Lei"

Essa frase — "task validada é lei" — é um princípio central do AIOX que vale aprofundar.

Quando uma task é definida, validada, e publicada no sistema, ela representa o consenso coletivo sobre a melhor forma de executar aquele tipo de trabalho. Ela incorpora decisões arquiteturais, lições aprendidas, restrições de qualidade, e padrões do time.

Dizer que a task validada é lei significa que nenhum agente, por mais experiente que seja a pessoa interagindo com ele, pode decidir unilateralmente "vou fazer diferente desta vez porque faz mais sentido". Fazer diferente requer modificar a definição da task — o que é um processo deliberado, documentado, e revisado.

Isso pode parecer rígido. Na prática, é libertador: quando você confia que a task define o comportamento correto, você não precisa pensar sobre "será que estou fazendo da forma certa?" a cada execução. A forma correta está na task. Execute e confie.

4.4 Skills: O Agrupamento Contextual

Skills são o nível onde tasks ganham contexto. Uma skill é um conjunto de tasks relacionadas, com a informação de quando usá-las, como sequenciá-las, e por que elas pertencem juntas.

A distinção entre task e skill é sutil mas importante — e frequentemente confundida:

Uma task sabe como fazer algo e transforma o status de uma entidade. dev-develop-story.md sabe como implementar uma story seguindo todos os critérios de qualidade, e ao fazê-lo muda o status da story de Ready para InProgress. Toda task mapeia um processo global do sistema.

Uma skill sabe quando fazer algo e como combinar tasks relacionadas. Skills integram no Claude Code como capacidades instaláveis (filosofia Matrix — "eu sei kung fu"), mas não são mapeamento de processo global. Como Alan Nicolas enfatiza: "Sua empresa não pode ser Claude Code." Skills são ferramentas dentro do sistema, não o sistema em si. A skill de desenvolvimento do @dev sabe que antes de implementar você revisa o contexto, que depois de implementar você atualiza a story, que em certos contextos você precisa criar branches específicos, que em outros você faz commit atomicamente por task.

Se tasks são as instruções individuais de uma receita, skills são o capítulo completo — com as instruções, os contextos de substituição, os timings, e as notas do chef sobre quando adaptar.

Skills Como Encapsulamento de Expertise

Uma das propriedades mais poderosas das skills é que elas encapsulam expertise acumulada. Quando um time trabalha junto por meses, eles desenvolvem padrões implícitos: "quando fazemos isso, sempre verificamos aquilo antes". Esses padrões vivem nas skills.

Em equipes humanas, esse conhecimento fica na cabeça de pessoas-chave e é transmitido por osmose. Quando a pessoa sai, o conhecimento vai junto. Com skills documentadas e versionadas, o conhecimento é explícito, transferível, e evoluível.

Um novo membro do time — humano ou agente — que aprende as skills disponíveis está absorvendo o conhecimento acumulado de todos que trabalharam no sistema antes.

4.5 Agents: Persona, Skills, Autoridade, e Memória

Agentes são onde a magia acontece. Um agente não é apenas um conjunto de skills com um nome diferente. É uma entidade com quatro propriedades que, juntas, criam algo maior do que a soma das partes:

Persona: A identidade e o estilo de comunicação do agente. Não é cosmético — é funcional. A persona define como o agente interpreta contexto ambíguo, como ele prioriza quando há conflitos, e como ele comunica incertezas. Dex (@dev) tem uma persona de desenvolvedor focado em qualidade de código e clareza de implementação. Quinn (@qa) tem uma persona de revisor rigoroso mas construtivo. Essas diferenças de persona produzem comportamentos diferentes em situações idênticas.

Skills: O conjunto de capacidades do agente. Cada agente tem um conjunto específico de skills que define o que ele pode e sabe fazer. As skills de Dex incluem desenvolvimento de features, refatoração, debugging, e escrita de testes. As skills de Aria (@architect) incluem análise de complexidade, design de sistemas, e avaliação de trade-offs arquiteturais. Skills sem sobreposição excessiva garantem que cada agente tem um domínio claro.

Autoridade: As operações que o agente pode executar, e os limites que ele não pode cruzar. Isso é o que o Artigo II da Constitution protege. A autoridade define tanto os poderes exclusivos (o que só esse agente pode fazer) quanto os bloqueios (o que esse agente explicitamente não pode fazer). Gage (@devops) tem autoridade exclusiva sobre git push. Aria tem autoridade de decisão sobre arquitetura de sistema. Dex pode commitar mas não fazer push.

Memória: O arquivo MEMORY.md de cada agente. Onde informações relevantes sobre o projeto são acumuladas ao longo do tempo — decisões arquiteturais tomadas, padrões estabelecidos, problemas encontrados e soluções adotadas, preferências do cliente. A memória é o que torna um agente progressivamente mais útil: quanto mais contexto acumula, menos repetição você precisa fazer.

Por Que Persona Importa

A persona é frequentemente subestimada como "apenas estética". Não é.

Considere um conflito comum: o @dev identificou uma forma mais elegante de implementar algo, mas essa forma diverge levemente dos critérios de aceite da story. Dois agentes com personas diferentes vão reagir diferente:

Um agente sem persona consistente vai tomar uma decisão arbitrária — talvez implementar a forma elegante, talvez não, dependendo de variáveis não controláveis.

Dex, com sua persona de desenvolvedor focado em clareza e nos critérios de aceite, vai: implementar exatamente o que foi especificado, e registrar na story uma observação: "Identifiquei uma abordagem alternativa que pode ser mais elegante para uma refatoração futura — ver nota na seção de decisões técnicas."

Isso não é serendipidade. É comportamento previsível derivado de uma persona consistente. Persona cria previsibilidade de comportamento, o que é exatamente o que você precisa em um sistema automatizado.

4.6 Workflows: A Sequência Que Produz Resultado

Workflows são o topo da pirâmide — onde tasks executadas por agentes em sequência produzem resultados que nenhum agente poderia produzir sozinho.

O Princípio Task-First

Um conceito crucial sobre workflows no AIOX: workflows são compostos por tasks conectadas, não por agentes conectados. Isso é o Task-First Principle.

A distinção pode parecer sutil, mas tem implicações profundas. Se você pensa em workflows como "agentes em sequência", você cria dependência de personas: "preciso do Pax para validar, depois do Dex para implementar, depois do Quinn para revisar." Se um agente não está disponível, o workflow para.

Se você pensa em workflows como "tasks em sequência", os agentes são executores — mas o que define o processo são as tasks. A task validate-next-story.md define o que validação de story significa. Quem executa pode ser @po, pode ser @aiox-master em ausência do po, pode até ser um humano que segue o checklist manualmente. O processo é o mesmo.

Tasks são o coração do sistema. Agentes são os braços.

Os 4 Workflows Primários

1. Story Development Cycle (SDC) — O Workflow Principal

O ciclo completo de desenvolvimento de uma story, do zero à entrega. Quatro fases:

  • Criar (@sm): A story nasce como Draft.
  • Validar (@po): A story é avaliada em 10 pontos. Com score ≥ 7, avança para Ready.
  • Implementar (@dev): A story sai de Ready, passa por InProgress. Código é escrito.
  • Qualidade (@qa): A story entra em InReview. Sete verificações. PASS → Done.

Ao final do ciclo, @devops recebe a story Done e faz o push para o repositório.

2. QA Loop — Revisão Iterativa

Quando o QA Gate retorna REJECT em vez de PASS, o QA Loop entra em ação: @qa aponta os problemas, @dev corrige, @qa revisa novamente. Máximo de 5 iterações. Se após 5 iterações ainda não passou, o loop escalona para @aiox-master. É o mecanismo de auto-cura do ciclo de desenvolvimento.

3. Spec Pipeline — Especificação Antes da Implementação

Para features complexas que precisam ser especificadas antes de virar stories, o Spec Pipeline transforma requisitos informais em especificação executável. Seis fases: coletar requisitos, avaliar complexidade, pesquisar, escrever spec, revisar com critique, planejar implementação. Para features simples (score ≤ 8), três das seis fases são puladas.

4. Brownfield Discovery — Avaliação de Projeto Existente

Quando você está entrando em um projeto existente e precisa entender a dívida técnica antes de começar a trabalhar, o Brownfield Discovery mapeia o sistema atual em dez fases, envolvendo arquiteto, engenheiro de dados, especialista de UX, e QA — culminando em um relatório executivo de dívida técnica e um epic pronto para o desenvolvimento.

Por Que Quatro Workflows?

Não é por acaso que o AIOX tem exatamente esses quatro workflows. Eles cobrem os quatro cenários fundamentais de qualquer projeto de software:

  • Novo trabalho em projeto ativo → SDC
  • Trabalho que não passou na qualidade → QA Loop
  • Feature complexa que precisa ser especificada → Spec Pipeline
  • Entrada em projeto existente → Brownfield Discovery

Todo trabalho de desenvolvimento que você vai fazer cabe em um desses quatro cenários. E para cada cenário, há um workflow com tasks definidas, agentes designados, gates de qualidade, e critérios de conclusão.

Exercício do Capítulo 4

Objetivo: Internalizar a hierarquia de primitivas relacionando-a com situações reais.

Parte 1: Desenhe a pirâmide

Em papel (não no computador — o ato físico de desenhar cria memória diferente), desenhe a pirâmide das cinco camadas. Para cada camada, escreva: - O nome da camada - A analogia de linguagem (letra, palavra, frase, parágrafo, texto) - Um exemplo concreto do AIOX (um command, uma task, uma skill, um agente, um workflow)

Parte 2: Mapeie um exemplo do cotidiano

Pense em um processo do seu dia-a-dia profissional — pode ser onboarding de um novo colaborador, processo de code review, ou qualquer sequência de trabalho que você repete. Decomponha-o nas cinco camadas:

  • Commands: Quais são as instruções atômicas básicas? (checar email, abrir arquivo, rodar teste)
  • Tasks: Quais são as unidades de trabalho completas? (revisar PR, criar branch, escrever documentação)
  • Skills: Quais grupos de tasks um profissional específico sabe fazer? (desenvolvedor backend, QA, tech lead)
  • Agents: Quem são as personas envolvidas, com seus escopos e autoridades?
  • Workflow: Qual é a sequência completa do processo, do início ao fim?

Parte 3: Identifique o nível do seu problema atual

Pense em um problema recorrente no seu trabalho com IA. Em qual nível da pirâmide o problema está?

  • Se é "não sei como pedir para o agente fazer X" → problema de Commands
  • Se é "o agente faz coisas inconsistentes na mesma situação" → problema de Tasks
  • Se é "o agente não tem contexto suficiente" → problema de Skills
  • Se é "não sei qual agente usar" → problema de Agents
  • Se é "o processo tem muito re-trabalho" → problema de Workflows

Identificar o nível correto do problema é o primeiro passo para resolvê-lo no lugar certo. Soluções no nível errado não funcionam: você não resolve um problema de Workflow adicionando mais Commands.
