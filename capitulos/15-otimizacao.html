15.1 A Anatomia do Custo

Usar IA com método significa entender o que custa. Não de forma obsessiva — mas com a mesma consciência que um engenheiro tem sobre a complexidade computacional do código que escreve.

O custo de um sistema AIOX tem três componentes:

Componente 1: Custo de Input (Tokens Enviados)

Cada mensagem que você envia para um modelo inclui: o contexto do sistema (personas, regras, memória), o histórico da conversa, e a instrução atual. O total de tokens de input é a soma de todas essas partes.

O maior inimigo do custo de input é o contexto não gerenciado. Quando uma sessão cresce sem controle, o contexto acumula: personas antigas, histórico de conversas de fases anteriores, memória carregada desnecessariamente. Cada mensagem nova carrega todo esse peso.

Componente 2: Custo de Output (Tokens Gerados)

Tokens de output custam mais que tokens de input (tipicamente 3-5x). Um agente que responde com análises longas e detalhadas quando uma resposta concisa seria suficiente é um agente que custa mais do que precisa.

A otimização aqui é de instrução: personas bem escritas incluem orientações sobre verbosidade esperada. "Quando analisando mercado, produza análise em 400-600 palavras focada em insights acionáveis, sem contextualização genérica" é uma instrução que afeta o custo de output.

Componente 3: Custo Operacional (Infraestrutura e Tempo)

Para squads em produção (Fase 3+), há custos de infraestrutura: servidores, banco de dados, Redis para cache. Esses custos são geralmente menores que os custos de tokens de LLM, mas crescem com o uso.

A Regra dos 80/20 de Custo

Em projetos típicos, 80% dos tokens são consumidos por 20% das operações. Identificar e otimizar essas operações de alto custo tem impacto desproporcional no custo total.

As operações de alto custo mais comuns: - Análises de código em arquivos grandes (o arquivo inteiro é passado como contexto) - Sínteses de múltiplos inputs (o contexto combina múltiplas análises) - Iterações de QA Loop onde o contexto acumula histórico de cada iteração - Brownfield Discovery onde o contexto inclui todo o assessment progressivo

Os Vazamentos Silenciosos

Existem custos que a maioria dos operadores não percebe:

Servidores MCP consomem contexto passivamente. Cada MCP registrado no seu ambiente consome entre 8% e 30% da janela de contexto apenas por estar registrado — mesmo sem usar nenhuma ferramenta. Solução: desabilite MCPs que não está usando ativamente.

Claude Teams custa ~7x mais tokens que uso individual, com 44% de perda de contexto em cenários de equipe. Para a maioria dos operadores AIOX, contas individuais são mais eficientes.

O valor real da assinatura: Claude Max a $200/mês equivale a ~$2.500-3.500 em créditos de API. Nunca ative "uso extra" — compre uma segunda conta Max em vez disso. Duas contas = ~$5.000-7.000 em tokens por $400.

A regra dos 91,5%: Pedro Valério compartilhou uma métrica de produção — 91,5% das operações são determinísticas (scripts), 6,8% por modelo. Se você está gastando muito com API, algo está errado no seu processo.

15.2 Code First Avançado: Maximizando a Proporção Determinística

O princípio Code First foi introduzido no início do livro. Na otimização avançada, a pergunta não é "o que Python pode fazer?" — é "o que Python pode fazer que eu ainda estou deixando para a IA?"

Audit de Operações

Faça uma lista de todas as operações que acontecem em um ciclo típico de development no seu projeto. Para cada operação, classifique honestamente: Python ou IA necessária.

Operações que frequentemente acabam com IA quando deveriam ser Python:

Verificação de completude de story: "Todos os ACs estão marcados como [x]?" é uma operação de parsing de Markdown. Python resolve em 3 linhas.

def count_ac_completion(story_path: str) -> tuple[int, int]:
    """Retorna (completos, total) de ACs na story."""
    with open(story_path) as f:
        content = f.read()
    completed = content.count("- [x] AC-")
    total = content.count("- [") - content.count("- [ ]")
    # Método mais robusto com regex
    import re
    ac_pattern = re.findall(r'- \[(x| )\] AC-', content)
    completed = sum(1 for status, _ in [m.split('] AC-') for m in
                    re.findall(r'- \[.\] AC-\d+', content)]
                    if '[x]' in m)
    return completed, len(ac_pattern)

Geração de relatório de progresso de épico: O EPIC-EXECUTION.yaml tem todos os dados necessários. Python pode gerar o relatório de progresso sem IA:

import yaml

def generate_epic_report(epic_yaml_path: str) -> str:
    with open(epic_yaml_path) as f:
        epic = yaml.safe_load(f)

    stories = epic.get('stories', [])
    done = [s for s in stories if s['status'] == 'Done']
    in_progress = [s for s in stories if s['status'] == 'InProgress']
    blocked = [s for s in epic.get('blockers', []) if s['status'] != 'resolved']

    pct = len(done) / len(stories) * 100 if stories else 0

    return f"""
## {epic['epic']['title']}

Progresso: {len(done)}/{len(stories)} stories ({pct:.1f}%)
Em progresso: {len(in_progress)} stories
Blockers ativos: {len(blocked)}
"""

Validação de schema de story: Verificar se uma story tem todos os campos obrigatórios é validação de estrutura — Python com qualquer schema validator.

Extração de métricas de QA: O arquivo qa/loop-status.json tem todos os dados. Python pode gerar dashboard de QA sem IA.

O Mapa de Economia

Para cada operação que você migra de IA para Python, estime: - Frequência por semana - Tokens médios por operação - Custo semanal atual - Custo semanal após migração (R$0)

Em projetos ativos, essa análise frequentemente revela R$50-200/mês em operações que Python poderia resolver.

15.3 Gerenciamento de Contexto

O contexto de uma sessão AIOX cresce naturalmente com o uso. Sem gestão, esse crescimento resulta em: - Custos crescentes (mais tokens de input em cada mensagem) - Latência crescente (modelos processam mais devagar com contextos gigantes) - Degradação de qualidade acima de 100k tokens (queda de até 50% na precisão — dado validado pelo cohort)

/clear vs /compact — A Escolha Certa

O cohort convergiu numa recomendação clara: /clear é melhor que /compact.

O /compact tenta preservar contexto resumindo a conversa. Mas o resumo nem sempre captura o que importa — e o processo de compactação consome ~20.000 tokens por sessão. O /clear zera completamente o contexto.

A estratégia correta: /clear entre fases + reler a story. Quando o agente faz /clear e depois relê o arquivo da story, ele recupera instantaneamente: o que está sendo feito, quais critérios devem ser atendidos, quais tarefas já foram completadas. O custo de reler a story (~500-1.000 tokens) é muito menor que o custo de manter contexto acumulado.

Regras práticas: - Desabilite auto-compact (economiza ~20k tokens/sessão) - Faça /clear entre trocas de agente (contexto do agente anterior não serve) - Nos 80% de uso da janela, use o método Edu: prompt de extração de checklist antes do /compact, capturando decisões e estado atual - Nunca use /compact quando pode usar /clear + reler story

O crescimento do contexto é o problema mais subestimado:

O Protocolo de Compactação de Handoff

O protocolo de Agent Handoff (definido nas regras do AIOX) é a primeira ferramenta de gestão de contexto. Quando você troca de agente, o contexto do agente anterior é compactado num handoff artifact de ~500 tokens em vez de ser carregado integralmente.

Sem compactação: cada troca de agente acumula ~3-5K tokens de persona do agente anterior. Com compactação: cada troca de agente adiciona ~500 tokens de handoff artifact.

Em uma sessão típica com 4 trocas de agente, a diferença é 20K tokens vs. 2K tokens — uma redução de 90% no custo de contexto acumulado de agentes.

Gestão Manual de Contexto

Além do handoff automático, há situações onde você precisa gerenciar contexto manualmente:

Quando iniciar nova sessão: Para cada nova story, considere iniciar uma sessão nova em vez de continuar a sessão atual. Uma sessão nova começa com contexto limpo — apenas o que é carregado pelo MEMORY.md e pelos arquivos de contexto do projeto.

Quando compactar memória: Se o MEMORY.md de um agente cresceu muito, compacte-o. Mantenha apenas as informações que efetivamente afetam comportamento futuro. Decisões de 6 meses atrás que já não são relevantes ocupam tokens em cada sessão.

Quando usar arquivos de contexto seletivos: Em vez de carregar todo o MEMORY.md em cada sessão, considere arquivos de contexto modulares que são carregados apenas quando relevantes.

A Estratégia de Contexto por Camadas

Camada 1: Context permanente (sempre carregado)
  - Personas de agentes
  - Regras fundamentais do projeto
  - Padrões de código estabelecidos

Camada 2: Context de projeto (carregado por sessão)
  - Estado atual do epic ativo
  - Decisões arquiteturais relevantes
  - Dependências conhecidas

Camada 3: Context de tarefa (carregado por story)
  - Story atual
  - Arquivos específicos da implementação
  - Histórico de QA Loop desta story

Camada 4: Context efêmero (descartado após uso)
  - Código de arquivo analisado pontualmente
  - Output de ferramentas externas
  - Resultados intermediários

A Camada 4 é a mais problemática se não gerenciada: resultados de análise de arquivos grandes, outputs de tools, e dados intermediários que foram úteis num momento mas não precisam mais estar no contexto. Descarte-os explicitamente quando não mais necessários.

15.4 Code Intelligence: Enriquecendo Operações

O AIOX tem um sistema de Code Intelligence que enriquece operações com dados de análise de código — mas apenas quando disponível. É sempre opcional, com graceful degradation quando não configurado.

O Que é Code Intelligence

Code Intelligence é um provider de metadados sobre o código: quais funções existem, quais módulos importam o quê, onde uma função específica é chamada, quais são as dependências de um módulo.

Com Code Intelligence disponível, operações de desenvolvimento ganham contexto adicional sem precisar que o agente leia arquivos manualmente:

Sem Code Intelligence:

@dev: "Para implementar esta feature, preciso entender onde o módulo
de autenticação é usado. Leia todos os arquivos que importam auth..."
[LLM lê 15 arquivos, processa, extrai informação]

Com Code Intelligence:

@dev: "Code Intelligence: onde auth.module é importado?"
[Resposta imediata de análise estática: 8 módulos, listados]
[LLM recebe lista estruturada, não precisa ler os arquivos]

A diferença em custo: 0 tokens adicionais para a consulta de dependências vs. centenas de tokens para cada arquivo lido manualmente.

Configurando Code Intelligence

O provider de Code Intelligence é configurado em core-config.yaml:

code_intelligence:
  provider: "nogic"  # ou outro provider configurado
  enabled: true
  cache_ttl: 3600  # segundos

Se o provider não está disponível, o sistema opera normalmente sem enrichment — isCodeIntelAvailable() retorna false e as operações continuam sem o metadado extra.

Quando Code Intelligence Tem Mais Impacto

Análise de impacto: "Quais módulos serão afetados se eu mudar a interface do módulo X?" Sem Code Intelligence, requer análise manual de dependências. Com Code Intelligence, resposta imediata.

Sugestão de onde implementar: "Em qual módulo esta função deveria viver?" Com Code Intelligence, o agente pode olhar o mapa de módulos existentes e fazer uma sugestão informada sem ler cada arquivo.

Detecção de duplicação: "Já existe algo no codebase que faz isso?" Com Code Intelligence, busca por padrões semelhantes é rápida e sem custo de tokens de leitura de arquivo.

15.5 Debug e Diagnóstico de Performance

Quando o AIOX está lento, caro, ou produzindo outputs de qualidade abaixo do esperado, você precisa de ferramentas para diagnosticar.

aiox doctor

Com essa visibilidade, você pode decidir: o custo de Brownfield Discovery de $12.71 foi uma vez só (esperado). O QA Loop de $3.51 está em linha com o esperado? Ou há mais iterações do que deveria?

Detectando Degradação de Qualidade

A degradação de qualidade é mais sutil que problemas de custo — você não recebe um erro, você recebe um output pior. Os sinais a observar:

Aumento na taxa de rejeição do PO: Se Pax está rejeitando mais stories que antes, ou as stories estão saindo com scores mais baixos, algo mudou. Pode ser que o MEMORY.md de @sm acumulou contexto confuso, ou que as personas foram alteradas inadvertidamente.

Aumento de iterações no QA Loop: Se o número médio de iterações até APPROVE aumentou, os outputs de Dex degradaram. Pode ser contexto de projeto desatualizado, ou MEMORY.md de @dev com decisões antigas que conflitam com o estado atual.

Inconsistência entre sessões: Outputs para inputs similares são muito diferentes entre sessões. Isso é sinal de que o contexto que os agentes carregam está inconsistente — diferentes versões de MEMORY.md, ou estado de épico desatualizado.

15.6 Estratégia Multi-LLM

O AIOX não está preso a um único modelo de LLM. Uma estratégia multi-LLM usa o modelo certo para cada tipo de operação.

O Princípio de Adequação de Modelo

Nem toda tarefa cognitiva é igual. A síntese final de um memo de investimento exige capacidades diferentes de uma classificação simples de tipo de bug. Usar o modelo mais poderoso para tudo é desperdício; usar o modelo mais fraco para tudo é degradação de qualidade.

Haiku — Quando Usar: - Classificação de baixa complexidade - Extração de estrutura de texto bem formado - Geração de código simples e repetitivo - Respostas factuais sobre dados estruturados - Validação de formato

Sonnet — Quando Usar: - A maioria das operações de desenvolvimento - Análise de código de complexidade média - Geração de conteúdo padrão - QA Gate de stories normais - Análises de domínio sem alta especialização

Opus — Quando Usar: - Decisões arquiteturais complexas - Síntese de análises de múltiplas perspectivas com alto stake - Brownfield Discovery de sistemas críticos - Análise de segurança e código sensível - Casos edge e problemas não-padrão

Implementando Routing de Modelos

# core-config.yaml
model_routing:
  default: "claude-sonnet-4-5"

  overrides:
    # Operações que justificam Opus
    - operations: ["investment-synthesis", "security-audit", "architecture-decision"]
      model: "claude-opus-4-5"

    # Operações que funcionam com Haiku
    - operations: ["schema-validation", "format-check", "simple-classification"]
      model: "claude-haiku-4-5"

    # Por agente
    - agent: "@architect"
      operations: ["complexity-assessment", "adr-creation"]
      model: "claude-opus-4-5"

    - agent: "@dev"
      operations: ["implement-simple-feature", "write-tests"]
      model: "claude-sonnet-4-5"

O Impacto Real no Custo

Para um projeto com 50 stories por mês, 30% de operações simples (Haiku), 60% de operações padrão (Sonnet), e 10% de operações complexas (Opus):

  • Sem routing (tudo Opus): ~$X/mês
  • Com routing: ~$0.35X/mês (65% de economia)

A variância exata depende dos preços atuais dos modelos e do volume específico do projeto, mas a ordem de grandeza é consistente: routing inteligente tipicamente resulta em 50-70% de redução de custo de LLM em projetos de desenvolvimento ativo.

15.7 Waves: Desenvolvimento Paralelo

O conceito de Waves representa uma fronteira na otimização do AIOX: múltiplos agentes trabalhando em tasks independentes simultaneamente, cada um em sua própria instância.

Como Funciona

Em vez de executar stories sequencialmente (Story 1 → Story 2 → Story 3), você identifica stories independentes que podem ser desenvolvidas em paralelo. Cada story roda em sua própria "wave" — uma instância separada do Claude Code com seu próprio contexto.

GitHub Worktrees como Base

Alan Nicolas testou e recomenda GitHub worktrees como a infraestrutura para Waves. Um worktree é uma cópia do repositório que compartilha o mesmo .git mas tem seu próprio working directory e branch:

# Criar worktrees para desenvolvimento paralelo
git worktree add ../wave-1 feat/story-3.1
git worktree add ../wave-2 feat/story-3.2
git worktree add ../wave-3 feat/story-3.3

Cada worktree opera independentemente. Três instâncias de Claude Code, cada uma em um worktree, cada uma desenvolvendo uma story diferente. Ao final, consolidação via merge.

Pré-Requisitos para Waves

Waves só funcionam quando: - As stories são genuinamente independentes (sem dependências entre si) - O planejamento do épico identificou quais stories podem paralelizar - Cada wave tem contexto suficiente (story + CLAUDE.md + relevant files) - A consolidação é feita por @devops com cuidado em merge conflicts

Agent Teams (Experimental)

A feature CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS do Claude Code vai além de worktrees: permite que agentes em instâncias separadas se comuniquem entre si. Isso é o futuro das Waves — agentes que colaboram em tempo real em tasks paralelas.

Checklist de Otimização

Use este checklist mensalmente para garantir que o seu projeto AIOX está operando de forma eficiente:

Custo: - [ ] Rodou aiox stats --period=30d e revisou top operações por custo? - [ ] Há operações nas top 5 que poderiam ser determinísticas? - [ ] O routing de modelos está configurado e atual? - [ ] Há spikes de custo não explicados?

Contexto: - [ ] MEMORY.md de cada agente ativo tem menos de 5KB? - [ ] Há entradas no MEMORY.md com mais de 3 meses que não são mais relevantes? - [ ] O EPIC-EXECUTION.yaml está atualizado?

Code Intelligence: - [ ] O provider está operacional? (aiox doctor) - [ ] Cache hit rate está acima de 70%? - [ ] Cache TTL está configurado adequadamente?

Qualidade: - [ ] Taxa de rejeição do PO está estável ou melhorando? - [ ] Número médio de iterações de QA Loop está estável ou diminuindo? - [ ] Outputs entre sessões similares são consistentes?
