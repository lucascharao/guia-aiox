Spec Pipeline e Brownfield: Workflows Avançados

SDC não é o único workflow. Spec Pipeline para features com incerteza, Brownfield Discovery para projetos legados. Escolha o certo para cada situação.

Introdução: Além do SDC

O Story Development Cycle é o workflow de referência do AIOX — ele cobre a grande maioria do trabalho de desenvolvimento. Mas existem situações onde o SDC, por si só, não é suficiente.

Quando você está diante de uma feature complexa que ainda não tem requisitos bem definidos, começar pelo SDC é como tentar construir uma casa sem plantas. Você vai construir algo, mas a probabilidade de precisar demolir e reconstruir é alta.

Quando você está entrando num projeto existente com dívida técnica acumulada, iniciar stories sem entender a arquitetura atual é como navegar num mapa que não corresponde ao território.

Para essas situações, o AIOX oferece dois workflows avançados: o Spec Pipeline e o Brownfield Discovery.

10.1 O Spec Pipeline

O Problema que o Spec Pipeline Resolve

Features complexas têm uma propriedade inconveniente: o custo de descobrir os requisitos errados cresce exponencialmente conforme você avança no desenvolvimento. Um erro de requisito descoberto durante o Spec Pipeline custa uma conversa de 30 minutos. O mesmo erro descoberto após três semanas de implementação custa dias de retrabalho.

O Spec Pipeline é o processo de transformar requisitos informais — uma ideia, um pedido de stakeholder, uma necessidade de produto — em uma especificação executável antes de qualquer linha de código ser escrita.

As 6 Fases do Spec Pipeline

O Spec Pipeline tem 6 fases, mas nem sempre todas são executadas. A complexidade da feature determina quais fases são necessárias.

Fase 1: Gather (@pm) Morgan inicia o processo coletando os requisitos no formato estruturado. O output é requirements.json — um arquivo que registra os requisitos funcionais (FR-), não-funcionais (NFR-), e restrições (CON-*).

Esta fase nunca é pulada. Não importa a simplicidade da feature — ter requisitos documentados em formato estruturado é o ponto de partida de tudo.

O processo de gather inclui: - Entrevista de stakeholder (simulada pelo Morgan a partir das informações que você fornece) - Classificação de requisitos em FR/NFR/CON - Identificação de dependências externas - Validação de completude básica (cada requisito é rastreável?)

Fase 2: Assess (@architect) Aria avalia a complexidade da feature com base em 5 dimensões e determina a classe de complexidade. O output é complexity.json.

Esta fase é pulada apenas quando source=simple — ou seja, quando você está executando o Spec Pipeline a partir de uma fonte que já foi classificada como simples.

Fase 3: Research (@analyst) Alex pesquisa padrões existentes, soluções de mercado, e considerações técnicas relevantes para a feature. O output é research.json.

Esta fase é pulada para features de classe SIMPLE. Para STANDARD e COMPLEX, a pesquisa adiciona contexto que melhora significativamente a qualidade da spec resultante.

Fase 4: Write Spec (@pm) Morgan, agora com contexto completo de requisitos, complexidade, e pesquisa, escreve a especificação formal. O output é spec.md.

Esta fase nunca é pulada. É o produto central do Spec Pipeline.

A spec.md inclui: - Resumo executivo da feature - Requisitos detalhados com rastreabilidade explícita para FR-/NFR-/CON-* - Considerações técnicas e arquiteturais - Critérios de aceite de alto nível - Dependências e riscos identificados

Fase 5: Critique (@qa) Quinn revisa a spec com o olhar crítico de qualidade. O output é critique.json com scores por dimensão e um verdict.

Esta fase nunca é pulada. Uma spec não revisada é uma spec que pode ter problemas de qualidade, ambiguidade, ou lacunas que só serão descobertos durante a implementação.

Fase 6: Plan (@architect) Aria cria o plano de implementação: quais stories serão necessárias, em que ordem, com quais dependências. O output é implementation.yaml.

Esta fase é pulada apenas se o verdict do Critique for APPROVED — o que significa que a spec está suficientemente detalhada para que o trabalho de planejamento seja simples e possa ser feito como parte do SDC normal.

As 3 Classes de Complexidade

A Fase 2 do Spec Pipeline classifica a feature em uma de três classes:

SIMPLE (score <= 8): Features que afetam poucos arquivos, não têm integrações externas significativas, não precisam de mudanças de infraestrutura, e a equipe tem familiaridade alta com a área de código. Fluxo: gather → spec → critique (3 fases, sem research e sem plan formal).

STANDARD (score 9-15): Features de complexidade moderada com alguma integração externa, mudanças que afetam múltiplos módulos, ou áreas onde a familiaridade da equipe é média. Fluxo: todas as 6 fases.

COMPLEX (score >= 16): Features de alta complexidade — múltiplas integrações, mudanças arquiteturais, alto risco, ou baixa familiaridade com o domínio. Fluxo: 6 fases + ciclo de revisão. Após o critique, se o verdict for NEEDS_REVISION, a spec passa por revisão e o critique é executado novamente.

As 5 Dimensões de Complexidade

A classificação de complexidade é feita em 5 dimensões, cada uma pontuada de 1 a 5:

Dimensão 1: Scope (Escopo) Quantos arquivos/módulos serão afetados?

Pontuação Critério
1 1-3 arquivos em área isolada
2 4-10 arquivos em módulo específico
3 10-25 arquivos, múltiplos módulos
4 25-50 arquivos, impacto cross-cutting
5 > 50 arquivos, mudança arquitetural global

Dimensão 2: Integration (Integração) Quantas integrações externas estão envolvidas?

Pontuação Critério
1 Nenhuma integração externa
2 1 integração com API interna conhecida
3 1-2 integrações externas com docs
4 3+ integrações ou integração com legacy sem docs
5 Múltiplas integrações críticas com incerteza alta

Dimensão 3: Infrastructure (Infraestrutura) Que nível de mudanças de infra são necessárias?

Pontuação Critério
1 Nenhuma mudança de infra
2 Mudança de configuração simples
3 Novo serviço ou banco de dados
4 Mudança em pipeline de CI/CD ou deploy
5 Mudança arquitetural de infraestrutura

Dimensão 4: Knowledge (Conhecimento) Qual é a familiaridade da equipe com a área?

Pontuação Critério
1 Alta familiaridade, foi feito antes
2 Boa familiaridade, área conhecida
3 Familiaridade média, alguma exploração necessária
4 Pouca familiaridade, domain novo
5 Sem familiaridade, área completamente nova

Dimensão 5: Risk (Risco) Qual é o nível de criticidade?

Pontuação Critério
1 Não-crítico, reversível facilmente
2 Baixo risco, rollback possível
3 Risco médio, rollback complexo
4 Alto risco, afeta dados ou segurança
5 Crítico, impacto irreversível ou em produção

O score total é a soma das 5 dimensões. Uma feature com scope=3, integration=2, infrastructure=1, knowledge=3, risk=2 teria score=11 — classificada como STANDARD.

Os Verdicts do Critique

A Fase 5 (Critique) produz um dos três verdicts:

APPROVED (average score >= 4.0): A spec está bem escrita, com rastreabilidade clara, critérios verificáveis, e sem lacunas significativas. O Spec Pipeline termina aqui (exceto o Plan para COMPLEX).

NEEDS_REVISION (average score 3.0-3.9): A spec tem qualidade adequada mas precisa de melhorias específicas. Quinn documenta exatamente o que precisa ser melhorado e a spec passa por revisão antes de avançar.

BLOCKED (average score < 3.0): A spec tem problemas fundamentais de qualidade — ambiguidades irresolvíveis, lacunas de requisitos que impedem especificação, ou contradições internas. Escalado para @architect para mediação.

O Gate do Artigo IV: Nenhuma Invenção

O gate mais importante do Spec Pipeline está na Fase 5. Quinn verifica que cada afirmação na spec.md tem rastreabilidade para um requisito documentado. Não pode haver features na spec que não foram explicitamente solicitadas.

Isso é o Artigo IV da Constitution em ação: No Invention. Uma feature não solicitada que aparece na spec pode parecer um presente, mas é um risco: não foi priorizada, não foi orçada, pode ter dependências não mapeadas, e não corresponde a um valor de produto validado.

A regra é simples: cada declaração na spec que descreve comportamento do sistema deve ter a forma [FR-N] ou [NFR-N] ou [CON-N] após ela, indicando de qual requisito documentado esse comportamento se origina.

Iniciando o Spec Pipeline

@pm *spec-pipeline

Morgan vai iniciar o processo de elicitação da Fase 1. Você fornece o contexto inicial — a ideia, o pedido de stakeholder, os objetivos de negócio — e Morgan transforma isso no processo estruturado.

10.2 Brownfield Discovery

O Contexto: Entrando num Projeto Existente

Você foi contratado para modernizar um sistema que existe há 8 anos. Ou você está retomando um projeto que ficou parado por meses. Ou você quer usar o AIOX num projeto que já tem código mas foi desenvolvido sem o framework.

Em todos esses casos, o problema é o mesmo: você não sabe o que está lá dentro. Há dívida técnica acumulada, decisões arquiteturais não documentadas, integrações mal entendidas, e esquemas de banco de dados que nunca passaram por review formal.

Tentar criar stories e implementar sem entender essa realidade é garantia de surpresas. Surpresas no desenvolvimento são bugs e retrabalho.

O Brownfield Discovery é o processo de 10 fases para fazer um assessment técnico completo de um sistema existente antes de iniciar qualquer desenvolvimento com o AIOX.

As 10 Fases do Brownfield Discovery

Coleta de Dados (Fases 1-3)

Fase 1: System Architecture Assessment (@architect) Aria mapeia a arquitetura atual do sistema. O output é system-architecture.md, um documento que registra: - Componentes e suas responsabilidades - Padrões arquiteturais utilizados (e os que deveriam ter sido utilizados) - Fronteiras de domínio - Pontos de integração internos e externos - Débitos arquiteturais identificados

Fase 2: Database & Data Assessment (@data-engineer) Dara analisa o banco de dados e o modelo de dados. Os outputs são: - SCHEMA.md: Documentação completa do schema atual - DB-AUDIT.md: Auditoria de problemas no banco de dados — indices faltantes, inconsistências de nomenclatura, constraints violáveis, queries N+1 identificadas, modelos de dados que não refletem o domínio

Esta fase só é executada se o projeto tem banco de dados. Para projetos sem persistência própria, é pulada.

Fase 3: Frontend & UX Assessment (@ux-design-expert) Uma faz a análise do frontend existente. O output é frontend-spec.md: - Componentes e sua estrutura atual - Padrões de UX utilizados (e os problemas) - Estado de acessibilidade - Inconsistências de design system - Débitos de UX identificados

Esta fase é executada para qualquer projeto com interface visual.

Rascunho e Validação (Fases 4-7)

Fase 4: Technical Debt Draft (@architect) Com os dados coletados nas três primeiras fases, Aria compila o documento de avaliação de dívida técnica. O output é technical-debt-DRAFT.md.

Este documento categoriza a dívida técnica em: - Crítica: Impede features ou causa bugs em produção - Alta: Aumenta significativamente o custo de mudanças - Média: Causa fricção no desenvolvimento - Baixa: Problema estético ou de qualidade sem impacto funcional

Fase 5: Database Specialist Review (@data-engineer) Dara revisa o draft com foco específico no componente de banco de dados e dados. O output é db-specialist-review.md. Dara verifica se a avaliação de dívida técnica de banco de dados está completa, precisa, e priorizada corretamente.

Fase 6: UX Specialist Review (@ux-design-expert) Uma revisa o draft com foco no componente de frontend e UX. O output é ux-specialist-review.md. Uma verifica se os problemas de UX foram identificados corretamente e se a priorização reflete o impacto real na experiência do usuário.

Fase 7: QA Gate (@qa) Quinn revisa o assessment completo — arquitetura, banco de dados, e frontend — e emite um veredicto sobre a qualidade do documento de dívida técnica.

QA Gate Verdicts na Fase 7:

  • APPROVED: O assessment está completo, todos os débitos foram validados, não há lacunas críticas, e as dependências entre débitos foram mapeadas. O Brownfield Discovery avança para a fase de finalização.

  • NEEDS WORK: Há lacunas significativas no assessment — áreas do sistema não cobertas, débitos identificados mas não avaliados, ou dependências não mapeadas. O documento volta para a Fase 4 para complementação.

Finalização (Fases 8-10)

Fase 8: Final Technical Debt Assessment (@architect) Incorporando o feedback de todas as revisões e do QA Gate, Aria produz o documento final. O output é technical-debt-assessment.md — a versão definitiva e assinada do assessment técnico.

Fase 9: Executive Report (@analyst) Alex transforma o documento técnico num relatório executivo acessível para stakeholders não-técnicos. O output é TECHNICAL-DEBT-REPORT.md.

Este documento responde às perguntas que executivos fazem: - Qual é o estado do sistema? - Quais são os riscos se não agirmos? - O que precisa ser feito e em que ordem? - Qual é o custo estimado (em esforço) de cada item?

Fase 10: Epic Planning (@pm) Morgan usa o assessment para criar o backlog de modernização. O output é um epic (ou múltiplos epics) com stories prontas para o SDC normal.

Esta fase transforma o resultado do Brownfield Discovery num plano de trabalho executável. Ao final da Fase 10, você tem stories em status Ready que podem imediatamente entrar no ciclo normal de desenvolvimento.

O QA Gate da Fase 7 em Detalhe

O QA Gate do Brownfield Discovery é diferente do QA Gate do SDC porque o que está sendo avaliado é um documento de análise, não código implementado. Quinn verifica:

  1. Completude: Todos os componentes do sistema foram analisados?
  2. Validação dos débitos: Cada débito identificado tem evidência? (não é apenas uma opinião)
  3. Sem lacunas críticas: Há componentes ou áreas que foram ignorados?
  4. Mapeamento de dependências: Os débitos que dependem uns dos outros foram mapeados?
  5. Priorização consistente: A severidade atribuída a cada débito é consistente com seu impacto real?
  6. Acionabilidade: Os débitos são descritos de forma suficientemente clara para gerar stories?
  7. Estimativas razoáveis: As estimativas de esforço são realistas e justificadas?

10.3 Escolhendo o Workflow Certo

Com três workflows disponíveis (SDC, Spec Pipeline, Brownfield Discovery), a pergunta natural é: como sei qual usar?

A resposta está num conjunto de perguntas diagnósticas:

O Guia de Seleção de Workflow

Situação Workflow
Você tem uma story clara com requisitos bem definidos SDC diretamente
Feature nova mas simples, critérios óbvios SDC com modo YOLO
Feature complexa com requisitos ainda vagos Spec Pipeline → depois SDC
Feature com múltiplas integrações externas Spec Pipeline (STANDARD ou COMPLEX)
Feature com risco alto ou mudança arquitetural Spec Pipeline → review com @architect
Entrando em projeto existente sem documentação Brownfield Discovery primeiro
Projeto existente com dívida técnica acumulada Brownfield Discovery → stories de modernização via SDC
Bug fix simples com causa identificada SDC (YOLO mode)
Bug intermitente de causa desconhecida Spec Pipeline para investigação antes de fix
Onboarding de novo membro no projeto Brownfield Discovery (pode ser parcial)

Os Sinais de que Você Escolheu o Workflow Errado

Sinal de que deveria ter usado Spec Pipeline: - Dex está frequentemente bloqueando por falta de clareza nos requisitos - O QA Loop está em 3+ iterações por causa de ambiguidade de critérios - O PO está rejeitando stories consistentemente por AC mal definidos

Sinal de que deveria ter feito Brownfield antes: - Dex está descobrindo surpresas arquiteturais que não estavam na story - As stories estão sendo bloqueadas por dependências não mapeadas - Cada implementação quebra algo que não estava no escopo

Sinal de que está complicando desnecessariamente: - Você está usando Spec Pipeline para features de complexidade 1 (escopo de 1 arquivo, sem integração, equipe familiar) - O Brownfield Discovery está levando semanas para um projeto de 3 meses de desenvolvimento - Você está esperando a spec perfeita antes de começar qualquer coisa

O Spec Pipeline não é um ritual obrigatório — é uma ferramenta para quando a incerteza de requisitos justifica o investimento de tempo em especificação antes da implementação. Para features simples e bem entendidas, o SDC direto é o caminho certo.

Composição de Workflows

Os workflows são compostos, não exclusivos. O Brownfield Discovery resulta em stories que entram no SDC. O Spec Pipeline produz uma spec que informa as stories do SDC. O SDC inclui QA Loop como mecanismo iterativo.

O fluxo de trabalho típico num projeto novo com feature complexa:

1. Brownfield Discovery (se projeto existente)
   ↓
2. Spec Pipeline (para features complexas)
   ↓
3. Story Development Cycle (para cada story)
   ↓ (se QA falhar)
4. QA Loop (ciclo iterativo)
   ↓
5. Entrega (@devops)

Para um projeto greenfield com features simples:

1. Story Development Cycle diretamente
   ↓
2. QA Loop se necessário
   ↓
3. Entrega

A beleza do AIOX é que você não precisa escolher um único workflow para o projeto inteiro. Você escolhe o workflow certo para cada situação.

Exercício: Aplicando o Workflow Correto

Este exercício desenvolve a habilidade de seleção de workflow.

Cenário 1: Você precisa adicionar um campo de "bio" ao perfil de usuário. O campo é um textarea com máximo 500 caracteres, exibido na página de perfil público. - Qual workflow você usaria? - Por quê? - Se Spec Pipeline, qual classe de complexidade e quantas fases?

Cenário 2: Você precisa integrar um sistema de pagamentos (Stripe) no produto. É a primeira vez que o time trabalha com pagamentos. Há múltiplos cenários: pagamento único, assinatura recorrente, cancelamento, reembolso. - Qual workflow você usaria? - Justifique cada dimensão de complexidade. - Quais fases do Spec Pipeline seriam executadas?

Cenário 3: Você assumiu manutenção de um e-commerce em PHP com 7 anos de histórico, sem documentação de arquitetura, com banco de dados MySQL sem migrations versionadas. - Qual workflow você iniciaria? - O que você esperaria encontrar no output de cada fase? - Como você priorizaria os débitos técnicos identificados?

Parte Prática: Para o seu projeto atual, execute:

@pm *spec-pipeline

...para a próxima feature que você planeja implementar. Observe o processo de classificação de complexidade. Observe como Morgan estrutura os requisitos. Observe como o output da spec é diferente de uma story normal.

Compare: quanto tempo levou o Spec Pipeline versus quanto tempo levou uma story typical no SDC? Para features da mesma complexidade, o Spec Pipeline reduziu o número de QA Loop iterations necessárias?
