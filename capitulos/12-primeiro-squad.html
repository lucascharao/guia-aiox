Introdução: Da Teoria à Prática

O capítulo anterior explicou o que é um squad e por que o conceito existe. Este capítulo vai guiar você passo a passo pelo processo de montar um squad funcional, do zero até o primeiro output em produção.

Vamos usar um exemplo concreto ao longo de todo o capítulo: um squad de análise de startups — uma equipe de agentes especializada em avaliar startups para investimento, parceria, ou due diligence.

Por que esse exemplo? Porque ele é suficientemente complexo para ilustrar as decisões reais que você vai enfrentar (múltiplas perspectivas, dados de domínio específico, saídas que precisam ser confiáveis), mas suficientemente concreto para que os princípios sejam transferíveis para qualquer outro domínio.

12.1 Escolhendo o Domínio: A Decisão Mais Importante

Antes de criar qualquer arquivo, você precisa de clareza sobre o domínio do squad. Essa é a decisão mais importante do processo — e é onde a maioria dos squads mal projetados nasce.

As 4 Perguntas de Definição de Domínio

Pergunta 1: Qual é o problema específico que este squad resolve?

Errado: "Análise de empresas" Certo: "Avaliar startups em estágio early-stage (pré-seed a Series A) para decisão de investimento anjo, com foco em mercados de tecnologia B2B no Brasil"

A especificidade não é limitação — é qualidade. Um squad com domínio bem definido tem agentes com expertise real. Um squad com domínio vago tem agentes com expertise superficial em tudo.

Pergunta 2: Quais são as perspectivas especializadas necessárias?

Para o nosso squad de startups: você precisa de alguém que entenda de mercado e competição, alguém que entenda de tecnologia e execução técnica, alguém que entenda de equipe e fundadores, e alguém que consolide as análises em uma recomendação coerente.

Cada perspectiva que você identifica corresponde a um agente no squad.

Pergunta 3: Qual é o output esperado?

Um squad de análise de startups produz: memo de investimento, avaliação de risco, perfil de fundadores, análise de mercado. Saber os outputs antes de criar os agentes te ajuda a definir as skills de cada agente corretamente.

Pergunta 4: Qual é o grau de especialização necessário?

Para análise de startups de alto valor, você precisa de agentes que pareçam especialistas sérios — não assistentes genéricos que "sabem sobre startups". Isso vai impactar a profundidade das personas que você vai criar.

O Mapa de Domínio

Com as quatro perguntas respondidas, você tem o que chamamos de Mapa de Domínio:

Domínio: Análise de startups B2B early-stage (Brasil)
Perspectivas: mercado, tecnologia, equipe, síntese
Outputs: memo de investimento, avaliação de risco, perfil de equipe
Especialização: alta (agentes precisam parecer especialistas credíveis)

12.2 Mapeando as Entidades do Squad

Com o domínio definido, o próximo passo é mapear as entidades — os agentes que vão compor o squad.

Princípio de Separação de Concerns

Cada agente em um squad deve ter uma perspectiva que não se sobrepõe significativamente com os demais. Não crie dois agentes que fazem a mesma coisa de formas ligeiramente diferentes. Crie agentes com perspectivas genuinamente diferentes que se complementam.

Para o squad de startups:

Analista de Mercado (Marina): - Foco: TAM/SAM/SOM, competição, posicionamento, timing de mercado - Pergunta central: "O mercado existe e é grande o suficiente?" - Não analisa: tecnologia, equipe, financeiro

Avaliador de Tecnologia (Theo): - Foco: stack técnico, arquitetura, escalabilidade, velocidade de execução técnica - Pergunta central: "A tecnologia é defensável e executável?" - Não analisa: mercado, equipe fundadora, modelo de negócio

Analista de Equipe (Sofia): - Foco: background dos fundadores, dinâmica de equipe, track record, capacidade de execução - Pergunta central: "A equipe tem o que precisa para ir de 0 a 1?" - Não analisa: mercado, tecnologia

Sintetizador de Investimento (Victor): - Foco: consolida as análises das outras perspectivas em recomendação coerente - Pergunta central: "Vale o investimento? Em que termos?" - Depende dos outputs de Marina, Theo, e Sofia

Essa é uma hierarquia natural: Marina, Theo, e Sofia são analistas especializados. Victor é o sintetizador que usa o trabalho deles para produzir a recomendação final.

12.3 Scaffolding com Python

O Princípio Code First se aplica diretamente aqui: a criação da estrutura de arquivos do squad é uma operação determinística. Não use IA para criar pastas e arquivos — escreva um script Python.

O Script de Scaffolding

#!/usr/bin/env python3
"""
scaffold_squad.py — Cria estrutura base de um squad AIOX
Uso: python scaffold_squad.py startup-analysis
"""
import os
import sys
import yaml
from pathlib import Path

def scaffold_squad(squad_name: str, agents: list[dict]) -> None:
    """Cria estrutura de diretórios e arquivos base para um squad."""
    base_path = Path(f"squads/{squad_name}")

    # Estrutura de diretórios
    dirs = [
        base_path,
        base_path / "agents",
        base_path / "data" / "knowledge",
        base_path / "data" / "examples",
        base_path / "data" / "references",
        base_path / "tasks",
    ]

    for d in dirs:
        d.mkdir(parents=True, exist_ok=True)
        print(f"Criado: {d}")

    # squad.yaml
    manifest = {
        "name": squad_name,
        "version": "1.0.0",
        "description": f"Squad de {squad_name}",
        "agents": [
            {"id": a["id"], "persona": f"agents/{a['id']}.md", "role": a["role"]}
            for a in agents
        ],
        "domain": {"primary": squad_name},
        "commands": []
    }

    with open(base_path / "squad.yaml", "w") as f:
        yaml.dump(manifest, f, default_flow_style=False, allow_unicode=True)
    print(f"Criado: {base_path}/squad.yaml")

    # Templates de agentes
    agent_template = """# {name} — {role}

## Persona

{name} é um especialista em {specialty}.

## Background

[Descreva o background profissional e experiência do agente]

## Skills

- [Skill principal 1]
- [Skill principal 2]
- [Skill principal 3]

## Perspectiva

[Descreva a perspectiva única que este agente traz]

## O Que Este Agente Analisa

[Lista de aspectos que este agente avalia]

## O Que Este Agente NÃO Analisa

[Lista explícita do que está fora do escopo]

## Output Esperado

[Formato e conteúdo do output deste agente]
"""

    for agent in agents:
        agent_path = base_path / "agents" / f"{agent['id']}.md"
        content = agent_template.format(
            name=agent["name"],
            role=agent["role"],
            specialty=agent.get("specialty", agent["role"])
        )
        with open(agent_path, "w") as f:
            f.write(content)
        print(f"Criado: {agent_path}")

    # MEMORY.md
    memory_content = f"""# {squad_name} — Memória Compartilhada

## Decisões de Análise
[Registre aqui análises significativas e suas conclusões]

## Padrões Identificados
[Padrões observados ao longo do tempo no domínio]

## Calibrações
[Ajustes de critérios e preferências do operador]

## Contexto Acumulado
[Contexto relevante que os agentes devem carregar entre sessões]
"""

    with open(base_path / "MEMORY.md", "w") as f:
        f.write(memory_content)
    print(f"Criado: {base_path}/MEMORY.md")

    print(f"\nSquad '{squad_name}' criado com sucesso em squads/{squad_name}/")
    print("Próximo passo: configure as personas dos agentes em squads/{squad_name}/agents/")

if __name__ == "__main__":
    if len(sys.argv) < 2:
        print("Uso: python scaffold_squad.py <squad-name>")
        sys.exit(1)

    squad_name = sys.argv[1]

    # Defina seus agentes aqui
    agents = [
        {"id": "market-analyst", "name": "Marina", "role": "Análise de mercado", "specialty": "análise de mercados e competição"},
        {"id": "tech-evaluator", "name": "Theo", "role": "Avaliação de tecnologia", "specialty": "arquitetura e execução técnica"},
        {"id": "team-analyst", "name": "Sofia", "role": "Análise de equipe", "specialty": "avaliação de fundadores e dinâmica de equipe"},
        {"id": "investment-synthesizer", "name": "Victor", "role": "Síntese de investimento", "specialty": "tomada de decisão de investimento"},
    ]

    scaffold_squad(squad_name, agents)

Execute:

python scaffold_squad.py startup-analysis

Em segundos, você tem toda a estrutura de diretórios e arquivos base criada. Zero tokens de IA gastos nessa operação.

12.4 Configurando os Agentes: A Parte que Exige IA

Com a estrutura criada, você vai preencher os templates de persona para cada agente. Esta é a parte que genuinamente exige inteligência artificial — criar personas convincentes, com background credível, skills específicas, e perspectiva distinta requer o tipo de julgamento que Python não pode fornecer.

Anatomia de uma Persona de Squad

Uma persona de squad bem escrita tem quatro camadas:

Camada 1: Identidade Profissional Quem é esta pessoa? Qual é sua trajetória? O que a torna especialista nesse domínio específico? Seja específico — não "30 anos de experiência em finanças" mas "10 anos como analista em fundos de venture capital early-stage no Brasil, passagem por Y Combinator como advisor, co-fundou duas startups antes de ir para o lado do investimento".

Camada 2: Perspectiva Característica Qual é o ângulo único que este agente traz? Quais são as perguntas que ele faz que outros não fazem? Quais são os sinais que ele busca que revelam insights que outros perderiam?

Marina (nossa analista de mercado) sempre começa pela pergunta: "Por que agora? O que mudou no mercado que torna esse produto possível hoje quando não era cinco anos atrás?" Essa é uma perspectiva característica que informa o tipo de análise que ela vai produzir.

Camada 3: Skills Operacionais O que concretamente este agente sabe fazer? Não em termos abstratos mas em termos de operações específicas: - Marina sabe fazer análise de TAM bottom-up e top-down - Marina sabe mapear competidores diretos, indiretos, e substitutos - Marina sabe identificar dinâmicas de mercado que favorecem ou dificultam adoção

Camada 4: Limitações Explícitas O que este agente NÃO faz? Essa camada é frequentemente omitida e é crítica. Um agente sem limites explícitos vai tentar fazer tudo. Um agente com limites bem definidos vai saber quando dizer "isso não é meu domínio — Theo vai olhar a parte técnica".

Exemplo Completo: Persona da Marina

# Marina — Analista de Mercado

## Persona

Marina opera como uma analista de mercado sênior com 12 anos de experiência
avaliando oportunidades de mercado para fundos de VC. Ela passou pelos dois lados:
analista de mercado num fundo Tier 1 e depois como gerente de produto numa startup
que escalou de zero a R$50M ARR. Essa dupla perspectiva a torna única: ela sabe
tanto avaliar mercados quanto entender o que é necessário para capturá-los.

## Perspectiva Característica

Marina sempre começa pela pergunta de timing: "Por que agora?" Mercados existem
há décadas; o que mudou que torna este produto viável, necessário, e capturável
neste momento específico? Ela busca os "unfair advantages estruturais" — mudanças
regulatórias, shifts tecnológicos, mudanças comportamentais — que criam janelas
de oportunidade que se fecham.

## Framework de Análise de Mercado

Quando Marina avalia uma startup, ela estrutura a análise em quatro blocos:

**Bloco 1: Tamanho e Segmentação**
- TAM (Total Addressable Market) — top-down e bottom-up
- SAM (Serviceable Addressable Market) — quem pode ser alcançado realisticamente
- SOM (Serviceable Obtainable Market) — quem pode ser convertido nos próximos 3 anos
- Dinâmica de crescimento (o mercado está crescendo, estável, ou contraindo?)

**Bloco 2: Competição e Posicionamento**
- Mapa competitivo completo (diretos, indiretos, substitutos, status quo)
- Por que o usuário vai migrar do status quo ou do líder atual?
- Qual é a vantagem diferencial sustentável?
- Como os competidores estabelecidos vão reagir?

**Bloco 3: Timing e Catalysadores**
- Por que este produto é possível agora e não era antes?
- Quais são os catalisadores que aceleram ou retardam a adoção?
- Qual é o risco de "too early"? De "too late"?

**Bloco 4: Modelo de Entrada**
- Qual é o beachhead ideal? (segmento inicial mais fácil de penetrar)
- Qual é a estratégia de expansão de mercado a partir do beachhead?
- Qual é a dependência de canal de distribuição?

## O Que Marina NÃO Analisa

Marina não avalia:
- Qualidade técnica do produto ou stack (isso é domínio de Theo)
- Background e capacidade dos fundadores (isso é domínio de Sofia)
- Valuation ou estrutura do deal (isso é domínio de Victor)
- Financeiros detalhados além de métricas de mercado (Victor consolida)

Se durante a análise Marina identificar questões de tecnologia ou equipe que
impactam sua análise de mercado, ela vai notar explicitamente: "Esta análise
assume que a equipe tem capacidade técnica para executar o roadmap descrito —
recomendo que Theo valide esta premissa."

Essa é uma persona que vai produzir análises de mercado reais, não genéricas. O tempo investido na criação dessa persona é o que diferencia um squad medíocre de um squad genuinamente útil.

12.5 Alimentando Dados de Primeiro Partido

Personas bem escritas são o fundamento. Os dados de primeiro partido são o que transforma um squad genérico num squad expert no seu contexto específico.

O Que São Dados de Primeiro Partido

Dados de primeiro partido são informações únicas e proprietárias que você coloca no squad e que não estão disponíveis publicamente. Exemplos:

Para o squad de startups: - Teses de investimento específicas da sua firma - Histórico de análises anteriores e seus resultados - Critérios de filtro que foram desenvolvidos internamente - Base de dados de startups analisadas com outcomes documentados - Templates de memo de investimento da sua firma

Para um squad de análise jurídica: - Jurisprudência relevante para o seu nicho - Contratos-modelo com cláusulas preferidas - Histórico de decisões de casos semelhantes

Para um squad de conteúdo: - Guia de voz e tom da marca - Exemplos de conteúdo aprovado - Contexto de audiência e persona

Estrutura dos Dados no Squad

squads/startup-analysis/data/
├── knowledge/
│   ├── investment-thesis.md        # Tese de investimento da firma
│   ├── evaluation-criteria.md      # Critérios de avaliação desenvolvidos
│   └── market-definitions.md       # Definições e taxonomia do domínio
├── examples/
│   ├── strong-memo-example.md      # Exemplo de memo de investimento forte
│   ├── weak-memo-example.md        # Exemplo de memo com problemas identificados
│   └── analysis-examples/          # Análises anteriores (anonimizadas se necessário)
└── references/
    ├── market-benchmarks.md        # Benchmarks de mercado para contexto
    └── term-glossary.md            # Glossário de termos do domínio

Como os Agentes Usam os Dados

Os dados são referenciados nas personas dos agentes:

## Contexto de Primeiro Partido

Marina usa os seguintes dados proprietários ao analisar mercados:
- `/data/knowledge/investment-thesis.md` — tese de investimento que deve guiar
  o foco do mercado
- `/data/knowledge/evaluation-criteria.md` — critérios calibrados para o
  tamanho e estágio de mercado que buscamos
- `/data/examples/` — exemplos de análises aprovadas que definem o padrão
  esperado

Com esses dados, Marina não apenas sabe analisar mercados em geral — ela sabe analisar mercados da forma que sua firma específica analisa, com seus critérios, seus benchmarks, e seus padrões de qualidade.

12.6 Testando o Squad

Antes de usar o squad em situações reais, você precisa testá-lo. O teste de um squad tem quatro dimensões.

Teste 1: Coerência de Persona

Cada agente deve responder de forma consistente com sua persona, independente do que você perguntar.

Teste simples: faça a mesma pergunta para dois agentes diferentes. As respostas devem ser genuinamente diferentes — perspectivas diferentes, focos diferentes, conclusões possivelmente diferentes. Se as respostas parecerem intercambiáveis, as personas não estão suficientemente diferenciadas.

Teste 2: Respeito de Limites

Cada agente deve saber quando está fora de seu domínio e delegar adequadamente.

Teste: peça para Marina avaliar o stack técnico de uma startup. A resposta correta é: "Stack técnico está fora do meu domínio de análise. Theo vai avaliar isso. O que posso dizer do ponto de vista de mercado é..."

Se Marina tentar avaliar o stack técnico como se fosse expert em tecnologia, a persona está mal configurada.

Teste 3: Qualidade de Output

O output de cada agente deve ser utilizável, não apenas correto.

Teste: dê um caso real (ou realista) para cada agente e avalie: o output tem a profundidade que você esperaria de um especialista real? Falta alguma dimensão importante? Há afirmações que parecem superficiais ou genéricas?

Teste 4: Colaboração

Quando o squad opera como equipe (com Victor sintetizando os outputs de Marina, Theo, e Sofia), a síntese deve ser coerente e must refletir os inputs de forma balanceada.

Teste: execute uma análise completa, observe como Victor incorpora os inputs dos outros agentes. Se Victor está ignorando perspectivas críticas levantadas pelos outros agentes, a configuração de síntese está errada.

12.7 Comandos do Squad

Squads têm comandos, da mesma forma que agentes individuais têm comandos com o prefixo *.

Definindo Comandos no squad.yaml

commands:
  - name: analyze-startup
    description: "Análise completa de uma startup"
    agents: [market-analyst, tech-evaluator, team-analyst, investment-synthesizer]
    workflow: sequential  # ou parallel
    output: investment-memo

  - name: market-scan
    description: "Análise de mercado rápida"
    agents: [market-analyst]
    output: market-brief

  - name: team-check
    description: "Avaliação rápida de equipe fundadora"
    agents: [team-analyst]
    output: team-profile

Invocando Comandos do Squad

@startup-analysis *analyze-startup

Isso ativa o squad e executa o workflow analyze-startup, passando o input pelos agentes na sequência definida.

Para um comando mais específico:

@startup-analysis *market-scan

Executa apenas a análise de mercado com Marina, sem os outros agentes.

Workflow Sequential vs. Parallel

Sequential: Os agentes executam em ordem. O output de cada agente é disponível para os agentes subsequentes. Victor recebe os outputs de Marina, Theo, e Sofia antes de sintetizar. Mais lento, mas permite que a síntese seja genuinamente informada pelos inputs especializados.

Parallel: Todos os agentes executam simultaneamente, sem ver os outputs uns dos outros antes. Victor então recebe todos os outputs ao mesmo tempo para sintetizar. Mais rápido, mas perde a riqueza de agentes que podem ajustar suas análises com base em perspectivas dos outros.

Para análises de alta qualidade onde o custo de tempo é aceitável, sequential é recomendado. Para análises de triagem rápida onde velocidade importa mais que profundidade, parallel funciona bem.

12.8 O Caminho Simplificado: 6 Passos para Criar um Squad

Para quem está começando, existe um caminho simplificado que o Squad Creator automatiza:

  1. Defina em uma frase o que o squad faz: "Analisa startups para investimento"
  2. Chame o Squad Creator: /squad-creator:squad-chief
  3. Responda as perguntas de elicitação (domínio, especialistas, tasks)
  4. Aguarde o processo automático — 6 fases internas: elicitação, pesquisa, advogado do diabo, definição de tiers, criação paralela de agentes, validação
  5. Valide o resultado: *validate-squad verifica estrutura, schema, referências
  6. Teste: Execute os commands do squad com dados reais

O Caminho Avançado: Como Alan Faz

Para squads de alta qualidade, Alan segue um processo mais rigoroso:

  1. Round Table — Consulta o Board de Advisors (clones de Ray Dalio, Naval Ravikant, Charlie Munger, etc.) sobre a estratégia do squad
  2. Tech Research — Pesquisa via @analyst as melhores ferramentas e MCPs para o domínio
  3. Identificar especialistas via livros — Busca na Z-Library os autores referência do domínio, extrai frameworks e DNA de pensamento
  4. Criar com documentação completa — Alimenta o Squad Creator com a pesquisa feita
  5. Estressar por uma semana — Executa o squad repetidamente com cenários variados
  6. Iterar — Ajusta personas, tasks e workflows baseado nos outputs reais

A diferença entre os dois caminhos: o simplificado produz um squad funcional em minutos. O avançado produz um squad que supera consultores humanos no domínio.

12.9 Maturidade do Squad: As 5 Fases

Como Pedro Valério diz: "O primeiro squad é cocô — é adubo." E isso é intencional. Squads nascem imperfeitos e amadurecem com uso:

Fase 1: Infância (Criação)

O squad foi criado mas ainda não executou. Agentes têm personas básicas, tasks são genéricas. É um rascunho — destrua e reconstrua sem apego.

Fase 2: Adolescência (Validação)

O squad está sendo testado com dados reais. Você identifica problemas, ajusta personas, refina tasks. Semanas 1-2 de uso ativo.

Fase 3: Maturidade (Serviços e Tools)

O squad produz outputs consistentes. Você adiciona ferramentas (MCPs, Discovery Tools), conecta com outros squads. Semanas 3-4.

Fase 4: Adulto (Produtização)

O squad opera com confiança. Meta: 90%+ de consistência em 20+ execuções. Pronto para ser empacotado como serviço.

Fase 5: Empresa (Escala)

O squad atende múltiplos clientes/projetos. Tem monitoramento, métricas, e processo de melhoria contínua.

12.10 Squads Nascem de Processos Validados

Uma regra da Aula 07 que merece destaque: squads devem nascer de processos humanos validados.

O fluxo correto: 1. Execute manualmente — Faça o processo à mão primeiro 2. Identifique conversas demais — Se está explicando muito para a IA, falta uma task 3. Exporte o processo — Documente cada passo como POP (Processo Operacional Padrão) 4. Quebre em tasks — Cada POP vira uma ou mais tasks do squad 5. Itere — Execute o squad, compare com o resultado manual, ajuste

Fontes ricas para POPs: podcasts do domínio, eventos gravados, livros técnicos, conversas com clientes.

Regra do Max 3-4 Agentes

Talles, do cohort, validou: squads com máximo 3-4 agentes são mais eficientes. Squads grandes têm overhead de coordenação que degrada a qualidade. Se precisa de mais, segmente em squads menores e coordene entre eles.

Exercício: Monte Seu Squad

Escolha um domínio onde você tem expertise real ou onde você tem necessidade frequente de análise qualificada. Pode ser análise de concorrentes, criação de conteúdo especializado, avaliação de candidatos, pesquisa de produtos.

Passo 1: Responda as 4 Perguntas de Definição de Domínio para o seu domínio escolhido.

Passo 2: Mapeie as entidades — quantos agentes? Quais perspectivas? Quais os limites de cada um?

Passo 3: Execute o scaffold:

python scaffold_squad.py {seu-dominio}

Passo 4: Escreva a persona do agente mais importante do seu squad. Use a estrutura de 4 camadas: identidade profissional, perspectiva característica, skills operacionais, limitações explícitas.

Passo 5: Adicione um dado de primeiro partido que torna o squad específico para o seu contexto.

Passo 6: Teste com um caso real. Avalie: a resposta é melhor do que o que você obteria de um agente genérico? Em que dimensões?

O squad que você cria neste exercício é a base de algo que vai melhorar com o uso. Cada análise que você faz, cada ajuste de persona, cada dado de primeiro partido que você adiciona — tudo isso torna o squad progressivamente mais valioso.
