6.1 Pré-Requisitos: O Que Você Precisa Antes de Começar

Antes de instalar o AIOX, você precisa garantir que seu ambiente tem as ferramentas fundamentais. Diferente de alguns frameworks que tentam gerenciar suas próprias dependências, o AIOX assume que você tem um ambiente de desenvolvimento funcional e se integra a ele — não o substitui.

Esta seção detalha cada pré-requisito, por que ele existe, e como verificar se está instalado corretamente.

Node.js 18+

O AIOX é construído em Node.js. A versão mínima é 18 porque essa versão introduziu melhorias significativas de performance no runtime, suporte nativo a módulos ES, e APIs de filesystem mais robustas que o framework utiliza.

Para verificar sua versão atual:

node --version
# Output esperado: v18.x.x ou superior

Se você não tem Node.js instalado, ou tem uma versão antiga, a forma recomendada de gerenciar versões é via nvm (Node Version Manager) no macOS/Linux ou nvm-windows no Windows. Isso permite manter múltiplas versões instaladas e alternar entre elas por projeto:

nvm install 20
nvm use 20
nvm alias default 20

npm 9+

O npm (Node Package Manager) vem instalado com o Node.js, mas pode estar desatualizado em instalações antigas. A versão 9 introduziu melhorias no lockfile e na resolução de pacotes que são importantes para a estabilidade do AIOX.

npm --version
# Output esperado: 9.x.x ou superior

# Para atualizar o npm para a versão mais recente:
npm install -g npm@latest

Git

Git é obrigatório. O AIOX integra profundamente com Git para controle de versão de stories, branches por feature, e o ciclo de delivery via @devops. Sem Git, metade dos workflows do AIOX não funciona.

git --version
# Output esperado: git version 2.x.x

# Configure sua identidade se ainda não fez:
git config --global user.name "Seu Nome"
git config --global user.email "seu@email.com"

Certifique-se de que o repositório do projeto está inicializado (git init ou clonado de um remoto). O AIOX não inicializa o repositório Git por você — assume que o versionamento já está configurado.

GitHub CLI (gh)

O GitHub CLI é necessário para as operações que envolvem o repositório remoto: criação de Pull Requests, verificação de status de PRs, e integração com Actions. É de uso exclusivo do @devops (Gage), mas precisa estar instalado no ambiente.

gh --version
# Output esperado: gh version 2.x.x

# Para autenticar com sua conta GitHub:
gh auth login
# Siga as instruções interativas

# Para verificar autenticação:
gh auth status

Se você trabalha em uma organização com SSO (Single Sign-On), você pode precisar de autorização adicional:

gh auth login --scopes "read:org,workflow"

Editor com Suporte a Markdown e YAML

O AIOX é primariamente um sistema de arquivos de texto. Você vai criar, editar, e revisar arquivos Markdown e YAML constantemente. Um editor que renderiza Markdown, valida YAML, e tem bom suporte a arquivos de texto torna o trabalho significativamente mais confortável.

VS Code é o mais recomendado, com as extensões: - yzhang.markdown-all-in-one — Preview e edição de Markdown - redhat.vscode-yaml — Validação de YAML - formulahendry.auto-rename-tag — Se você trabalha com HTML também

Cursor também funciona bem e tem integração direta com modelos de IA.

Outros editores como Neovim, Zed, ou JetBrains IDEs funcionam — mas as extensões específicas variam.

Conforto com Terminal

Este pré-requisito não tem número de versão, mas é o mais importante: você precisa estar confortável usando o terminal. O AIOX é CLI First por princípio. Se você evita o terminal, vai lutar constantemente com o framework em vez de ser amplificado por ele.

"Confortável" não significa ser um expert em shell scripting. Significa: - Saber navegar entre diretórios - Entender output de comandos básicos - Saber o que fazer quando um comando falha com erro - Não ter medo de rodar comandos que modificam o sistema

Se você está desconfortável com o terminal, invista algumas horas em um tutorial básico de linha de comando antes de prosseguir. O retorno sobre esse investimento, trabalhando com AIOX, vai ser enorme.

6.2 Instalação por Projeto: Por Que e Como

Uma decisão arquitetural fundamental do AIOX é que ele é instalado por projeto, não globalmente. Isso é intencional e tem consequências importantes que você precisa entender.

Por Que Por Projeto, Não Global?

Versionamento por projeto. Projetos diferentes podem precisar de versões diferentes do AIOX. Um projeto legado pode ter sido iniciado com AIOX 1.x e ainda não migrou. Um projeto novo usa AIOX 2.x com features que quebrariam o projeto legado. Instalação por projeto garante que cada projeto usa a versão correta.

Configurações independentes. As configurações do AIOX — quais agentes estão disponíveis, quais workflows estão ativos, quais critérios de qualidade são obrigatórios — variam por projeto e por organização. Uma instalação global conflitaria entre projetos com configurações diferentes.

Sem poluição de ambiente global. Ferramentas instaladas globalmente podem criar conflitos de versão, poluir o PATH, e causar comportamentos inesperados em projetos não relacionados. Por projeto, o ambiente fica limpo e previsível.

Reproducibilidade. Quando você clona o repositório do projeto em uma nova máquina ou compartilha com um colega, as configurações do AIOX vêm junto — no package.json como dependência. Um npm install restaura o ambiente completo.

Comandos de Instalação

# Navegue até o diretório do projeto
cd meu-projeto

# Instalação inicial — cria a estrutura completa do AIOX no projeto
npx aiox-core init

# Se o AIOX já foi inicializado e você quer reinstalar:
npx aiox-core install

A diferença entre init e install: - init cria toda a estrutura desde zero — use em projetos novos ou para configuração inicial - install instala ou atualiza o AIOX em um projeto que já tem a estrutura básica

Templates de Projeto

Durante o npx aiox-core init, o sistema solicita que você escolha um template inicial. Três opções disponíveis:

Template default — Para projetos de desenvolvimento de software em geral. Inclui todos os 11 agentes, os 4 workflows primários, tasks de desenvolvimento padrão, e configurações de qualidade para TypeScript/JavaScript. Recomendado para a maioria dos casos.

Template minimal — Para projetos que querem o AIOX mas com overhead mínimo. Inclui apenas os agentes essenciais (@dev, @qa, @devops), o SDC simplificado, e configurações básicas. Ideal para projetos solo ou experimentação.

Template enterprise — Para organizações que precisam de governance adicional: aprovações multi-nível, integração com ferramentas de projeto externas, políticas de branch estritas, e relatórios de compliance. Requer configuração adicional para integrar com os sistemas da organização.

6.3 Anatomia do Projeto: Entendendo Cada Diretório

Após o npx aiox-core init, a estrutura do seu projeto terá os seguintes diretórios e arquivos:

seu-projeto/
├── .aiox-core/          # Framework core e templates
├── .claude/             # Configurações do Claude Code
├── docs/
│   ├── stories/         # Arquivos de story (.story.md)
│   ├── prd/             # Product Requirement Documents
│   └── architecture/    # Documentação de arquitetura
├── agents/              # Memória e configuração de agentes
├── core-config.yaml     # Configuração principal do AIOX
└── [seus arquivos]      # O código do seu projeto

Vamos explorar cada parte:

.aiox-core/ — O Núcleo do Framework

Este diretório contém o framework em si: as definições de tasks, templates de stories, checklists de qualidade, personas de agentes, e a Constitution. Você nunca edita esse diretório diretamente (com poucas exceções documentadas).

Subdivisões importantes: - .aiox-core/development/tasks/ — Definições das tasks executáveis - .aiox-core/development/templates/ — Templates para stories, relatórios, etc. - .aiox-core/development/checklists/ — Checklists de validação - .aiox-core/constitution.md — A Constitution formal do framework

.claude/ — Configurações do Claude Code

Este diretório contém regras e configurações específicas para o Claude Code. Ele é lido automaticamente pelo Claude Code ao iniciar uma sessão, carregando as regras do AIOX:

  • .claude/settings.json — Deny rules e allow rules de proteção de camadas
  • .claude/rules/ — Regras contextuais carregadas por relevância

docs/stories/ — Onde o Trabalho Vive

Este é o diretório mais ativo do projeto. Cada story é um arquivo .story.md nomeado com o padrão {epic}.{story}.story.md. Por exemplo, 2.4.story.md é a quarta story do segundo epic.

As stories são o registro vivo do trabalho: criadas pelo @sm, validadas pelo @po, atualizadas pelo @dev durante a implementação, e revisadas pelo @qa. Elas são tanto especificação quanto documentação — descrevem o que será feito e registram o que foi feito.

agents/ — Memória dos Agentes

Cada agente que opera no projeto tem um subdiretório aqui:

agents/
├── dev/
│   └── MEMORY.md
├── qa/
│   └── MEMORY.md
├── architect/
│   └── MEMORY.md
└── ...

O MEMORY.md de cada agente é o único arquivo no diretório .aiox-core/ que você (e os agentes) podem editar livremente — é o mecanismo de memória persistente.

core-config.yaml — A Configuração Central

O arquivo mais importante para customização do AIOX é o core-config.yaml na raiz do projeto. Ele controla o comportamento global do framework:

project:
  name: "meu-projeto"
  type: "webapp"
  stack:
    - "typescript"
    - "node"
    - "postgresql"

boundary:
  frameworkProtection: true  # false para contribuidores do framework

agents:
  enabled:
    - dev
    - qa
    - architect
    - devops
    - pm
    - po
    - sm

quality:
  minPOScore: 7
  qaAutoPass: false
  requireTestsForNewCode: true

workflows:
  sdcEnabled: true
  qaLoopEnabled: true
  maxQaIterations: 5

6.4 O Modelo de 4 Camadas: Entendendo o Que Você Pode e Não Pode Editar

Uma das decisões mais importantes do AIOX é a separação clara entre o que pertence ao framework e o que pertence ao projeto. Essa separação é chamada de modelo de 4 camadas (L1-L4), e entendê-la evita erros comuns que podem corromper o funcionamento do sistema.

L1 — Framework Core (NUNCA modifique)

Paths: .aiox-core/core/, .aiox-core/constitution.md, bin/aiox.js, bin/aiox-init.js

Esta é a camada mais interna do framework: o código que faz o AIOX funcionar, a Constitution que governa o sistema, e os scripts de inicialização. Modificar esses arquivos é equivalente a modificar o código-fonte de uma ferramenta que você está usando — você vai quebrar coisas de formas imprevisíveis.

Proteção: Deny rules no .claude/settings.json bloqueiam qualquer agente de modificar esses arquivos. Mesmo que você instrua um agente a modificar algo nesse caminho, o gate bloqueia.

L2 — Framework Templates (NUNCA modifique)

Paths: .aiox-core/development/tasks/, .aiox-core/development/templates/, .aiox-core/development/checklists/, .aiox-core/development/workflows/, .aiox-core/infrastructure/

Estes são os templates e definições de tasks que o AIOX usa para operar. Modificá-los não quebra o framework imediatamente, mas cria inconsistências: quando o AIOX é atualizado, suas modificações são sobrescritas. E se seus agentes estão treinados para esperar um comportamento específico dessas tasks, modificações silenciosas criam bugs difíceis de diagnosticar.

Extensão: Se você precisa de tasks ou templates customizados, crie-os em .aiox-core/data/ (L3) ou na raiz do projeto, referenciando os templates originais como base.

L3 — Project Config (Mutável com Exceções)

Paths: .aiox-core/data/, agents/*/MEMORY.md, core-config.yaml

Esta é a camada de configuração do projeto. Você pode e deve modificar esses arquivos para customizar o AIOX para seu contexto:

  • core-config.yaml para configurações gerais
  • agents/*/MEMORY.md para acumular memória dos agentes
  • .aiox-core/data/ para dados customizados de configuração (glossário, padrões específicos do projeto, etc.)

Exceções: Mesmo dentro da L3, algumas operações são restritas. Você pode adicionar conteúdo ao MEMORY.md, mas não pode modificar o formato estrutural. Você pode adicionar entradas ao glossário, mas não pode remover definições que outros agentes dependem.

L4 — Project Runtime (SEMPRE modifique)

Paths: docs/stories/, packages/, squads/, tests/, e o código do seu projeto

Esta é a camada onde o trabalho acontece. Stories são criadas, modificadas, e completadas aqui. Código é escrito aqui. Testes vivem aqui. Não há restrições de framework — este é seu território.

O Toggle de Proteção

O core-config.yaml tem uma configuração que controla se as deny rules estão ativas:

boundary:
  frameworkProtection: true

Com true (padrão), as deny rules bloqueiam modificações em L1 e L2. Com false, as deny rules são desativadas — usado por contribuidores do framework que precisam modificar as camadas protegidas.

Regra prática: Se você é usuário do AIOX (e não um contribuidor do framework), deixe sempre frameworkProtection: true. Os bloqueios existem para te proteger.

6.5 Multi-IDE: Usando AIOX com Diferentes Ferramentas

O AIOX foi projetado com Claude Code como IDE primário, mas o sistema pode ser usado com outras ferramentas de IA. A compatibilidade varia por ferramenta, e cada uma tem considerações específicas.

Claude Code (Compatibilidade Total)

O Claude Code é o ambiente para o qual o AIOX foi otimizado. A integração é completa:

  • Leitura automática de .claude/CLAUDE.md e .claude/rules/
  • Execução de commands via @agent *command
  • Suporte completo a deny rules e allow rules
  • Integração com o sistema de handoff de agentes
  • Carregamento automático de memória de agentes

Quando você usa Claude Code com AIOX, todos os recursos do framework estão disponíveis sem configuração adicional.

Gemini CLI (Alta Compatibilidade)

O Gemini CLI da Google tem boa compatibilidade com o AIOX, com algumas limitações:

  • Lê arquivos de contexto mas com formato levemente diferente (adaptar .claude/CLAUDE.md para GEMINI.md)
  • Suporta a maior parte dos workflows, mas gates automáticos precisam ser verificados manualmente
  • Commands funcionam via instrução de texto, não via prefixo * nativo

Para usar AIOX com Gemini CLI, você pode criar um script de sincronização que mantém GEMINI.md atualizado com as regras relevantes do AIOX. O projeto pode incluir um script scripts/sync-gemini-context.py que extrai as seções relevantes do CLAUDE.md e adapta para o formato do Gemini.

Codex CLI (Compatibilidade Parcial)

O Codex CLI da OpenAI tem compatibilidade parcial, focada principalmente na execução de código:

  • Excelente para a fase de implementação de stories
  • Limitado para orchestração multi-agente
  • Sem suporte nativo a gates e deny rules

Uso recomendado: como co-executor de Dex (@dev), para as partes de implementação de código onde a velocidade de geração é crítica.

Cursor (Compatibilidade Limitada)

O Cursor é um editor com IA integrada, não um CLI de agentes. Sua compatibilidade com AIOX é limitada a:

  • Leitura de arquivos de contexto (.cursorrules)
  • Execução de código assistida pela IA
  • Geração de código baseada em stories (fluxo manual)

Para times que usam Cursor como editor principal, a recomendação é usar Claude Code para orquestração do AIOX (criação de stories, validação, QA) e Cursor para a implementação de código em si. Os dois podem coexistir no mesmo projeto.

Scripts de Sincronização

Para projetos que usam múltiplos IDEs, o AIOX inclui utilitários de sincronização de contexto. Esses scripts mantêm os arquivos de contexto de cada IDE sincronizados com as configurações master do AIOX:

# Sincronizar contexto para Gemini CLI
npx aiox-core sync-context --target gemini

# Sincronizar contexto para Cursor
npx aiox-core sync-context --target cursor

# Sincronizar todos os alvos configurados
npx aiox-core sync-context --all

Os scripts de sincronização leem o core-config.yaml, as personas dos agentes, as regras ativas, e produzem arquivos de contexto no formato esperado por cada IDE.

6.6 Diagnóstico: Verificando a Saúde do Sistema

Após a instalação — e sempre que algo parecer estranho — o primeiro comando a rodar é aiox doctor. Ele é o diagnóstico completo do seu ambiente AIOX.

O Que aiox doctor Verifica

O comando executa uma série de verificações em cascata:

Dependências do sistema: - Node.js versão ≥ 18 - npm versão ≥ 9 - Git instalado e configurado - GitHub CLI instalado e autenticado

Estrutura do projeto: - Presença de .aiox-core/ com versão compatível - Presença de core-config.yaml com schema válido - Diretório docs/stories/ existe - Diretório agents/ com MEMORY.md para cada agente habilitado

Configuração: - core-config.yaml tem todos os campos obrigatórios - Agentes habilitados têm arquivos de configuração correspondentes - Workflows habilitados têm tasks correspondentes disponíveis

Integridade de stories ativas: - Stories em InProgress têm branch Git correspondente - Stories em InReview têm agente @qa identificado - Sem stories em estados inválidos ou inconsistentes

Conectividade (quando aplicável): - GitHub acessível (para operações de @devops) - Modelos de IA acessíveis

Saída do Comando

O output do aiox doctor usa um formato visual claro:

AIOX Doctor — Verificação de Ambiente
══════════════════════════════════════

✓ Node.js v20.11.0 (mínimo: 18.0.0)
✓ npm 10.2.4 (mínimo: 9.0.0)
✓ Git 2.43.0 configurado
✓ GitHub CLI 2.45.0 autenticado como @seu-usuario

✓ .aiox-core/ presente (versão 2.1.3)
✓ core-config.yaml válido
✓ docs/stories/ presente
✓ agents/ presente com 8 agentes configurados

⚠ Story 3.2.story.md está InProgress mas branch feat/story-3-2 não encontrado
  → Execute: git checkout -b feat/story-3-2

✓ Sem outros problemas encontrados

Estado geral: AVISO (1 problema menor)

A Flag --fix

Para problemas que o sistema pode corrigir automaticamente, a flag --fix tenta a correção:

npx aiox-core doctor --fix

Problemas que --fix consegue corrigir automaticamente: - Diretórios faltantes (agents/qa/ não existe → cria com MEMORY.md vazio) - MEMORY.md faltando para um agente habilitado → cria com template padrão - Schema do core-config.yaml desatualizado → migra para nova versão

Problemas que --fix NÃO corrige (requerem ação manual): - Branches Git faltantes para stories ativas - Autenticação do GitHub CLI - Versões de Node.js incompatíveis

Para esses casos, o doctor exibe as instruções precisas do que você precisa fazer.

O Comando aiox info

Complementar ao doctor, o aiox info exibe informações sobre o estado atual do sistema sem fazer verificações de saúde:

npx aiox-core info

Output típico:

AIOX Info — Estado do Sistema
═══════════════════════════════

Projeto: meu-projeto (webapp/typescript)
Stories ativas: 2 (1 InProgress, 1 InReview)
Agentes habilitados: 8 (dev, qa, architect, devops, pm, po, sm, analyst)
Workflows ativos: SDC, QA Loop
Proteção de framework: ATIVA

Último commit: feat: implement story 3.1 (há 3 horas)
Branch atual: feat/story-3-2

Use aiox info quando quiser entender rapidamente em que estado o projeto está, sem a verbosidade do doctor.

6.7 Na Prática: Bootstrap Completo de um Projeto do Zero

Até aqui você viu cada componente isoladamente: pré-requisitos, instalação, anatomia de diretórios, camadas, editores, diagnóstico. Agora vamos juntar tudo em uma sequência prática — do terminal vazio ao projeto configurado, com GitHub, editor, e agentes prontos para trabalhar.

Esta seção reproduz o processo exato demonstrado em aula e que todo operador AIOX deveria ser capaz de executar de olhos fechados. A recomendação é repetir esse processo quantas vezes forem necessárias até virar automático — como músculo. Não adianta assistir e entender. Precisa fazer.

Passo 1: Criar a Pasta do Projeto

Sempre comece com uma pasta vazia. Esse detalhe é crítico: se você instalar o AIOX em uma pasta que já contém um projeto, o instalador vai encontrar arquivos existentes (como .claude/CLAUDE.md) e apresentar um prompt de merge/overwrite que pode confundir quem está começando.

mkdir meu-projeto
cd meu-projeto

No Windows com WSL, lembre-se de que suas pastas ficam acessíveis tanto pelo terminal quanto pelo Explorador de Arquivos. No Mac, o Finder mostra a mesma pasta — use Cmd+Shift+. para alternar a visibilidade de arquivos ocultos.

Passo 2: Instalar o AIOX

npx aiox-core install

O instalador vai apresentar uma série de perguntas. As opções recomendadas para quem está começando:

Pergunta Opção Recomendada Observação
Idioma Português —
Modo Assistido (primeira opção) Modo avançado para operadores experientes
Tipo de projeto Greenfield Para projetos novos do zero
Ferramentas de IA Selecione com barra de espaço Claude, Gemini, Cursor, Anti-Gravity — marque todas que pretende usar
Plano Community (Free) Primeira opção. O Pro adiciona features avançadas
Framework Next.js (primeira opção) Ou conforme seu stack

Uma observação importante: o AIOX é um organismo vivo. Ele evolui constantemente. Pode ser que hoje você digite npx aiox-core install e as perguntas estejam de um jeito. Amanhã, podem estar ligeiramente diferentes — uma opção nova aqui, uma reformulação ali. Não tente decorar as telas. Entenda o raciocínio por trás de cada escolha: tipo de projeto (greenfield vs brownfield), ferramentas que você usa, nível de experiência. A lógica não muda mesmo quando a interface evolui.

Passo 3: Abrir o Claude Code

Com o AIOX instalado, abra o Claude Code no terminal:

claude

Este é o ambiente onde os agentes vivem. A partir daqui, você pode ativar qualquer agente com @ e executar commands com *.

Passo 4: Abrir o Editor de Código

Em paralelo ao terminal, abra seu editor preferido na mesma pasta do projeto. Três opções principais:

VS Code — O editor clássico, com extensões para Markdown e YAML. Sólido e confiável.

Cursor — Editor com IA integrada, popular entre os desenvolvedores. Boa compatibilidade com AIOX quando selecionado na instalação.

Anti-Gravity — Editor com IA integrada, similar ao Cursor em conceito. Quando você seleciona Anti-Gravity durante a instalação do AIOX, os agentes aparecem automaticamente na área de chat do editor. Suporta terminal integrado (Ctrl+J ou Cmd+J), visualização de arquivos, e acesso direto aos mesmos agentes que você usa no terminal.

Qualquer um dos três funciona. O editor serve para visualizar e navegar pelos arquivos enquanto o terminal (ou o terminal integrado do editor) é onde os agentes operam.

Passo 5: Bootstrap com @devops

Este é o passo que conecta tudo — GitHub, estrutura de projeto, e ambiente de desenvolvimento:

@devops *bootstrap

O bootstrap vai fazer uma série de perguntas:

  1. Nome do projeto — confirme o padrão (geralmente o nome da pasta)
  2. Organização/usuário do GitHub — seu usuário pessoal do GitHub
  3. Visibilidade — privado (recomendado para projetos pessoais)
  4. Instalar dependências adicionais — não (a menos que você saiba o que precisa)

Após responder, o bootstrap: - Inicializa o repositório Git local - Cria o repositório remoto no GitHub (via gh) - Configura o .gitignore adequado - Faz o primeiro commit e push - Deixa o projeto pronto para desenvolvimento

Verificação: Após o bootstrap, abra seu GitHub no navegador e dê um refresh. O repositório do projeto deve aparecer lá — criado automaticamente pelo Devops.

Passo 6: Configurar Permissões (Opcional)

Por padrão, o Claude Code pede confirmação para cada operação que modifica arquivos. Isso é seguro, mas pode se tornar repetitivo quando você já sabe o que está fazendo. Existem duas formas de ajustar:

Método 1: settings.local.json (Recomendado)

Abra o arquivo .claude/settings.local.json no editor e adicione as permissões que deseja liberar. Esse método é granular — você escolhe exatamente o que autorizar.

Após modificar o arquivo, você precisa sair do Claude Code (exit) e abrir novamente (claude) para que as mudanças tenham efeito.

Método 2: Flag de linha de comando (PERIGOSO)

claude --dangerously-skip-permissions

O nome do flag não é acidental. Com essa opção, o Claude Code pode fazer qualquer coisa na sua máquina sem pedir permissão — incluindo deletar arquivos, modificar configurações do sistema, ou formatar o disco. Use apenas quando você tem absoluta certeza do que está fazendo e entende os riscos envolvidos.

A recomendação para quem está aprendendo: use o Método 1 ou simplesmente conviva com os prompts de permissão. A segurança vale mais do que a conveniência.

Passo 7: Verificar e Praticar

Neste ponto, você tem: - ✓ Pasta do projeto criada - ✓ AIOX instalado - ✓ Claude Code operacional - ✓ Editor aberto com os arquivos - ✓ Repositório no GitHub configurado - ✓ Permissões ajustadas (opcional)

O projeto está pronto para o primeiro ciclo de desenvolvimento. A partir daqui, o próximo passo é ativar um agente e começar a trabalhar — seja com @pm *create-prd para criar um PRD, ou @sm *create-story para criar uma story, ou qualquer outro ponto de entrada que faça sentido para o seu contexto.

Erros Comuns na Instalação

Instalar em pasta com projeto existente: O instalador encontra arquivos como .claude/CLAUDE.md e apresenta prompt de merge/backup/overwrite. Solução: sempre crie uma pasta vazia antes de instalar.

Selecionar Pro em vez de Community: Se você acidentalmente selecionou o plano Pro, pode reinstalar na mesma pasta. O instalador vai perguntar sobre backup — aceite o backup e prossiga selecionando Community na próxima vez.

Erro de autenticação no bootstrap: Se o @devops *bootstrap falha ao criar o repositório, verifique se o GitHub CLI está autenticado: gh auth status. Se não estiver, execute gh auth login no terminal fora do Claude Code.

Bootstrap em pasta errada: Se você executou o bootstrap em uma pasta que já tinha um projeto, o comportamento pode ser imprevisível. A regra é simples: pasta nova, instalação limpa, bootstrap do zero. Na dúvida, comece de novo.

AIOX Pro não ativa no Mac: Limpe o cache do NPX (npx clear-npx-cache ou delete ~/.npm/_npx/), teste em uma pasta nova, verifique se o nome da pasta não contém espaços ou caracteres especiais. O license server valida por IP (máximo 3 dispositivos). Para reset de senha, contate o suporte.

iCloud e OneDrive interferem: Serviços de sincronização de nuvem (iCloud, OneDrive, Google Drive) podem corromper arquivos do AIOX enquanto agentes escrevem neles. Mantenha seus projetos AIOX em pastas fora de diretórios sincronizados.

6.8 Permissões e Segurança

O Flag --dangerously-skip-permissions

O Claude Code pede confirmação para cada operação de escrita e execução de comando. Isso é seguro, mas lento para desenvolvimento ativo. A recomendação do cohort é clara: use --dangerously-skip-permissions combinado com deny rules em .claude/settings.json.

As deny rules atuam como guardrails: bloqueiam operações destrutivas independente do modo de permissão.

{
  "deny": [
    "rm -rf",
    "mkfs",
    "dd",
    "chmod 777",
    "DROP TABLE",
    "TRUNCATE"
  ]
}

Hooks para Automação Determinística

Hooks são comandos shell que executam automaticamente em resposta a eventos no Claude Code. Eles são 100% determinísticos — diferente de skills (50-80%) ou agentes (variável).

Exemplo prático: injetar a data atual no início de cada sessão via hook SessionStart:

#!/bin/bash
echo "Data atual: $(date '+%Y-%m-%d %H:%M:%S')"

Hooks são configurados em settings.json e executam antes que o agente processe qualquer coisa — garantindo que informações contextuais críticas estejam sempre disponíveis.
