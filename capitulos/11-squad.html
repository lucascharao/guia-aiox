11.1 O Que É Um Squad

Quando a maioria das pessoas pensa em usar IA para trabalhar com um domínio específico de conhecimento — uma área de produto, uma vertical de mercado, um conjunto de práticas — elas pensam em "treinar um modelo" ou "criar um prompt especializado". Ambas as abordagens funcionam para casos simples. Nenhuma escala bem.

Um squad no AIOX é algo fundamentalmente diferente. E a primeira coisa a entender é o que um squad NÃO é:

  • Não é um prompt. Um prompt é uma instrução. Um squad é uma formação.
  • Não é um agente. Um agente é um indivíduo com persona e skills. Um squad é um time.
  • Não é um workflow. Um workflow é uma sequência de tasks. Um squad é quem executa.
  • Não é uma task. Uma task é uma unidade de trabalho. Um squad é a equipe que a executa.

A analogia mais precisa vem do mundo militar e empresarial: um squad é uma Business Unit (BU) — uma formação com pessoas, processos e entregas próprias.

Um squad é uma equipe de agentes especializados em um domínio específico, cada um com papel, personalidade, e conhecimento acumulado próprio, operando de forma coordenada para resolver problemas daquele domínio. Não é um prompt especializado. Não é um único agente com "contexto extra". É uma equipe com a mesma complexidade e especialização que você esperaria de uma equipe humana real.

Para tornar isso concreto: imagine que você está construindo um produto de investimentos. Você não quer um único agente genérico que "sabe um pouco sobre finanças". Você quer um analista fundamentalista com 10 anos de experiência em avaliação de empresas, um especialista em risco com foco em volatilidade de mercados emergentes, um estrategista de portfólio que pensa em alocação de ativos, e um analista de macroeconomia que conecta cenários globais com movimentos de mercado.

Essas são personas diferentes. Têm backgrounds diferentes. Fazem perguntas diferentes. Chegam a conclusões diferentes e complementares. Juntas, cobrem o domínio de investimentos de uma forma que nenhum agente individual conseguiria.

Um squad encapsula exatamente isso.

11.2 A Anatomia de um Squad

Todo squad no AIOX tem a mesma estrutura base, independente do domínio. Entender essa estrutura é fundamental para criar squads eficazes.

Diretório Base

squads/
└── {squad-name}/
    ├── squad.yaml          # Manifesto do squad
    ├── agents/             # Definições dos agentes
    │   ├── {agent-1}.md    # Persona, skills, autoridade
    │   ├── {agent-2}.md
    │   └── {agent-N}.md
    ├── data/               # Dados de primeiro partido
    │   ├── knowledge/      # Conhecimento do domínio
    │   ├── examples/       # Exemplos e casos de uso
    │   └── references/     # Material de referência
    ├── tasks/              # Tasks específicas do squad
    └── MEMORY.md           # Memória compartilhada do squad

O squad.yaml: O Manifesto

O manifesto define o squad como um todo:

name: investment-analysis
version: 1.0.0
description: "Squad de análise de investimentos"

agents:
  - id: fundamentalista
    persona: agents/fundamentalista.md
    role: "Análise fundamentalista de empresas"
  - id: risk-analyst
    persona: agents/risk-analyst.md
    role: "Análise e gestão de risco"
  - id: macro-strategist
    persona: agents/macro-strategist.md
    role: "Análise macroeconômica e estratégia"

domain:
  primary: "Análise de investimentos"
  verticals: ["Renda variável", "Renda fixa", "Fundos"]
  geography: ["Brasil", "Global"]

data:
  first_party:
    - data/knowledge/
    - data/references/

commands:
  - name: analyze-company
    description: "Análise completa de uma empresa"
    agents: [fundamentalista, risk-analyst, macro-strategist]
  - name: portfolio-review
    description: "Review de portfólio com análise de risco"
    agents: [risk-analyst, macro-strategist]

Os Agentes do Squad

Cada agente em um squad tem uma definição que segue o mesmo padrão dos agentes de desenvolvimento do AIOX — persona, skills, autoridade — mas especializada para o domínio do squad.

Um agente de squad não tem as mesmas autoridades que um agente de desenvolvimento. Ele não faz git push, não cria stories. Ele tem autoridade sobre as operações do seu domínio específico: analisar empresas, avaliar riscos, formular estratégias.

A diferença central entre um agente de squad e um agente de desenvolvimento é o domínio de ativação. Um agente de desenvolvimento é ativado por um humano num fluxo de trabalho de software. Um agente de squad pode ser ativado por humanos, por outros agentes, ou por workflows automatizados — e seu output é conhecimento, análise, ou decisões, não código.

A Memória do Squad: MEMORY.md

Cada squad tem um arquivo MEMORY.md que funciona como a mente coletiva da equipe. Diferente do MEMORY.md individual de cada agente de desenvolvimento, a memória do squad captura:

  • Decisões de análise feitas anteriormente sobre entidades do domínio
  • Padrões identificados ao longo do tempo
  • Preferências calibradas do operador
  • Casos especiais e exceções estabelecidas
  • Contexto acumulado sobre o domain específico

Essa memória compartilhada é o que diferencia um squad maduro de um squad recém-criado. Com o tempo, o squad "aprende" — não no sentido de fine-tuning de modelo, mas no sentido de acumulação de contexto relevante que melhora a qualidade das análises.

11.3 Squad vs. Agente: Quando Cada Um

A diferença entre usar um agente individual e usar um squad não é apenas de complexidade — é de natureza do problema.

Quando Um Agente Individual É Suficiente

Use um agente individual quando:

  • O problema tem uma perspectiva dominante (é essencialmente um problema de desenvolvimento, ou essencialmente um problema de qualidade)
  • A tarefa é bem delimitada e não exige múltiplas especialidades
  • Você precisa de velocidade e a profundidade de análise não é crítica
  • O contexto é específico de um projeto, não de um domínio

Exemplo: "Implemente esta feature de autenticação." Isso é um problema de desenvolvimento. @dev é o agente certo.

Quando Um Squad É Necessário

Use um squad quando:

  • O problema requer múltiplas perspectivas especializadas
  • A qualidade da análise depende de expertise acumulada no domínio
  • O output precisa ser consistente ao longo do tempo (não apenas numa sessão)
  • Você quer que o sistema "aprenda" sobre o domínio com o uso
  • O problema não é de desenvolvimento de software, mas de outro domínio

Exemplo: "Analise esta empresa para potencial de investimento." Isso requer perspectivas fundamentalista, macroeconômica, e de risco. Um único agente genérico vai produzir análise superficial. Um squad especializado vai produzir análise com profundidade real.

A Hierarquia de Uso

Em termos de profundidade e especialização:

Prompt direto → Agente individual → Squad especializado

Cada nível adiciona especialização, consistência, e profundidade — mas também adiciona setup e custo de tokens. Escolha o nível certo para o problema em mãos.

11.4 O Sistema de Tiers: Organizando a Complexidade

Squads não são todos iguais em escopo ou responsabilidade. O AIOX organiza squads em um sistema de tiers que define sua posição na hierarquia operacional:

Tier 0 — Fundação

Squads de infraestrutura que servem como base para todos os outros. São "meta-squads" — squads que gerenciam squads. Exemplo: o Squad Creator, que cria e valida outros squads, opera no Tier 0.

Tier 1 — Tático

Squads que coordenam operações entre múltiplos squads de execução. São os "gerentes" — decidem prioridade, alocam recursos, resolvem conflitos. O gap mais comum em projetos AIOX é a ausência do tier tático. Muitos operadores criam squads de execução (Tier 2) sem um squad tático que os coordene — e depois se perguntam por que os outputs são inconsistentes.

Tier 2 — Operacional

Squads de execução que produzem outputs concretos de um domínio específico. A maioria dos squads que você vai criar são Tier 2: squads de copy, de análise, de design, de pesquisa.

Tier 3 — Experts Cirúrgicos

Squads ultra-especializados que fazem uma única coisa com profundidade extrema. Um squad que só faz análise de risco de crédito, ou que só escreve cláusulas contratuais de M&A. São chamados "cirúrgicos" porque atuam com precisão máxima em um domínio mínimo.

Serviços Compartilhados entre Squads

Squads não operam em isolamento completo. O AIOX suporta serviços compartilhados que qualquer squad pode consumir:

  • Registry: Catálogo centralizado de todos os squads e seus commands
  • Refresh Registry: *refresh-registry atualiza os slash commands disponíveis quando squads são adicionados ou modificados
  • Tools Discovery: *discovery-tools mapeia ferramentas, MCPs e capacidades disponíveis no ambiente, evitando duplicação entre squads

Esses serviços compartilhados evitam a duplicação mais comum em projetos com múltiplos squads: cada squad reinventando a roda de buscar ferramentas ou registrar seus commands.

11.5 Quando Criar Um Squad

A decisão de criar um squad deve ser guiada por critérios claros. Criar um squad desnecessariamente adiciona overhead sem benefício. Não criar um squad quando necessário resulta em análises inconsistentes e superficiais.

Os 5 Sinais de Que Você Precisa de Um Squad

Sinal 1: Você está explicando o mesmo contexto de domínio repetidamente. Toda vez que você abre uma sessão, gasta 10-15 minutos re-explicando o contexto do domínio para a IA. Isso é sinal de que esse contexto precisa ser encapsulado num squad com memória persistente.

Sinal 2: O problema exige múltiplas perspectivas especializadas. Quando a resposta de uma única perspectiva é insuficiente e você precisa de análise a partir de ângulos diferentes — risco e retorno, tecnologia e negócio, longo prazo e curto prazo — você precisa de agentes com especialidades distintas.

Sinal 3: A qualidade da análise melhora com o acúmulo de casos. Quando cada análise que você faz deveria informar as próximas (você está vendo padrões, calibrando julgamentos, refinando critérios), a memória persistente do squad é o que permite isso.

Sinal 4: O problema é recorrente e de domínio estável. Se você vai usar essa capacidade de análise repetidamente ao longo do tempo, o investimento de criar um squad é justificado. Para problemas únicos ou esporádicos, um agente individual com contexto ad-hoc é mais eficiente.

Sinal 5: Você quer que outros usem a capacidade sem conhecer o domínio. Se você quer que sua equipe, seus clientes, ou outros sistemas usem a expertise de domínio sem precisar entender o domínio profundamente, um squad empacota essa expertise de forma acessível.

Os Domínios Mais Comuns

Squads são criados para uma variedade de domínios. Os mais comuns em projetos AIOX:

Squads de Análise de Negócio: Análise competitiva, análise de mercado, due diligence de empresas, avaliação de oportunidades.

Squads de Conteúdo: Criação de conteúdo para nichos específicos — saúde, finanças, tecnologia, jurídico. Cada agente tem uma voz e especialidade diferente dentro do nicho.

Squads de Research: Pesquisa científica, análise de literatura, síntese de estudos em áreas específicas.

Squads de Produto: Análise de UX, pesquisa de usuário, ideação de features, priorização de roadmap.

Squads de Suporte: Squads especializados em responder perguntas sobre domínios complexos — compliance, regulação, configuração de sistemas.

11.6 Os Squads Oficiais do AIOX

O AIOX vem com um conjunto de squads oficiais pré-configurados. Esses squads foram desenhados para os domínios mais comuns e servem como referência de qualidade para squads customizados.

Squad: book-squad

O squad para criação de livros e conteúdo longo de alta qualidade. Composto por:

  • Estrategista de conteúdo: Define estrutura, audiência, e objetivos do livro
  • Pesquisador: Faz research de mercado, benchmarks, e fontes
  • Escritor principal: Produz o texto com voz consistente
  • Editor: Revisa qualidade, consistência, e clareza
  • Formatador: Garante formatação e estrutura técnica

Squad: analytics-squad

Squad para análise de dados e métricas de negócio:

  • Analista de dados: Interpreta datasets e métricas
  • Estatístico: Valida metodologia e significância
  • Storyteller de dados: Transforma análise em narrativa executiva

Squad: research-squad

Squad para pesquisa e síntese de informação:

  • Pesquisador primário: Coleta e valida fontes
  • Analista de padrões: Identifica tendências e conexões
  • Sintetizador: Compila insights em documentos coesos

11.7 Níveis de Distribuição de Squads

Squads podem operar em diferentes níveis de distribuição — do uso pessoal até squads como serviço para múltiplos usuários.

Nível 1: Uso Pessoal

O squad vive no seu projeto AIOX local. Você é o único operador. O squad acessa seus dados locais, tem memória que só você alimenta, e produz outputs para seu uso pessoal ou profissional.

Este é o ponto de entrada. Ideal para squads de análise pessoal, criação de conteúdo individual, ou exploração de novas capacidades.

Nível 2: Uso em Equipe

O squad é versionado no repositório do projeto e a equipe inteira tem acesso. A memória é compartilhada (arquivo MEMORY.md no repositório) e os dados de primeiro partido são centralizados.

Este nível requer atenção ao controle de acesso: quem pode alimentar dados para o squad? Quem pode modificar as personas dos agentes? Essas decisões precisam ser explícitas.

Nível 3: Squad como API

O squad expõe uma interface de API que outros sistemas podem chamar. Um sistema de CRM pode chamar o squad de análise de clientes. Um pipeline de dados pode chamar o squad de research para enriquecer registros.

Esse nível envolve empacotamento do squad como serviço e é tratado em detalhe no Capítulo 13.

Nível 4: Squad como Produto

O squad se torna um produto que usuários externos acessam através de uma interface. Pode ser um CLI, uma interface web, um bot de Slack, ou uma integração com outro sistema.

Este é o nível de productização máximo — o squad deixa de ser uma ferramenta interna e se torna um produto entregável.

Conectando Squads e o Ciclo de Desenvolvimento

Uma pergunta comum: squads e o SDC são coisas separadas?

A resposta é: são complementares e integrados.

O SDC governa o desenvolvimento do software que implementa os squads, ou que usa os squads como componente. Se você está construindo um produto onde um squad de análise é um feature, o desenvolvimento desse produto segue o SDC normal. O squad é o que é entregue pelo produto, não o processo de desenvolvimento do produto.

Pense assim: você usa o SDC para construir o carro. O squad é o motor do carro. São camadas diferentes que trabalham juntas.
