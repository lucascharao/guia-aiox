8.1 O SDC em Visão Geral

Todo sistema de desenvolvimento profissional tem um ciclo de vida. No AIOX, esse ciclo se chama Story Development Cycle, ou simplesmente SDC. Ele é o coração pulsante do framework — o processo que transforma uma ideia no backlog em código versionado e entregue.

Antes de entrar nos detalhes de cada fase, é importante entender por que o SDC existe e por que ele tem exatamente essa forma.

O desenvolvimento de software sem processo tem um padrão universal de falha: alguém tem uma ideia, alguém implementa (nem sempre a mesma pessoa que entendeu a ideia), o código vai para produção sem revisão adequada, e bugs surgem porque ninguém validou se a implementação correspondia ao requisito original. O post-mortem revela sempre a mesma coisa: "ninguém especificou claramente o que precisava ser feito", ou "ninguém revisou antes de fazer deploy", ou "não havia critérios de aceite claros".

O SDC resolve esses três problemas estruturais de uma vez. Ele força que toda unidade de trabalho passe por quatro portões antes de ser entregue:

Criação com contexto suficiente → Validação antes da implementação → Implementação dentro do escopo → QA antes do deploy

O fluxo completo envolve cinco agentes em sequência, cada um com autoridade exclusiva sobre sua fase:

@sm (River)   → Fase 1: Criar a story
@po (Pax)     → Fase 2: Validar a story
@dev (Dex)    → Fase 3: Implementar
@qa (Quinn)   → Fase 4: QA Gate
@devops (Gage)→ Entrega: Push e PR

Essa sequência não é arbitrária. Cada transição representa uma mudança de responsabilidade e uma validação de qualidade. Pular qualquer uma dessas fases é possível tecnicamente — o AIOX não te impede fisicamente de fazer git push com @dev. Mas fazer isso é como dirigir em alta velocidade sem cinto de segurança. Você pode chegar ao destino. Mas quando algo dá errado, as consequências são piores do que precisariam ser.

Vamos percorrer cada fase em detalhe.

8.2 Fase 1: Criar a Story — @sm (River)

Ativando River

Para iniciar o SDC, você ativa o Scrum Master:

@sm

River entra em modo ativo e responde com um resumo do estado atual: epic ativo, stories pendentes, e o que está pronto para a próxima story a ser criada. Se não há epic ativo, River vai pedir que você defina um ponto de partida antes de continuar.

O comando central de River nessa fase é:

@sm *create-story

Esse comando executa a task create-next-story.md — um dos processos mais importantes do framework. Essa task não é simplesmente "escreva uma story". É um processo estruturado de elicitação de contexto, validação de dependências, e formatação segundo o template oficial.

O Processo create-next-story

Quando você executa *create-story, River vai passar por um processo de elicitação que inclui:

1. Identificar o epic pai. Toda story pertence a um epic. River verifica qual é o epic ativo e extrai o contexto relevante: objetivos, critérios de sucesso do epic, e as stories já existentes para evitar duplicação.

2. Identificar a próxima story natural. Com base no epic e nas stories já criadas, River sugere qual é a próxima unidade de trabalho lógica — o menor incremento que entrega valor sem criar dependências circulares.

3. Elicitar os critérios de aceite. Essa é a parte mais crítica. River não aceita critérios vagos. Cada critério de aceite precisa ser verificável: "o botão de submit funciona" não é um critério — "ao clicar em submit com formulário válido, a requisição POST /api/users retorna HTTP 201 em menos de 500ms" é um critério.

4. Identificar dependências e riscos. River verifica se há dependências de outras stories ou de sistemas externos, e mapeia os riscos técnicos que o desenvolvedor vai precisar considerar.

5. Estimar o esforço. Uma estimativa simples de complexidade — não em horas, mas em pontos ou tamanho relativo — que ajuda a dimensionar se a story está bem delimitada ou se é grande demais para uma única iteração.

O Template de Story

O output de *create-story é um arquivo no formato docs/stories/{epicNum}.{storyNum}.story.md. O template padrão tem a seguinte estrutura:

# Story {epicNum}.{storyNum}: {Título}

## Status
Draft

## Context
{Contexto relevante para quem vai implementar}

## Requirements
{Lista numerada de requisitos funcionais}

## Acceptance Criteria
- [ ] AC-1: {Critério verificável}
- [ ] AC-2: {Critério verificável}
- [ ] AC-3: {Critério verificável}

## Technical Notes
{Notas técnicas, restrições, considerações de arquitetura}

## Dependencies
- {Story ou sistema do qual esta story depende}

## Risk Assessment
- {Risco identificado e mitigação proposta}

## Estimation
{Complexidade estimada}

## File List
{Preenchido durante implementação}

## Dev Notes
{Preenchido durante implementação}

O status Draft é o ponto de partida. Uma story em Draft está pronta para ser validada pelo PO, mas não está pronta para implementação. Essa distinção é fundamental.

Quando Usar Modo draft vs create-story

*draft é uma versão mais rápida do processo — cria o esqueleto da story com menos elicitação, útil quando você tem contexto suficiente e quer iterar rapidamente. *create-story é o processo completo com elicitação guiada, recomendado para qualquer story de complexidade média ou alta, ou quando o contexto ainda não está totalmente claro.

8.3 Fase 2: Validar — @po (Pax)

O Problema que a Validação Resolve

Existe um padrão de falha clássico em desenvolvimento: o desenvolvedor começa a implementar uma story, no meio do caminho percebe que os critérios de aceite são ambíguos, toma uma decisão de implementação baseada na sua interpretação, e entrega algo que não corresponde ao que o stakeholder queria.

O resultado: retrabalho. Às vezes algumas horas. Às vezes dias inteiros.

A Fase 2 existe para eliminar essa possibilidade. Antes de qualquer linha de código ser escrita, Pax verifica se a story está suficientemente bem especificada para implementação sem ambiguidade.

Ativando Pax e Executando a Validação

@po *validate-story-draft

Pax executa a task validate-next-story.md e aplica o checklist de 10 pontos sobre a story em Draft:

Ponto 1: Critérios de aceite são mensuráveis e verificáveis? Cada AC precisa ter uma condição clara de verdadeiro/falso. "Sistema funciona corretamente" falha nesse ponto. "Endpoint retorna HTTP 200 com payload JSON contendo campo id quando autenticado" passa.

Ponto 2: Todos os critérios rastreiam a documentação de requisitos? Os ACs precisam ter rastreabilidade para FR- (Functional Requirements) ou NFR- (Non-Functional Requirements) documentados. Isso garante que não há ACs inventados que não foram solicitados.

Ponto 3: O contexto é suficiente para implementar sem perguntas adicionais? Um desenvolvedor que nunca viu o projeto antes deve conseguir ler a story e saber o que precisa ser feito. Se há lacunas de contexto que obrigariam o desenvolvedor a perguntar, a story volta para completar esse contexto.

Ponto 4: As dependências estão identificadas e em status adequado? Se a story depende de uma outra story que ainda está em Draft, a implementação vai bloquear. Pax verifica se as dependências estão em status compatível com o início da implementação.

Ponto 5: O escopo está claramente delimitado? O que está FORA do escopo é tão importante quanto o que está dentro. Pax verifica se há uma declaração explícita de escopo, prevenindo que o desenvolvedor adicione features não solicitadas (scope creep).

Ponto 6: Os riscos técnicos foram identificados? Toda story tem riscos: integração com sistema externo instável, mudança de comportamento de módulo compartilhado, performance em escala. Pax verifica se esses riscos foram mapeados.

Ponto 7: A estimativa de esforço é razoável para o escopo? Uma story estimada como "pequena" com dezessete critérios de aceite é uma inconsistência. Pax verifica o alinhamento entre escopo e estimativa.

Ponto 8: Os critérios de teste estão contemplados? Como a story será testada? Testes unitários, integração, end-to-end? Isso precisa estar explícito.

Ponto 9: A documentação necessária foi identificada? Algumas features precisam de documentação — um novo endpoint precisa de documentação de API, uma nova configuração precisa de atualização no guia de setup. Pax verifica se isso foi considerado.

Ponto 10: Um desenvolvedor lendo a story saberia exatamente o que fazer? Este é o ponto de síntese. Mesmo que os nove anteriores estejam bem respondidos, Pax faz a pergunta holística: alguém com o background técnico adequado conseguiria implementar isso sem bloquear uma vez?

A Decisão GO / NO-GO

O resultado da validação é um score de 0 a 10:

  • Score >= 7 (GO): A story avança para status Ready. Está pronta para implementação.
  • Score 5-6 (NO-GO com lista): A story volta para revisão com a lista específica dos pontos que falharam. River ou o humano corrige e submete novamente.
  • Score <= 4 (REJEIÇÃO): A story precisa ser reescrita. Há problemas fundamentais de especificação que correções parciais não vão resolver.

Esse gate pode parecer burocrático quando você está em ritmo acelerado. Não é. Cada minuto investido em validar uma story antes da implementação evita potencialmente horas de retrabalho depois.

8.4 Fase 3: Implementar — @dev (Dex)

Ativando Dex

@dev

Dex assume o controle com contexto da story em Ready. O comando central desta fase é:

@dev *task dev-develop-story

Isso executa a task dev-develop-story.md — o processo completo de implementação guiado pelo AIOX.

Os 3 Modos de Implementação

O AIOX oferece três modos de implementação, cada um com trade-offs diferentes entre velocidade, controle humano, e segurança.

Modo Interactive (Padrão)

No modo Interactive, Dex apresenta o plano de implementação antes de cada decisão técnica significativa e pede confirmação antes de avançar. Ideal para:

  • Features que você não entende completamente antes de começar
  • Código em áreas críticas ou sensíveis
  • Sua primeira implementação num projeto novo
  • Qualquer situação onde você quer manter controle próximo

O fluxo parece assim:

Dex: "Vou criar o módulo de autenticação em src/modules/auth/.
      Pretendo usar JWT com refresh token. Confirma?"

Você: "Sim, mas usa 24h de expiração para o access token."

Dex: "Entendido. Implementando com access token de 24h..."

Mais lento, mais controlado, mais seguro para código crítico.

Modo YOLO

No modo YOLO, Dex implementa de forma autônoma sem pedir confirmações intermediárias. Você fornece a story, Dex implementa, e você revisa o resultado no final.

@dev *task dev-develop-story --mode=yolo

Ideal para: - Stories simples e bem especificadas - Features que você entende bem tecnicamente - Bugs fixes com critérios claros - Código em áreas não-críticas

Cuidado: YOLO não significa sem regras. Dex ainda segue todos os padrões do projeto, ainda registra as decisões tomadas, e ainda atualiza a story com o File List. A diferença é que ele não para para pedir permissão a cada passo.

Modo Pre-Flight

Pre-Flight é um modo de revisão antes da implementação. Dex analisa a story, identifica todos os pontos de decisão técnica, e apresenta um plano completo de implementação sem escrever nenhum código. Você revisa o plano, ajusta o que precisar, e depois decide em qual modo executar a implementação real.

@dev *task dev-develop-story --mode=preflight

Ideal para: - Stories de alta complexidade ou alto risco - Features que envolvem decisões arquiteturais - Antes de implementações que vão afetar múltiplos módulos - Quando você quer garantir alinhamento antes de qualquer código

Atualizando Checkboxes e File List

Um dos padrões mais importantes do AIOX durante a implementação é a atualização contínua da story. Dex não escreve código e depois atualiza a story no final — ele atualiza conforme avança.

Cada critério de aceite completado recebe um [x]:

## Acceptance Criteria
- [x] AC-1: Endpoint POST /api/users retorna HTTP 201 com payload correto
- [x] AC-2: Senha é armazenada com bcrypt, mínimo 12 rounds
- [ ] AC-3: Email de boas-vindas é enviado via serviço de email

O File List é atualizado com cada arquivo criado ou modificado:

## File List
- src/modules/auth/auth.controller.ts (criado)
- src/modules/auth/auth.service.ts (criado)
- src/modules/auth/auth.module.ts (criado)
- src/modules/auth/dto/create-user.dto.ts (criado)
- src/app.module.ts (modificado — adicionado AuthModule)
- tests/auth/auth.controller.spec.ts (criado)

Esse registro tem dois propósitos: accountability (você sabe exatamente o que mudou), e facilitação do QA (Quinn sabe exatamente onde olhar).

Git Local: A Regra do @dev

Dex tem autoridade total sobre operações de git no repositório local. Ele pode e deve fazer commits frequentes durante a implementação:

git add src/modules/auth/
git commit -m "feat: implement user registration endpoint [Story 1.3]"

O que Dex não pode fazer é git push. Enviar código para o repositório remoto é responsabilidade exclusiva de Gage (@devops). Essa separação existe para garantir que todo código que vai para o repositório passou pelo gate de QA.

Por que esse limite? Imagine que Dex termina a implementação às 23h, está satisfeito com o resultado, e faz git push diretamente. O código vai para produção (se há CI/CD automático) sem revisão de qualidade. Na manhã seguinte, você descobre que havia um bug crítico que um QA de cinco minutos teria identificado.

O gate de QA existe exatamente para essa situação. E o bloqueio de git push para @dev é o que garante que o gate seja respeitado, mesmo quando a tentação de "só mandar logo" é grande.

8.5 Fase 4: QA Gate — @qa (Quinn)

A Filosofia de Quinn

Quinn não é um inimigo do progresso. É o guardião da qualidade — e existe uma diferença fundamental entre os dois papéis.

Um inimigo do progresso busca encontrar problemas para atrasar entregas. Um guardião da qualidade busca identificar problemas antes que eles cheguem ao usuário final — e portanto está acelerando o progresso no longo prazo, mesmo que às vezes cause atrasos no curto prazo.

Quinn opera com esse mindset: rigoroso sem ser paralisante, criterioso sem ser perfeccionista, e orientado a veredictos claros e acionáveis.

Os 7 Quality Checks

@qa *task qa-gate

Quinn executa a task qa-gate.md e passa a story por 7 verificações de qualidade:

Check 1: Todos os critérios de aceite foram implementados? Quinn verifica cada AC da story contra o código produzido. Cada [x] na story corresponde a uma implementação verificável no código.

Check 2: Os testes cobrem os cenários críticos? Não apenas "existem testes" — mas "os testes cobrem os casos que importam". Cenários felizes, casos de erro, edge cases.

Check 3: O código segue os padrões estabelecidos? Naming conventions, estrutura de módulos, patterns arquiteturais definidos no projeto. Quinn não aceita código funcionalmente correto mas estruturalmente inconsistente.

Check 4: Não há regressões introduzidas? A implementação da story não deve quebrar funcionalidade existente. Quinn verifica os testes existentes e analisa as mudanças no código para identificar potenciais regressões.

Check 5: A documentação foi atualizada quando necessário? Se a story criou um novo endpoint, a documentação de API foi atualizada? Se criou uma nova configuração, o guia de setup foi atualizado? Quinn verifica o File List da story e valida as documentações correspondentes.

Check 6: Não há vulnerabilidades de segurança óbvias? Inputs não sanitizados, senhas hardcoded, queries SQL construídas via concatenação de strings. Quinn não é um security audit completo, mas captura os problemas óbvios.

Check 7: A implementação está dentro do escopo definido? Às vezes desenvolvedores adicionam features não solicitadas por acharem útil — "já que eu estava aqui, adicionei também X". Essas adições não foram especificadas, não foram testadas adequadamente, e introduzem risco. Quinn identifica e questiona qualquer coisa que está no código mas não está na story.

Os 4 Veredictos

O resultado do QA Gate é um dos quatro veredictos:

PASS: Todos os checks passaram. A story está pronta para entrega. O status muda para InReview e o processo avança para @devops.

CONCERNS: Um ou mais checks identificaram problemas que não são bloqueantes mas que precisam de atenção. Quinn documenta os concerns, e a decisão de avançar ou corrigir antes do push é escalada para o humano responsável. Concerns comuns: cobertura de testes ligeiramente abaixo do esperado, documentação incompleta em parte não crítica, refatoração sugerida mas não obrigatória.

FAIL: Um ou mais checks identificaram problemas bloqueantes. A story volta para @dev com lista específica do que precisa ser corrigido. O status volta para InProgress. Não há negociação nesse ponto — código que falha no QA Gate não vai para produção.

WAIVED: Em situações excepcionais e documentadas, um ou mais checks podem ser explicitamente dispensados. Isso requer justificativa documentada na story e é uma decisão consciente, não uma omissão. Exemplos legítimos de waiver: teste de integração impossível em ambiente de CI por dependência de sistema externo sem mock disponível; documentação de API postergada por acordo com stakeholder.

O Status na Story

Ao final do QA Gate, o status da story reflete o resultado:

  • PASS → InReview
  • FAIL → InProgress (volta para @dev)
  • CONCERNS → InReview (com nota)
  • WAIVED → InReview (com justificativa documentada)

8.6 Entrega — @devops (Gage)

A Última Milha

O código passou por criação, validação, implementação, e QA. Está pronto para ir para o mundo. Essa última milha é propriedade exclusiva de Gage.

@devops *push

Gage executa uma sequência de operações que garante que a entrega seja feita de forma controlada e rastreável:

1. Verificação pré-push. Gage confirma que a story está em status InReview, que todos os ACs estão marcados como completos, e que o QA Gate foi executado com resultado PASS ou WAIVED documentado.

2. Git push. O branch com a implementação é enviado para o repositório remoto:

git push origin feat/story-1-3-user-registration

3. Pull Request. Gage cria o PR com título, descrição, e referências adequadas:

gh pr create \
  --title "feat: user registration endpoint [Story 1.3]" \
  --body "## Summary
Implementa o endpoint POST /api/users para registro de novos usuários.

## Story
[Story 1.3](docs/stories/1.3.story.md)

## Changes
- Novo módulo auth com controller, service e DTOs
- Testes unitários com cobertura de 94%
- Documentação de API atualizada

## QA Gate
PASS — Todos os 7 checks passaram."

4. Atualização de status. A story muda de InReview para Done.

Por Que Apenas @devops Pode Fazer Push?

Esta regra está entre as mais importantes — e mais frequentemente questionadas — do AIOX. A resposta tem três dimensões:

Accountability clara. Quando você olha o histórico de merges no repositório, sabe exatamente que cada push passou por Gage. Se algo problemático chegou ao repositório, o ponto de investigação é a entrega feita por Gage — e você pode verificar se o processo foi seguido.

Gate de qualidade final. Gage verifica status, ACs, e QA Gate antes de qualquer push. É um segundo par de olhos no processo completo, garantindo que nada passou pelas brechas.

Separação de concerns. Desenvolvedores focam em implementar. DevOps foca em entregar de forma segura. Misturar essas responsabilidades cria situações onde a pressão para "entregar logo" sobrepõe a necessidade de "entregar com qualidade".

8.7 Retrospectiva: O Que Aconteceu

Ao completar seu primeiro SDC, vale pausar para entender o que acabou de acontecer.

O Que Foi Criado

Uma story que começou como uma ideia no backlog percorreu um caminho estruturado:

  1. Especificação com critérios verificáveis (River)
  2. Validação por um guardião de valor (Pax)
  3. Implementação com rastreabilidade completa (Dex)
  4. QA com checks objetivos (Quinn)
  5. Entrega controlada e documentada (Gage)

O resultado é código no repositório com uma trilha completa: sabe-se por que foi pedido (story), o que foi implementado (File List), como foi testado (QA Gate), e quem aprovou cada etapa (logs de agente).

A Proporção Determinístico vs. IA

Um dos insights mais valiosos que o SDC revela é a proporção entre o que é determinístico e o que realmente exige inteligência artificial.

Python/determinístico: - Scaffolding de arquivos e pastas - Validação de schema da story - Verificação de status (story está em estado correto?) - Contagem de ACs completos vs. pendentes - Checks estruturais do QA (arquivo existe? testes passam?) - Git operations (add, commit, push, pr create)

IA genuinamente necessária: - Avaliar se os critérios de aceite são suficientemente claros - Implementar a lógica de negócio - Identificar riscos técnicos não óbvios - Avaliar se o código segue padrões (não apenas estrutura, mas estilo e intenção) - Identificar potenciais bugs e vulnerabilidades

Em projetos típicos, 60-70% das operações do SDC são determinísticas. A IA é invocada seletivamente para as partes que genuinamente requerem julgamento. Isso tem implicações diretas no custo: cada token economizado em operação determinística é um token disponível para o trabalho que a IA faz melhor.

O Papel do Humano no SDC

Uma pergunta legítima: "Se os agentes fazem tudo isso, qual é o meu papel?"

Seu papel é o de operador consciente. Você:

  • Define o epic e as prioridades de negócio
  • Decide quando a story está pronta para avançar para cada fase
  • Revisa os outputs dos agentes com olhar crítico
  • Toma decisões de escalação quando verdicts são ambíguos
  • Mantém o contexto de produto que nenhum agente pode ter

O AIOX não substitui a inteligência humana no processo. Amplifica ela. Um agente pode validar se um AC é verificável; você decide se o AC correto foi pedido. Um agente pode identificar que o código segue os padrões; você decide se os padrões corretos foram estabelecidos.

Exercício: Execute o SDC Completo

Este exercício vai guiar você pelo seu primeiro SDC completo. Escolha uma feature simples — criação de um endpoint de health check, por exemplo — e execute as cinco fases.

Pré-requisito: Um projeto com epic ativo em docs/ e configuração básica do AIOX.

Passo 1 — Criar a Story:

@sm *create-story

Responda às perguntas de elicitação de River. Defina pelo menos 3 critérios de aceite verificáveis. Salve a story em status Draft.

Passo 2 — Validar:

@po *validate-story-draft

Observe o checklist sendo aplicado. Se o score for < 7, identifique quais pontos falharam e corrija a story. Execute novamente até atingir GO.

Passo 3 — Implementar:

@dev *task dev-develop-story

Use modo Interactive para seu primeiro SDC. Observe Dex atualizar os checkboxes e o File List conforme avança. Note que os commits são feitos localmente.

Passo 4 — QA Gate:

@qa *task qa-gate

Observe Quinn executar os 7 checks. Identifique quais checks são automatizáveis (estrutural) e quais requerem julgamento de IA.

Passo 5 — Entrega:

@devops *push

Observe Gage verificar as condições, fazer o push, e criar o PR.

Reflexão pós-exercício: - Quantos itens do checklist de Pax a sua story inicial falhou? - Em qual modo de implementação você se sentiu mais confortável? Por quê? - Qual dos 7 checks do QA Gate você consideraria mais importante para o seu tipo de projeto? - Qual foi a proporção entre operações determinísticas e operações que genuinamente exigiram julgamento de IA?

O SDC pode parecer pesado na primeira vez. Na décima vez, vai parecer natural. Na centésima, vai ser tão automático quanto commit/push manual — mas com uma qualidade de entrega completamente diferente.

8.8 Caso Prático: Landing Page do Zero ao Deploy

Teoria sem prática é arquitetura sem construção. Esta seção documenta um caso real — demonstrado em aula no dia 26 de março de 2026 — onde o SDC foi aplicado do zero para entregar uma landing page completa de assessoria esportiva individual, com captura de leads armazenada no Supabase e publicada na web via Vercel.

Em aproximadamente 40 minutos de aula, saímos do zero para uma URL pública funcional capturando nome e WhatsApp de visitantes reais. Este é o AIOX em operação.

O Cenário

Uma landing page de vendas para assessoria esportiva individual. Público-alvo: pessoas que treinam mas se sentem sem suporte e direção. O objetivo da página é converter visitantes em leads — capturando nome e WhatsApp — e armazenar esses dados em banco de dados para contato posterior.

Stack definida: Next.js com Tailwind CSS e Shadcn/ui no frontend, Supabase com PostgreSQL como banco, e Vercel como plataforma de deploy.

A Analogia do Restaurante

Antes de detalhar os passos, vale entender a arquitetura do ponto de vista de um leigo. Pense no sistema como um restaurante:

Componente Analogia Tecnologia
GitHub A cozinha — onde tudo é preparado Git/GitHub
Vercel O salão — o que o cliente vê Vercel (deploy)
Supabase O depósito — onde os dados ficam guardados Supabase/PostgreSQL

O GitHub guarda o código e o histórico de tudo que foi feito. O Vercel pega esse código e o transforma em um site acessível por qualquer pessoa com a URL. O Supabase é o banco de dados onde os leads capturados pelo formulário são persistidos.

Nenhum dos três faz sentido isolado. Os três juntos formam um sistema de produção completo.

O Passo a Passo

Passo 1 — Criar um Squad de Copywriting

Antes de chamar o @dev, você precisa do conteúdo. Desenvolvedores sem copy constroem páginas com texto placeholder — e placeholder nunca vira copy de conversão. A ordem certa é: copy primeiro, código depois.

Para isso, o primeiro passo foi criar um squad especializado via Squad Creator:

Crie um squad com um único agente copywriter para escrever copy
de alta conversão para uma landing page. Simples e rápido.

O resultado foi a pasta squads/copywriter/ com a definição do agente e a task correspondente.

Passo 2 — Gerar o Copy

Com o agente copywriter ativo, as coordenadas foram passadas:

  • Produto: Assessoria esportiva individual
  • Público: Pessoas que treinam mas se sentem sem suporte
  • Promessa: Resultados rápidos sem complicação
  • CTA: Capturar nome e WhatsApp

Uma instrução crítica foi incluída de forma explícita: "Salve apenas o copy em markdown, NÃO crie HTML. Arquivo em docs/copy/assessoria-esportiva.md."

Por quê essa instrução? Porque sem ela, um agente de copywriting pode interpretar "criar a página" como criar o HTML diretamente — pulando etapas e gerando um arquivo que conflita com o que o @dev vai criar depois. Copy em markdown é apenas conteúdo; a estrutura HTML será responsabilidade de Dex.

Passo 3 — Criar o PRD com @pm

Com o copy pronto, o Product Manager foi ativado:

@pm *create-prd

O contexto fornecido incluiu: página de vendas simples, copy já disponível em /docs/copy/assessoria-esportiva.md, stack com Tailwind + Shadcn/ui, formulário capturando nome e WhatsApp, dados persistidos no Supabase.

O PRD gerado por Morgan formalizou os requisitos funcionais, os requisitos não-funcionais, os critérios de aceite, a definição de stack e o schema do banco de dados.

Aqui vale uma distinção fundamental que o PRD deixa explícita:

Requisitos funcionais descrevem O QUE o sistema faz: "formulário captura nome e WhatsApp", "dados são salvos no banco", "usuário vê mensagem de confirmação após envio".

Requisitos não-funcionais descrevem COMO o sistema se comporta: "página carrega em menos de 3 segundos", "funciona em dispositivos móveis", "formulário valida formato de WhatsApp antes de enviar".

A distinção importa porque influencia diretamente quais testes precisam ser escritos e quais são os critérios de aceite verificáveis.

Passo 4 — Criar a Story com @sm

Com PRD em mãos, River criou a story:

@sm *create-story

A story resultante continha: critérios de aceite verificáveis, notas de urgência, definição de stack, caminhos de arquivo esperados. Status inicial: Draft.

Passo 5 — Implementar com @dev (modo YOLO)

Para uma story bem especificada como essa — copy pronto, PRD detalhado, stack definida — o modo YOLO foi a escolha certa. Dex implementou de forma autônoma:

  • Criou a pasta landing/ com projeto Next.js completo
  • Instalou as dependências: next, tailwindcss, shadcn/ui, @supabase/supabase-js
  • Construiu os componentes da landing page usando o copy do arquivo markdown
  • Criou .env.local com as variáveis de ambiente placeholder
  • Gerou o SQL para criação da tabela de leads no Supabase

Ao final, a story estava com todos os ACs marcados como [x] e o File List completo.

Passo 6 — Criar o Projeto no Supabase

O banco de dados não se cria sozinho. Acessando supabase.com:

  1. New Project → nome do projeto, senha do banco, região (South America — São Paulo)
  2. Aguardar cerca de 2 minutos para provisionamento
  3. Acessar as credenciais: Project Settings → API → copiar Project URL e anon public key

Um detalhe crítico de segurança: o Supabase oferece duas chaves de acesso. A anon key é para uso no frontend — tem permissões limitadas definidas pelas políticas de RLS. A service_role key tem permissões de administrador completo e nunca deve ser usada no frontend. Exposta no código cliente, ela daria a qualquer visitante da página controle total sobre o banco de dados.

Passo 7 — Configurar o .env.local

Com as credenciais em mãos, o arquivo .env.local foi preenchido:

NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbG...

Esse arquivo nunca vai para o GitHub — está no .gitignore por padrão em projetos Next.js. As variáveis prefixadas com NEXT_PUBLIC_ ficam expostas no bundle do cliente, o que é aceitável para a URL e a chave anônima do Supabase. Variáveis sem esse prefixo ficam apenas no servidor.

Passo 8 — Criar a Tabela no Supabase

No SQL Editor do dashboard do Supabase, o SQL gerado por Dex na Fase 5 foi executado:

CREATE TABLE leads (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  nome TEXT,
  whatsapp TEXT,
  created_at TIMESTAMP DEFAULT NOW()
);

Simples e direto. Cada lead capturado recebe um ID único gerado automaticamente, nome, WhatsApp, e timestamp de criação. Extensões de schema — como campos opcionais ou índices — podem ser adicionadas depois, quando houver necessidade real.

Passo 9 — Push com @devops

Com tudo pronto localmente, Gage foi ativado:

@devops
Sobe meu GitHub com tudo.

Gage executou uma sequência importante antes de qualquer commit:

  1. Verificou que .env.local está no .gitignore — credenciais nunca vão para o repositório
  2. Detectou e removeu o .git interno da pasta landing/ — quando o npx create-next-app inicializa um projeto, ele cria um repositório Git próprio dentro da pasta. Se esse .git interno não for removido, o repositório pai trata a pasta como um submódulo Git, e ela aparece vazia no GitHub
  3. Executou git add + commit + push

O problema do submódulo é um dos mais comuns nesse fluxo — e um dos mais confusos para quem não conhece o comportamento. O sintoma: pasta landing/ aparece no GitHub mas está vazia, sem nenhum arquivo visível. A causa: repositório aninhado. A solução: remover o .git interno antes do push.

Passo 10 — Deploy no Vercel

Com o código no GitHub, o deploy na Vercel seguiu o processo padrão:

  1. vercel.com → Add New → Project → Import from GitHub
  2. CRÍTICO: configurar o Root Directory como landing/ — o Vercel precisa saber que o projeto Next.js está dentro de uma subpasta, não na raiz do repositório
  3. Importar as variáveis de ambiente do .env.local
  4. Deploy → aguardar 2-3 minutos → URL de produção disponível

O detalhe do Root Directory é o segundo erro mais comum nesse fluxo. Sem ele, o Vercel tenta fazer build da raiz do repositório, não encontra um projeto Next.js válido, e falha com erro de configuração.

Passo 11 — Corrigir Erros via Vercel Logs

Em builds reais, erros acontecem. O fluxo de correção é direto:

  1. Copiar o erro do painel de Logs da Vercel
  2. Colar no Claude Code com contexto: "O build falhou com esse erro, corrija"
  3. O agente corrige o problema
  4. @devops faz push
  5. A Vercel detecta o novo commit e re-executa o deploy automaticamente

Esse ciclo de correção é rápido porque a integração GitHub-Vercel é automática. Todo push para a branch configurada dispara um novo build. Você não precisa re-importar o projeto ou fazer configurações adicionais.

Erros Comuns Documentados

Dois problemas se repetiram em diferentes execuções desse fluxo e merecem registro explícito:

Submódulo vazio no GitHub. Causa: npx create-next-app cria um .git interno na pasta landing/. O repositório pai interpreta isso como submódulo Git, e a pasta aparece vazia no GitHub. Solução: antes do primeiro push, Gage remove o .git interno (rm -rf landing/.git) e recomeça o processo de staging.

Vercel não encontra o projeto. Causa: Root Directory não configurado. O Vercel tenta fazer build da raiz do repositório em vez da pasta landing/. Solução: durante a importação do projeto, configurar Root Directory como landing/. Se o projeto já foi importado sem essa configuração, acesse Settings → General → Root Directory no dashboard da Vercel.

Um terceiro problema é menos técnico mas igualmente comum: a página não atualiza depois do deploy. Causa: cache do navegador. Solução: hard refresh com Ctrl+Shift+R (ou Cmd+Shift+R no Mac).

O Que Este Caso Ensina

Quarenta minutos do zero à URL pública. Isso não é marketing — foi o tempo real em aula.

O que torna isso possível não é velocidade imprudente. É a combinação de responsabilidades bem distribuídas: o copywriter fez o copy antes do código existir, o PM formalizou os requisitos antes da implementação começar, o Dev implementou dentro de um escopo claro, e o DevOps garantiu que as credenciais ficaram seguras e o repositório estava limpo antes do push.

Cada etapa fez exatamente o que deveria fazer. Nenhum agente inventou requisitos. Nenhuma etapa foi pulada por pressão de tempo. E o resultado foi um sistema funcional — não um protótipo, não um mockup — um sistema que captura dados reais e os persiste em banco de dados de produção.

Esse é o princípio fundamental do AIOX: a IA amplifica a inteligência humana, não a substitui. As decisões estratégicas — o que construir, para quem, com qual mensagem — foram humanas. A execução — scaffolding, configuração, implementação, deploy — foi dos agentes. A combinação produz resultados em uma fração do tempo que qualquer um dos dois conseguiria sozinho.
