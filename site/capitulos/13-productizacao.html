Capítulo 13 · AVANCADO

Productizando Squads: Do CLI ao Serviço

Jornada de 4 fases: solo → equipe → API → produto. Determinize o que é determinístico, faça routing de modelos e venda pelo valor.

A Jornada de Maturidade

Um squad nasce como uma ferramenta pessoal. Você o usa para analisar startups, gerar conteúdo, ou pesquisar tópicos — e ele funciona bem para você. Mas em algum momento surge uma pergunta: "E se outras pessoas pudessem usar isso?"

Essa pergunta marca o início da jornada de productização. Transformar um squad local em um serviço acessível a múltiplos usuários é uma jornada de quatro fases, cada uma com suas próprias decisões, trade-offs, e ferramentas.

Este capítulo descreve essa jornada em detalhe.

13.1 As 4 Fases da Jornada de Productização

Fase 1: Squad Local (Solo)

O que é: O squad vive no seu projeto AIOX local. Você é o único usuário. A memória é local, os dados são locais, e você interage via CLI.

Características: - Setup mínimo — basta criar o squad e começar a usar - Custo zero de infraestrutura - Iteração rápida — você muda personas, dados, e comandos sem preocupação com impacto em outros usuários - Limitado a um usuário num dispositivo

Quando ficar aqui: Se o squad é genuinamente para uso pessoal e a necessidade de compartilhar não existe, não há razão para avançar. A Fase 1 é um destino válido, não apenas uma parada no caminho.

Sinal para avançar: Você está frequentemente compartilhando outputs do squad com outras pessoas e elas querem poder gerar esses outputs por conta própria.

Fase 2: Squad em Equipe (Repositório Compartilhado)

O que é: O squad é versionado no repositório do projeto. Múltiplos membros da equipe têm acesso. A memória compartilhada fica no arquivo MEMORY.md commitado.

Características: - Qualquer membro da equipe pode usar o squad - Memória compartilhada através do MEMORY.md no repositório - Dados de primeiro partido são centralizados (também no repositório ou em storage compartilhado) - Evolução do squad é colaborativa — qualquer um pode propor mudanças via PR

O que precisa mudar do Fase 1: - MEMORY.md precisa de convenções para não criar conflitos de merge - Dados sensíveis precisam ser separados do repositório (env vars, credenciais) - Personas precisam ser mais robustas — outros usuários podem não ter o contexto que você tem

Sinal para avançar: Pessoas fora da sua equipe técnica querem usar o squad, mas o fluxo via CLI não é acessível para elas.

Fase 3: Squad como API

O que é: O squad expõe uma interface de API HTTP. Outros sistemas podem chamar o squad programaticamente. Usuários não-técnicos podem acessar via frontend simples ou integração com ferramentas que eles já usam (Slack, Notion, etc.).

Características: - API REST com endpoints para cada comando do squad - Autenticação e autorização - Rate limiting e controle de custos - Logging e observabilidade - Múltiplos usuários concorrentes

O que precisa ser construído: - Servidor API (FastAPI, Express, ou similar) - Sistema de autenticação - Fila de requests para gestão de concorrência - Logging de uso para auditoria e controle de custos

Sinal para avançar: O squad tem volume suficiente de uso e usuários suficientes para justificar uma interface dedicada com UX própria.

Fase 4: Squad como Produto

O que é: O squad tem uma interface de produto própria — uma aplicação web, um bot, ou integração nativa com outra plataforma. Usuários finais interagem sem saber que há um squad AIOX por baixo.

Características: - UX desenhada especificamente para o domínio do squad - Onboarding de usuário - Histórico e gestão de outputs - Billing e monetização (se aplicável) - SLA e suporte

O que precisa ser construído: Tudo de Fase 3 mais uma aplicação frontend completa, sistema de gestão de usuários, e infraestrutura de produto.

13.2 O Que Determinizar: A Decisão Mais Importante

Na jornada de productização, a decisão mais importante não é "qual framework de API usar" ou "como estruturar o banco de dados". É: o que pode ser determinizado?

Determinizar significa transformar uma operação que usa IA em uma operação que usa código Python. Cada operação determinizada: - Executa mais rápido (ms vs. segundos) - Custa zero tokens de LLM - É 100% reprodutível - Não tem variância de output

O Mapa de Determinização para Squads

Operações que SEMPRE devem ser determinísticas:

Operação Código Python
Validar input do usuário Schema validation (Pydantic)
Formatar output para API JSON serialization
Autenticação e autorização JWT verification
Rate limiting Redis counter
Logging de uso Write to database
Caching de resultados Redis/memcached
Routing de requests Router logic
Aggregação de métricas SQL queries

Operações que DEVEM usar IA:

Operação Por que IA
Análise de mercado Requer julgamento sobre relevância e qualidade
Avaliação de equipe Requer interpretação de contexto
Síntese de múltiplas perspectivas Requer integração de insights
Identificação de riscos Requer raciocínio sobre situações complexas

A regra prática: Se você consegue escrever um if/else que resolve, use Python. Se a decisão depende de compreensão de contexto semântico, use IA.

O Anti-Pattern de Productização

O anti-pattern mais comum é usar IA para operações determinísticas durante a productização porque "é mais fácil". É mais fácil no curto prazo. No longo prazo, você tem:

  • Custos crescendo proporcionalmente ao uso (quando poderiam ser constantes)
  • Latência alta em operações que poderiam ser instantâneas
  • Variância em outputs que deveriam ser idênticos
  • Dependência de disponibilidade do modelo para funções não-cognitivas

13.3 A Hierarquia de Custo de Modelos

Na Fase 3 e Fase 4, você vai lidar com múltiplos usuários e volume crescente de requests. Nesse contexto, a escolha de modelo para cada operação tem impacto direto no custo operacional.

A Hierarquia

Claude Opus (mais poderoso, mais caro)
    ↓
Claude Sonnet (balanceado)
    ↓
Claude Haiku (mais rápido, mais barato)
    ↓
Regra Python (custo zero)

A estratégia correta não é "sempre use o modelo mais poderoso" — é usar o modelo mínimo suficiente para cada operação.

Matching de Operação com Modelo

Opus: Reserve para operações que exigem o mais alto nível de raciocínio — síntese final de análises de alto valor, casos edge complexos, operações onde a qualidade tem impacto financeiro significativo.

Sonnet: Use para a maioria das operações cognitivas — análise de mercado padrão, avaliação de equipe, geração de conteúdo de média complexidade. É o sweet spot entre qualidade e custo para a maioria dos casos de uso.

Haiku: Use para operações de baixa complexidade que ainda precisam de IA — classificação simples, extração de entidades, verificação de formato, respostas a perguntas simples sobre dados bem estruturados.

Exemplo de estratégia multi-modelo para o squad de startups:

routing:
  market-scan-quick: haiku          # Triagem inicial de 30s
  market-analysis-full: sonnet      # Análise completa de mercado
  investment-memo: opus             # Memo final de alta qualidade
  team-profile-quick: haiku         # Extração de dados de perfil
  team-analysis-deep: sonnet        # Avaliação qualitativa de equipe
  synthesis: opus                   # Síntese final (impacto alto)

Estimativa de Custo Real

Para calibrar a estratégia de modelos, faça a matemática com uso esperado:

Se você espera 100 análises completas por mês: - Com tudo em Opus: X tokens × $15/MTok = custo alto - Com routing inteligente: 60% Sonnet + 30% Haiku + 10% Opus = custo 60-70% menor

A diferença de custo justifica o esforço de implementar o routing, especialmente quando o squad escala para centenas de usuários.

13.4 Empacotando o Squad

O Servidor API Mínimo

Para a Fase 3, você precisa de um servidor API que exponha os comandos do squad. Um exemplo usando FastAPI:

from fastapi import FastAPI, HTTPException, Depends
from pydantic import BaseModel
from typing import Optional
import asyncio

app = FastAPI(title="Startup Analysis Squad API")

class AnalysisRequest(BaseModel):
    company_name: str
    description: str
    market: str
    team_info: Optional[str] = None
    analysis_type: str = "full"  # full, market-only, team-only

class AnalysisResponse(BaseModel):
    request_id: str
    status: str
    result: Optional[dict] = None
    error: Optional[str] = None

@app.post("/analyze", response_model=AnalysisResponse)
async def analyze_startup(
    request: AnalysisRequest,
    # user: User = Depends(get_current_user)  # Auth
):
    """Executa análise completa de uma startup."""
    try:
        request_id = generate_request_id()

        # Log do request (determinístico)
        await log_request(request_id, request)

        # Verificar cache (determinístico)
        cached = await get_cached_result(request.company_name)
        if cached:
            return AnalysisResponse(
                request_id=request_id,
                status="cached",
                result=cached
            )

        # Executar squad (IA)
        result = await execute_squad_analysis(request)

        # Cache do resultado (determinístico)
        await cache_result(request.company_name, result)

        return AnalysisResponse(
            request_id=request_id,
            status="completed",
            result=result
        )

    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))

O Manifesto de Deployment

Para deployar o squad como serviço, você precisa de um manifesto que descreva os recursos necessários:

13.5 Operação em Equipe

Quando o squad atende múltiplos usuários, você precisa de práticas operacionais que vão além do que é necessário para uso solo.

Controle de Qualidade em Produção

Um problema que surge na Fase 3 e 4: como você sabe que o squad continua produzindo outputs de qualidade após mudanças de personas, dados, ou modelos?

Práticas essenciais:

Golden set de testes: Crie um conjunto de inputs com outputs esperados (ou outputs de referência aprovados). Antes de qualquer deploy de nova versão do squad, rode todos os inputs do golden set e compare os outputs com os de referência. Desvios significativos indicam regressão.

Human-in-the-loop sampling: Para squads de alto risco (análises com impacto financeiro, conteúdo que vai para clientes), sample uma porcentagem dos outputs para revisão humana. Isso detecta degradação de qualidade que testes automatizados não capturam.

Feedback loop: Implemente um mecanismo de feedback nos outputs. Se um usuário marca um output como "ruim", essa informação precisa chegar de volta para quem mantém o squad — e idealmente para o MEMORY.md do squad como aprendizado.

Gestão de Custos em Produção

O custo de um squad em produção tem dois componentes: o custo fixo (infraestrutura, API overhead) e o custo variável (tokens de LLM). O custo variável é o que pode escalar descontroladamente.

Práticas de controle de custo:

class CostGuard:
    """Monitora e controla custos de LLM em produção."""

    def __init__(self, daily_budget_usd: float):
        self.daily_budget = daily_budget_usd

    async def check_budget(self, user_id: str, estimated_tokens: int) -> bool:
        """Verifica se o request está dentro do orçamento."""
        # Custo atual do usuário hoje
        user_cost_today = await get_user_cost(user_id, period="today")

        # Custo estimado deste request
        estimated_cost = estimate_cost(estimated_tokens)

        # Custo global do sistema hoje
        system_cost_today = await get_system_cost(period="today")

        if system_cost_today + estimated_cost > self.daily_budget:
            # Budget do sistema atingido — enfileira para amanhã
            await queue_request_for_tomorrow(user_id)
            return False

        return True

13.6 Monitoramento e Observabilidade

Um squad em produção precisa de observabilidade. Sem ela, você descobre problemas quando usuários reclamam — não antes.

O Que Monitorar

Métricas de negócio: - Volume de análises por período - Taxa de outputs aprovados por usuários (via feedback) - Tempo médio de análise - Distribuição de tipos de análise solicitados

Métricas técnicas: - Latência por percentil (p50, p95, p99) - Taxa de erro por endpoint - Custo de tokens por análise - Cache hit rate

Métricas de qualidade: - Score de feedback por tipo de análise - Taxa de reanalise (usuário pediu refazer = output insatisfatório) - Comparação de outputs contra golden set

Dashboard Mínimo

Para a Fase 3, um dashboard simples mas funcional cobre as perguntas mais importantes:

Análises hoje: 47 (↑ 12% vs ontem)
Custo total hoje: $4.23 (budget: $20/dia)
Latência média: 8.2s
Taxa de erro: 0.4%
Satisfaction score: 4.3/5 (n=23)

Esse nível de visibilidade permite detectar anomalias rapidamente — um spike de custo, degradação de latência, ou queda de satisfaction score antes que se tornem crises.

A Transição Entre Fases É Reversível

Uma nota importante antes de encerrar o capítulo: você não está preso numa fase. Squads podem avançar e retroceder.

Se você construiu uma Fase 3 e o volume de uso não justifica a infraestrutura, você pode voltar para Fase 2 sem perder nada — o squad continua funcional, apenas sem o servidor API.

Se você está na Fase 2 e alguém sugere construir uma interface de produto (Fase 4) antes de ter a API (Fase 3) estabilizada, resista. A jornada de fases existe por uma razão: cada fase valida demandas e aprende sobre o uso real antes de investir na próxima.

O squad de maior valor que você vai criar vai começar como uma ferramenta pessoal estranha que você usa para resolver um problema específico. Vai evoluir gradualmente à medida que o valor se comprova e a demanda cresce. A productização prematura de um squad é um dos desperdícios mais comuns — e mais evitáveis — neste domínio.

13.7 A Mentalidade de Serviço Produtizado

A Era do Serviço Produtizado, Não do SaaS

Alan Nicolas é enfático: "Estamos na era do serviço produtizado, não do SaaS."

A fricção de multiplataforma mata SaaS. Cada nova ferramenta que o cliente precisa aprender, cada novo login, cada nova interface — é fricção que reduz adoção. Serviço produtizado elimina isso: o cliente recebe o resultado, não a ferramenta.

O potencial: R$100k-200k/mês em serviço produtizado com squads AIOX. Não é projeção otimista — é dado real de operadores do cohort.

Nunca Venda o Squad, Venda a Solução

Regra fundamental de monetização: não mencione AIOX para o cliente. O cliente não quer saber que você usa IA, que tem agentes, que tem workflows. O cliente quer o resultado. Venda o resultado.

Nunca comece vendendo. Gere valor até ouvir "quanto custa?" Reciprocidade é o motor: faça um POC para o cliente em meia hora na frente dele. Quando ele vê o resultado ao vivo, a venda acontece sozinha.

Precificação Pelo Valor

"Não meça pela sua régua." Se você cobra R$2k porque é o que acharia justo pagar, está subcobrando. Precifique pelo valor que o resultado gera para o cliente, não pelo seu custo de produção.

Exemplos reais do mercado: - Brand Books: R$10-15k (mentorados) até R$300-500k (Ana Couto) - Clone/agente especialista como serviço: R$15k → R$80k - Refatoração legacy para AI-First: R$200k-5M - Ray Schwartz: US$65M com Amy Porterfield (serviço produtizado de webinários)

Alto ticket dá menos trabalho que baixo ticket. Menos clientes, mais atenção, melhor resultado, mais margem.

Os 3 Caminhos de Negócio

O cohort identificou 3 modelos viáveis:

Caminho Descrição Escala
Consultoria Diagnóstico + implementação para empresas 5-10 clientes ativos
Serviço Produtizado Output recorrente via squad (copy, análise, design) 20-25 clientes via automação
SaaS Produto com interface para múltiplos usuários Ilimitado, mas só com 20+ cases antes

Recomendação: não lance SaaS sem pelo menos 20 cases de serviço bem sucedidos. Os cases ensinam o que automatizar e o que manter humano.

13.8 Microserviços e Automação

Cada Funcionalidade Vira Endpoint

Na Aula 07, Pedro introduziu o conceito de workers compartilhados — infraestrutura de microserviços que qualquer squad pode consumir:

infrastructure/services/
├── etl-worker/          # Extração e transformação de dados
├── youtube-worker/      # Download e transcrição de vídeos
├── ocr-worker/          # OCR de documentos
├── transcription-worker/ # Transcrição de áudio
└── scraping-worker/     # Web scraping

Cada worker é um endpoint independente. Um squad de resumo de livros chama o ocr-worker. Um squad de análise de concorrentes chama o scraping-worker. Reutilização máxima, duplicação zero.

n8n como Gateway

O n8n é a porta de entrada para quem quer aprender APIs e webhooks. Funciona self-hosted, já tem skill/MCP no GitHub, e permite criar automações visuais que conectam squads a ferramentas externas.

Alan usa ClickUp + 47 automações via webhooks para orquestrar squads em produção. O fluxo: ClickUp trigger → webhook → squad → output → ClickUp update. Cada mudança de status no ClickUp aciona o squad correspondente.

Workflows Pequenos e Modulares

Regra da Aula 07: workflows pequenos e modulares > workflow gigante. Cada mudança de status é um trigger que aciona um workflow específico. Isso permite: - Debug isolado (problema num workflow não afeta outros) - Reutilização (o mesmo workflow serve múltiplos squads) - Evolução incremental (atualiza um workflow sem tocar nos outros)

Exercício: Mapeie a Jornada do Seu Squad

Pegue o squad que você criou no Capítulo 12 (ou qualquer squad que você planeja criar).

Análise de Fase Atual: - Em qual fase o squad está hoje? - Quais são os sinais que indicariam que é hora de avançar para a próxima fase?

Mapa de Determinização: Faça uma lista de todas as operações que o squad vai executar. Para cada uma, classifique: Python (determinístico) ou IA (cognitivo). Calcule a proporção.

Estratégia de Modelos: Para as operações que precisam de IA, classifique: Haiku, Sonnet, ou Opus. Estime o custo mensal com 100 usuários.

Golden Set: Crie 5 casos de teste para o seu squad — inputs que você conhece bem o suficiente para saber como é uma resposta boa. Esses serão seus testes de regressão quando o squad evoluir.

A productização de um squad bem feito pode se tornar um produto real com valor de mercado. O caminho começa com uma ferramenta pessoal que você mesmo usa todos os dias.
